## Design (determined; fix against named files; no brainstorm)

resolveMode(deploymentId):

- resolved → `resolveConsentStateConfig(config).mode` (unchanged behavior for the healthy path).
- error → `getMetrics().bookingConsentResolverError.inc({deploymentId, fallback})`; then `consentPostureCache.lastKnown(deploymentId)?.mode === "enforce" ? "enforce" : "off"` (fail-closed ONLY when a warm enforce posture exists; cold/observe → off). `fallback` label = the returned mode.
- missing → `"off"` (genuinely not enrolled; NO counter — not an error).
  read → forwards to `contactConsentReader.read(orgId, contactId)` (unchanged).

NO self-warm in the booking adapter: the 1c hook runs as an afterSkill hook on EVERY turn and is the reliable producer; the booking adapter's resolveMode runs once per booking (one tool-exec BEFORE the same turn's 1c afterSkill), so a self-warm would add a redundant identical write with ZERO extra cold-window coverage (the only cold window — first-ever turn is a booking + simultaneous resolver error — precedes both producers). Cold→off is the documented/accepted acceptance edge.
