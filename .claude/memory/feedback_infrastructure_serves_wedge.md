---
name: Infrastructure serves the wedge
description: SP2 and internal infrastructure must be framed as wedge-serving, not platform-for-platform's-sake — judge by whether it accelerates the revenue loop
type: feedback
---

Internal infrastructure work (SP2 tool ecosystem, governance, migration patterns) must always be framed as serving the revenue wedge: ad → lead → qualification → booking → close → signal back to ads/creative.

**Why:** The danger is not that infrastructure is wrong — it's that it gets treated as the main thing. SP2 is correct as an enabler for expanding conversational agents in the revenue loop. It becomes wrong if it drifts into marketplace, abstract plugin systems, or solving distant scale problems.

**How to apply:** When writing specs or plans for internal systems work, use wedge-serving governing sentences (e.g., "so Switchboard can expand the revenue loop without bespoke orchestration per agent"). Judge every deliverable by: "Does this make the end-to-end revenue wedge easier to build and safer to operate?" If no, cut or defer.
