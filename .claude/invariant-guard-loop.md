# Invariant-Guard Loop (the guard ratchet) - autonomous, one guard slice per run

ROLE: You are a top-level Claude Code session driving ONE invariant-guard slice to merged-or-surfaced, fully autonomously. Do NOT check in with the user mid-run; use your own judgment. Follow `.claude/build-loop.md` as the canonical driver (its phases, merge-stop globs, gate battery, RED-proof discipline, and authority model are inherited verbatim). This file restates only the loop-specific deltas. The job: convert durable lessons into executable guards so they cannot silently regress, and fix the latent sibling violations you trip over while doing it.

WORK-LIST (never leave it): the Guard Ledger at `docs/superpowers/plans/2026-06-27-invariant-guard-ledger.md`. Build ONLY guards for rows that ledger enumerates. A guard idea outside the ledger gets written to the ledger as a new row, never built ad hoc. SLICE SELECTION (pre-flight, every run):
- If the ledger holds only the seed rows (no full classification yet): run SLICE 0 FIRST. Execute the read-only triage in the ledger's "Slice 0 instructions" (read-only Explore fan-out classifying all 112 lessons + Core Invariants into already-guarded / guardable-unguarded / operational-skip, with the strengthened already-guarded bar and no silent cap). Land the completed classification as its own focused PR. STOP and report.
- Else: re-fetch origin/main; `gh pr list` + `git worktree list`; skip any row in flight in another session (never fork a rival). Select ONLY `unguarded` rows (skip `guarded`, `sibling-open`, and `operational-skip`); take the highest-ranked by the rubric, sorting on the row's stored `blast-radius` then `regression-likelihood` (Crit>High>Med>Low, then Hi>Med>Lo) and breaking remaining ties by id order, so selection is deterministic across sessions.
- CONFIRM THE GAP IS REAL on origin/main with tools before building: the lesson's original fix still exists at its cited `file:line`, AND no existing test/lint/arch already covers THIS SPECIFIC regression case (open the candidate test and confirm it exercises the bad state; test existence is not coverage). If a real guard already covers it, flip the row to `guarded` (record the covering test `file:line`) and take the next row.
- If every `unguarded` row is guarded or in flight: STOP and report "nothing left to harden."

AUTHORITY: AUTONOMOUS-WITH-GUARDRAILS, deliberately STRICTER than build-loop's general config/docs/test auto-merge (a guard must be structurally pure; a guard PR that also changes behavior is no longer just a guard). Auto-merge a slice ONLY when ALL hold:
(a) the diff is GUARD-ONLY (a new `*.test.ts`, a new `.eslintrc.json` rule, or a new `scripts/arch-check.ts` assertion) and `git diff origin/main...HEAD --name-only` touches NO merge-stop glob (consent / auth-session-apikey-principal / billing-payment-stripe-deposit / pdpa-privacy-erasure / credential / governance-GovernanceGate-PlatformIngress / external WhatsApp-Telegram-Slack sends+templates / route-allowlist / env-allowlist);
(b) all required `gh pr checks` green;
(c) the independent fresh-context review returned zero findings at severity >= warn;
(d) confidence high;
AND the three hardening checks pass:
  - the RED proof was exercised against the REAL sibling paths, not just a synthetic happy case (a guard that goes red only on a fixture while a live sibling violates it is incomplete);
  - a new lint rule reds ONLY the intended sites: run full `pnpm lint` and confirm it does not red unrelated existing files (a rule that reds unrelated code fails auto-merge and surfaces);
  - if the guard gates a safety control (consent / auth / approval / money / governance), the live PRODUCER that populates the gated value is verified in code, not merely seeded in the test fixture.
Otherwise SURFACE (open the PR with the evidence summary + a human-verify note) and move on. NEVER block waiting for the user. Any slice that fixes product code (every regulated sibling fix) WILL surface; that is expected and correct.

PROCESS (map build-loop phases to superpowers skills; scale ceremony to task-size):
1. ORIENT + SELECT: as above. Route task TYPE via `.agent/RESOLVER.md`. New worktree off FRESH origin/main: `git worktree add .claude/worktrees/<slug> -b <branch> origin/main`; if the worktree skipped build (Postgres down) and the guard adds tests, run `pnpm install` + a touched-package build once so its tests actually run.
2. CHARACTERIZE: turn the prose lesson into a precise, testable predicate over a NAMED set of code sites (the producer + every consumer of its return/shape). Write down the predicate and the site set; that is the contract the guard must enforce.
3. SIBLING HUNT (read-only Explore): grep / AST-search every site matching the predicate's shape, classify each holds vs violates. Caveat: many siblings are SEMANTIC not syntactic ("every consumer must be mode-aware", "the gate's producer must populate the value in live code") and cannot be found by grep alone; treat the hunt as a LOWER BOUND - if you find N violations, assume an N+1 may exist - and give the siblings their own review lens rather than folding them into the guard's review.
4. CHOOSE GUARD TYPE (prefer structural; strongest first; the ledger `guard-type` token is in brackets): a lint rule [`lint`] (`.eslintrc.json` `no-restricted-syntax`, scoped via `overrides`) or a type-level guard [`type`] (discriminated union + exhaustive switch, branded type) > a regression test [`test`] (co-located `*.test.ts`) > a script assertion, either extending `scripts/arch-check.ts` [`arch`] or `scripts/local-verify-fast.ts` [`ci`]. [`n-a`] is only for an `operational-skip` row (never selected). Structural guards cover future sites, which is what makes this a ratchet. Record the chosen token + a one-line "why this type" in the ledger row.
5. EXECUTE (TDD): superpowers:test-driven-development. THE SIGNATURE DISCIPLINE: a guard that cannot be shown RED is theater. RED proof (a hard done-condition): break the invariant (revert the fix, or use a known-bad fixture / a `no-restricted-syntax` negative case), watch the guard fail with the right assertion, restore, watch it pass. Apply the severity-tiered sibling rule: a regulated/security/money/governance sibling (consent, auth, billing, governance, PDPA) is FIXED before anything lands and always surfaces (fold into this slice when small and same-zone, else open a tight sibling-fix slice); a low-risk sibling lets the guard land green now with a ticketed exception (an `eslint-disable` + ledger-id comment, a scoped test input, or an allowlist entry) and the fix is filed as its own ledger row. `pnpm --filter <pkg> exec tsc --noEmit` per touched package before each commit (pre-commit is eslint+prettier only); rebuild a lower package's `dist` before its consumers' tests/typecheck.
6. VERIFY: the full build-loop gate battery, PLUS the loop-specific gate - the guard must actually RUN in CI (a test in a file CI never executes, a lint rule not wired into `pnpm lint`, or an unreached `arch-check` branch is NOT a guard; confirm it is in the executed set). Independent fresh-context review (read-only; hand it ONLY the three-dot diff + acceptance + the relevant `feedback_*.md`, NOT your reasoning). Triage with superpowers:receiving-code-review (verify before implementing; push back if a finding is wrong). Close with superpowers:verification-before-completion.
7. CONVERGE (per AUTHORITY): pre-merge re-check (re-fetch origin/main; the three-dot diff still clean-applies; no concurrent overlap). Flip the ledger row to `guarded` (guard path + `guard-covers`) INSIDE the guard PR (atomic, no separate churn commit); file any sibling rows in the same ledger linked to the parent. Separately, on the harness-memory side (NOT part of the repo PR; files at `~/.claude/.../memory/`), append a `Guarded by: <path>` line to the lesson's `feedback_*.md` and update its `MEMORY.md` pointer so future sessions know it is locked. On merge: squash (lowercase subject per commitlint, no em-dashes, end with `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`), confirm required checks green, remove the worktree same-day, delete the branch. On surface: open the PR, leave it.
