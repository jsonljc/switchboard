## Goal (one line)

An ops CLI (`scripts/provision-pilot.mts`) that fully provisions one vetted pilot clinic org (user + org + comped entitlement + business hours + owner IdentitySpec + Riley deployment) so a real clinic can be onboarded while self-serve signup is closed (`waitlist` launch mode). F-05 flagged this as REQUIRED before flipping prod to waitlist.

## FRAME (design decision + why)

**Problem:** the full provisioning lives in `apps/dashboard/src/lib/provision-dashboard-user.ts` (`provisionDashboardUser`), which is app-private (`@switchboard/dashboard` is `private`, no `exports`), so a root `.mts` ops script cannot cleanly import it. Re-implementing its body in the script would create a THIRD org-create site that drifts from the F-01 (businessHours) / F-02 (entitlementOverride) fixes — the exact bug those fixes fought. Separately, the apiKey crypto (`sha256(CREDENTIALS_ENCRYPTION_KEY)` key + `iv:authTag:encrypted` hex, aes-256-gcm) is TRIPLICATED — `apps/dashboard/src/lib/crypto.ts`, `packages/db/prisma/seed.ts:31` (with a load-bearing "Must match... if this drifts every seeded user's apiKeyEncrypted becomes undecryptable" comment), and inline in `apps/api/src/routes/setup.ts`.

**Decision:** extract the canonical provisioning + apiKey crypto into `@switchboard/db` (where the sibling seeders `seedOrgDayOneAgents` / `provisionOrgAgentDeployments` already live and are imported by BOTH apps and ops scripts). Re-point the dashboard provisioning + crypto to the db canon. The script then composes db functions exactly like `scripts/provision-mira-for-org.mts` does. This kills 2 of the 3 crypto copies and makes provisioning single-source.

**Why db is the right layer:** everything `provisionDashboardUser` does is db-layer — it creates Prisma models (OrganizationConfig, Principal, IdentitySpec, DashboardUser), calls `seedOrgDayOneAgents` (already in db), uses `DEFAULT_BUSINESS_HOURS` (schemas; db imports schemas), and apiKey encryption ("credential encryption" = db's job per the codebase map). It is currently mis-located in the app. No new cross-layer deps; no cycle (db -> schemas only).

**Scope boundaries (deliberate):**

- LEAVE `apps/api/src/routes/setup.ts` (crypto copy #3) alone — its no-key `"unencrypted:bootstrap"` fallback is intentional and it is first-user-only/low-value; re-pointing would change that fallback semantics and pull apps/api into the slice. Documented follow-up.
- `provisionOrgWithOwner` (the moved signup body) must NOT call `provisionOrgAgentDeployments` — signup deliberately defers Riley's _deployment_ to the lazy `GET /config` (agent tiering). Eager Riley provisioning is a SCRIPT-only concern (`provisionPilotOrg`).
- Auth: prod login is CredentialsProvider (email+password). So the script sets a passwordHash (bcryptjs, self-describing cost — `verifyPassword`/bcrypt.compare verifies any cost) and prints a temp password. emailVerified is stamped now (vetted pilot; skip the verification round-trip).
