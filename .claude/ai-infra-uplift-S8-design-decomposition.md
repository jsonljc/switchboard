# S8 (Govern memory writes through PlatformIngress) — DESIGN + decomposition (multi-slice -> BLOCK)

Date 2026-06-20. Disposition: **[B] blocked** — discovered multi-slice at ORIENT (research itself rates f16 "Effort M-L"). This doc is the brainstorming/design deliverable + the decomposition. Did NOT autonomously commit, per the build-loop multi-slice + schema-migration-risk STOP conditions and because PlatformIngress is the core mutating-entry invariant.

## Why multi-slice (ORIENT ground truth, file:line)

1. NO memory-write governance intent exists. Memory writes go DIRECT: `compounding-service.ts` calls `memoryStore.create()/incrementConfidence()/delete()` (lines 356/372/386/441/481/509), never `PlatformIngress.submit()`. Routing through ingress needs a new `IntentRegistration` + (unless `system_auto_approved`) a seeded `Policy` (GovernanceGate default-DENIES unmatched intents). Governance seeding is its own slice (template: riley-pause-governance.ts / robin-recovery-governance.ts).
2. Schema migration MANDATORY. `DeploymentMemory` (schema.prisma:769-787) has id/org/deployment/category/content/canonicalKey?/confidence/sourceCount/lastSeenAt/lastDecayedAt/created/updated. It has NONE of `source`, `validFrom`, `validTo`, `invalidatedAt`. All net-new columns (+ store + Zod `deployment-memory.ts:57-69`). Postgres down -> hand-written migration, CI-only drift validation -> migration-risk STOP condition.
3. Caller has no actor. `gateway-bridge.ts:146 onConversationEnd -> processConversationEnd(event)` is a plain callback (ConversationEndEvent, not a CanonicalSubmitRequest). Submitting through ingress needs a `system` actor + `trigger:"internal"` threaded through gateway-bridge -> compounding-service, and the decay cron (`inngest.ts:701-725`, cross-org, no actor) too. (feedback_cron_submit_seeded_system_principal.)

## DESIGN MODEL (f16; brainstorming output — adopt provenance + valid-time + invalidate, NOT a vector DB/Graphiti)

- **Provenance (`source`)**: tag each write with origin (enum-ish: "conversation-compounding" | "pattern-merge" | "operator" | "decay"). Enables "who asserted this" + audit.
- **Bi-temporal valid-time (`validFrom`/`validTo`)**: when the FACT is asserted to hold, distinct from system time (createdAt/updatedAt). Enables "what did we believe at T". validTo null = still valid.
- **Invalidate-not-delete (`invalidatedAt`)**: REPLACE the hard `delete()` (evict-at-cap, line 372) AND the decay-to-floor with setting `invalidatedAt` (and validTo) instead of removing the row; reads filter `invalidatedAt IS NULL`. Preserves history + lets a future write supersede rather than erase. This is the ONE part with a live consumer today (the evict/decay paths), so it anchors slice A.
- **Deterministic conflict resolution** (f16): on a collision (same canonicalKey/content) resolve by a DETERMINISTIC rule (higher confidence, then later validFrom wins), reusing the existing dedup-axis (unique on CONTENT `@@unique([org,deployment,category,content])`; bucket on canonicalKey + cosine >= 0.84) + the check-then-store idempotency-fingerprint. NEVER an LLM/nondeterministic merge. (feedback_deployment_memory_dedup_axis, feedback_idempotency_reuse_checktime_fingerprint.)
- **Ingress-governed write**: new `memory.write` intent, `approvalMode: "system_auto_approved"` (non-financial internal write; NOT spend-bearing so it passes the F4 spend guard at intent-registration.ts:106-110), idempotent (reuse fingerprint), executor = an operator_mutation-style memory-write binding. The conversation-end path + decay cron submit as the seeded `system` actor, `trigger:"internal"`. Result: memory writes become audited (WorkTrace), governed, idempotent — closing the "memory writes bypass PlatformIngress" invariant gap (research finding 5).

## Decomposition (sequence A -> B -> C)

- **S8a — schema + invalidate-not-delete** [SURFACE: prisma stop-glob]. Add `source String?`, `validFrom DateTime?`, `validTo DateTime?`, `invalidatedAt DateTime?` to DeploymentMemory; update PrismaDeploymentMemoryStore (new `invalidate()` replacing hard delete; reads filter invalidatedAt null; decay sets invalidatedAt/validTo at floor); update Zod DeploymentMemorySchema; populate `source` from the existing writers. - **S8b — `memory.write` intent + governance** [SURFACE: governance/ingress]. IntentRegistration (system_auto_approved, non-spend) + parameterSchema (org/deployment/category/content/source/confidence/validFrom) + executor binding (operator_mutation-style) + the new-skill-intent governance recipe (anchored allow or system_auto_approved; entitlement gate; non-conversation executorBySlug if needed) + route-allowlist if a bypass route is added. Depends on A's types.
- **S8c — ingress integration** [SURFACE: ingress glob; touches PlatformIngress invariant]. Thread PlatformIngress + the seeded `system` actor through ConversationCompoundingService (deps + gateway-bridge.ts callback) and the decay cron; replace the direct store writes with `ingress.submit({intent:"memory.write", actor:system, trigger:"internal", idempotencyKey, parameters})`. Depends on A + B. Highest-stakes (core invariant); SURFACE + careful.

## Blockers / why not build slice A now

Slice A is a migration to a core table that can't be drift-validated locally (Postgres down) and is a build-loop migration-risk STOP; A's `source`/valid-time columns are unpopulated-by-ingress until B+C, so building A alone is partial (only invalidate-not-delete is live); and the whole arc is a genuine greenfield design touching the PlatformIngress invariant, which warrants a human greenlight on the design model above before the schema lands. Recommend the user approve the design + sequence (A->B->C), then run them as focused slices.
