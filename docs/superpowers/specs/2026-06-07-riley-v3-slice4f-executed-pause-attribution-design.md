## 1. What this slice does, in one paragraph

Today a Phase-C executed pause produces ZERO outcome rows: the executor records `recommendationId` in WorkTrace outputs and returns, but the source `PendingActionRecord` stays `status:"pending"` forever, and `findAttributableCandidates` requires `status:"acted"` plus `resolvedAt` (`recommendation-outcome-store.ts:256`). The 4d corroborated arm therefore fires only for operator-acted recommendations, which is backwards for the pilot: the actions that must earn trust are invisible to the ledger. This slice adds one conditional, race-safe, audit-logged transition on the executor's truthful success leg (Meta write accepted), anchored at execution time, attributed to a machine sentinel, and threads the executable work-unit id through to the outcome row (the `RecommendationOutcome.executableWorkUnitId` column already exists and has only ever held null). After this slice, "Riley paused it, spend fell, bookings held, corroborated" is measurable end to end before any pilot-org flag flip. Zero migrations, zero new flags, zero UI changes, zero new routes.

## 2. The transition contract (the load-bearing part)

A new store method on `PrismaRecommendationStore` (db, Layer 4):

```ts
markActedByExecution(args: {
  id: string;                    // PendingActionRecord id (the recommendation)
  organizationId: string;
  executableWorkUnitId: string;  // WorkUnit.id of the executing work unit
  resolvedBy: string;            // machine sentinel, supplied by the caller
  executedAt: Date;              // execution clock, becomes resolvedAt
}): Promise<{ transitioned: true } | { transitioned: false; reason: "not_found" | "not_pending" }>
```

Interactive `$transaction` (the `feedback_updatemany_drops_nomatch_abort` form, mirroring `applyAct`'s shape):

1. `findFirst({ id, organizationId, intent: { startsWith: "recommendation." } })`. Missing row returns `{ transitioned: false, reason: "not_found" }`. The read also supplies `parameters` for the metadata merge and `riskLevel`/`humanSummary` for the audit row.
2. Conditional `updateMany({ id, organizationId, status: "pending", intent: { startsWith: "recommendation." } })` setting `status: "acted"`, `resolvedAt: executedAt`, `resolvedBy`, and `parameters` merged as `__recommendation.executedWorkUnitId = executableWorkUnitId` (the established rec-metadata stash beside `note`/`riskContract`; all sibling keys preserved). `count === 0` returns `{ transitioned: false, reason: "not_pending" }`: a benign first-writer-wins no-op (operator acted or dismissed concurrently, lazy expiry won, or a retry already transitioned), never a throw.
3. On `count === 1` only: an `AuditEntry` mirroring `applyAct`'s (`eventType: "recommendation.act"`, `actorType: "system"` (precedented value), `actorId` = the sentinel, snapshot `{ from: "pending", to: "acted", note: null, executableWorkUnitId }`, per-row `entryHash` via the file's existing `buildEntryHash` with `ts: executedAt.getTime()`, NOT `Date.now()`: this slice anchors truth at execution time and the hash input must not silently depend on a second wall clock inside the db method. The hash stays unique regardless (it is salted with a random UUID), which also means the ts source is not black-box testable; the code comment is the record. `applyAct` keeps `Date.now()` because its event clock IS the wall clock).

Infra errors (DB down) propagate to the caller; the executor catches them (section 3). The `intent` guard is defense in depth: `PendingActionRecord` also hosts workflow approval rows (`workflowId` rows with a disjoint status vocabulary), and this method must be unable to touch them even on a forged or buggy id.

**Why the status predicate is the serialization point.** The operator path (`applyAct`) and this method both transition away from `pending` conditionally; whichever commits first wins, and the loser sees `P2025` (operator path, mapped to `RecommendationStaleStatusError` then `already_terminal`) or `count === 0` (machine path, benign no-op). The parameters merge rides the same conditional write, so a lost-update clobber of a concurrent operator note is unreachable: two writers cannot both pass the status predicate.

## 3. Decision record

1. **Transition site: the executor success leg, via a REQUIRED injected dep.** `buildRileyPauseExecutionWorkflow` deps gain `markRecommendationActed(args)`; only the leg that ran `updateCampaignStatus` without throwing (`metaWriteAccepted: true`) calls it, after the write and using the executor's injected clock. Initiator is impossible (at submit time nothing has executed; the submitter learns park truth only). A lifecycle/dispatch-layer hook is wrong twice: core cannot import db (layer doctrine), and the dispatch layer cannot distinguish a real pause from the executor's completed-with-skip legs (already-paused, stale, floor) without fragile structural reads of `outputs.paused`, which is exactly the truth-inference 4f must not rely on. The dep is required, not optional: an optional dep lets a future bootstrap forget the wiring and silently recreate today's attribution hole (the safety-gate-producer-population trap).
2. **Legs that must NEVER transition** (each pinned by a test): invalid input, class-ineligible, below-execution-floor, stale-approval, org-mismatch, no-connection, already-paused skip, not-pausable skip, Meta write failure. The already-paused pre-read skip deserves its own record: the campaign was paused by someone or something else before this work unit wrote, so there is no spend change attributable to this recommendation; transitioning it would let the ledger measure someone else's action. The recommendation stays pending and expires on the operator surface honestly (Riley did not act). Known product wrinkle, recorded as residue (section 8): in the manual-preempt sequence (operator pauses in Ads Manager, then approves the parked card), the approval card reads completed while the source recommendation stays pending until it expires. Truthful but cosmetically odd; a future cleanup rule (e.g. already_paused pre-read marks the rec `resolved_by_external`, a new status with its own consumer sweep) is recorded as an option, deliberately NOT shipped here. Never mark it acted.
3. **Transition failure never fails the work unit, but it is LOUD.** The Meta write already happened; that is the execution truth the operator was promised. The executor wraps the dep call; on `{ transitioned: false }` or a thrown infra error it still returns `outcome: "completed"` with the pause outputs intact, recording `recommendationTransition: "acted" | "not_found" | "not_pending" | "error"` plus `executedAt` in outputs (WorkTrace is the canonical record). The thrown-error leg additionally emits a greppable `console.error("[riley-pause] failed to mark recommendation acted ...")` carrying org/rec/work-unit ids and the error message, so "Meta paused but attribution linkage failed" is discoverable and alertable, not just trace-archaeology (review feedback: operational visibility). The benign `not_pending` no-op is NOT error-logged (first writer won; that is expected product behavior, recorded in outputs). The alternative (fail the unit) would convert a bookkeeping blip into a false "failed" claim about a pause that succeeded, plus an operator retry that abstains on the already-paused pre-read and STILL never transitions: strictly worse on both honesty axes.
4. **resolvedAt = execution time** (the executor's `now()` captured after the Meta write accept). `attributeOneRecommendation` anchors `anchorAt = candidate.resolvedAt` (`outcome-attribution.ts:47`), so this choice is what makes the attribution windows measure reality: pre-window = the regime before spend stopped, post-window = after. `requestedAt` can be up to 48h stale (executor cap) and approval time is a human latency artifact; both would smear the pre/post boundary. Pinned twice: a unit test executing ~47h after `requestedAt` (inside the executor cap, no lifecycle in between) and an e2e test approving 20h after the park (the platform's 24h park expiry, itself pinned by the existing stale-park test, makes a longer e2e gap unreachable through the respond path); both assert the row anchors on the execution clock.
5. **resolvedBy = `"riley_self_execution"`** (exported constant `RILEY_PAUSE_EXECUTION_RESOLVED_BY` beside the executor; the db method takes `resolvedBy` as a plain arg so the store stays generic and the riley-specific naming lives at the riley call site). Not the approving principal: they approved, they did not act, and at execution time the executor does not know them (threading the approver through dispatch would add plumbing in service of a misattribution). Not bare `"system"`: that is a real seeded principal id and would be ambiguous with actual system-principal acts. The same table already holds machine `resolvedBy` values for workflow rows (`"policy_engine"`, `"auto"` in `step-executor.ts`), so a machine sentinel is precedented, greppable, and cannot collide with a human principal id. Human-approval provenance is not lost: it lives on the lifecycle (`respondedBy`), the WorkTrace, and the audit snapshot's `executableWorkUnitId` link.
6. **resolvedBy/actedBy consumer verdicts (no surface lies).** `rowToRecommendation` maps `resolvedBy` to `actedBy`, surfaced only by `routes/recommendations.ts` (no dashboard consumer beyond the type; grep-verified). The wins surface (`projectWins`) never reads `actedBy` and renders every acted row in the AGENT's voice ("Adjusted. <humanSummary>"), so a machine-executed pause appearing there is the one case where that copy is literally true. No copy change is required anywhere; the early-exit condition (truthfulness impossible without breaking a consumer) was checked and does not fire.
7. **Expiry interplay (verified semantics, decision recorded).** Expiry is lazy: `act.ts` flips `pending -> expired` only when an operator acts on an expired row; there is no sweep. Pause recommendations carry urgency-derived `expiresAt` (8h/24h/168h), so `expiresAt` can pass while the row is still `status:"pending"`. The transition deliberately ignores `expiresAt`: the pause executed, execution staleness is already bounded by the platform's 24h park expiry plus the executor's 48h `requestedAt` cap, and `expiresAt` is operator-surface staleness, not an execution gate. The truly blocked corner (operator triggered the lazy flip to `expired` before the parked approval executed; the machine transition then no-ops and a really-executed pause stays unattributable) is accepted and recorded: it requires an operator act on an expired tile inside a window that the 24h park cap makes narrow, and un-expiring a terminal row would break the terminal-state invariant every consumer assumes. First-writer-wins is the doctrine.
8. **executableWorkUnitId ships in this slice.** The outcome-side column exists (`RecommendationOutcome.executableWorkUnitId String?`), `RileyOutcomeRow` already types it, and the insert already writes it; only the value is missing (`outcome-attribution.ts:178` hardcodes null). The write site has `workUnit.id` in scope (verified: the dispatched work unit IS the parked `ExecutableWorkUnit`; `lifecycle-service.ts:285`). Threading: the transition stashes it in `__recommendation.executedWorkUnitId`; `projectBaseCandidate` reads it tolerantly into a new required `AttributableRecommendation.executableWorkUnitId: string | null`; `attributeOneRecommendation` passes it through. Operator-acted rows have no stash and project null, byte-identical to today. Total cost ~15 lines and zero migrations; deferring it would leave the ledger row unable to answer "which execution produced this outcome", the exact provenance the pilot narrative needs.
9. **No new flag, no new env var.** The transition rides the execution gates (env kill switch + per-org `pauseSelfExecutionEnabled` + mandatory approval): it can only ever fire where an execution fired. A separate transition switch would create a mode where Riley executes but the ledger cannot see it, which is the precise dishonest state this slice exists to remove.

## 4. Race safety and idempotency (the matrix)

| Interleaving                                                         | Outcome                                                                                                                                                  |
| -------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Operator acts first, machine second                                  | machine `count === 0`, benign no-op recorded in outputs; rec carries the operator's truth                                                                |
| Machine first, operator second                                       | operator gets `RecommendationStaleStatusError` -> `already_terminal` (existing UX, honest)                                                               |
| Lazy expiry first, machine second                                    | `count === 0` no-op; recorded residue (decision 3.7)                                                                                                     |
| Recovery retry after Meta-write failure                              | first execution never reached the transition; retry pauses and transitions normally                                                                      |
| Re-dispatch after a successful execution                             | pre-read sees `PAUSED`, abstains, never calls the dep; even a degraded pre-read plus idempotent Meta re-write hits `count === 0`. Two independent layers |
| Duplicate parked approvals (cron-retry rec duplicates, distinct ids) | first approval executes and transitions ITS rec; second abstains already-paused and transitions nothing: one Meta pause, one acted row, one outcome row  |
| Concurrent double-dispatch of one work unit                          | lifecycle claim-first idempotency prevents it; if it ever raced, the conditional update still admits exactly one winner                                  |

## 5. The consumer sweep (every `status:"acted"` writer-effect, verified)

New writer to a lifecycle column means grep EVERY reader (`feedback_dual_lifecycle_every_consumer_mode_aware`). All five files that query `pendingActionRecord` (grep-verified, plus the raw SQL in `listPendingForAgent`):

| Consumer                                                      | Reads                                               | Verdict for machine-acted rows                                                                                                                                        |
| ------------------------------------------------------------- | --------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `findAttributableCandidates` (db)                             | acted + resolvedAt + intent prefix + no outcome row | the point of the slice; picks up machine rows, now with `executableWorkUnitId`                                                                                        |
| `findOverlapsForCampaign` (db)                                | acted same-campaign rows                            | a machine-executed pause IS a real confound for a neighboring rec's window; counting it is honest (flags `same_campaign_overlap`)                                     |
| `listResolvedForAgent` -> wins (db -> api -> core)            | acted/confirmed + resolvedAt window                 | machine pause renders "Adjusted. <summary>" in Riley's voice: literally true; undo unavailable (`undoableUntil` null on queue surface), matching rollback-stays-human |
| `listRileyActiveOrgs` (apps/api inngest dispatch, ~line 1232) | distinct orgs with acted rows                       | executed pauses make the org dispatchable for the outcome cron: the point, verified honest                                                                            |
| `applyAct`/`actOnRecommendation` (db/core)                    | terminal guard                                      | operator touching a machine-acted row gets `already_terminal`, the existing honest answer                                                                             |
| `listPendingForAgent` + greeting-signal pending counts (db)   | pending-only                                        | machine-acted rows drop out of queues and inbox-pressure counts: correct, the work is done                                                                            |
| `latestByAgent` (db)                                          | createdAt only                                      | unaffected                                                                                                                                                            |
| `getById`/`listBySurface` -> `routes/recommendations.ts`      | status + actedBy passthrough                        | status "acted" is an existing union member everywhere (dashboard `api-client-types` included); `actedBy` carries the sentinel, no UI consumer renders it              |
| `prisma-workflow-store.ts` `listByStatus`                     | org + status, no intent guard                       | disjoint status vocabulary (`proposed/approved/executing/completed/failed/rejected/expired`); its only live caller passes `"proposed"`; cannot see `"acted"` rows     |

No new status value, no new outcome value, no new enum member is introduced anywhere; the slice reuses the operator-act value with machine provenance, so no #860-class binary-consumer hunt applies beyond this table.

## 6. Architecture (layer doctrine check)

- **packages/db**: `markActedByExecution` on `PrismaRecommendationStore` (+ tests). `projectBaseCandidate` gains the stash read.
- **packages/core**: `AttributableRecommendation.executableWorkUnitId: string | null` (required field, so every constructor site decides); `attributeOneRecommendation` passes it through. Core imports nothing new; the value arrives as plain data.
- **apps/api**: executor deps + success-leg call + outputs enrichment; `RILEY_PAUSE_EXECUTION_RESOLVED_BY` constant; bootstrap closure in `riley-pause-executor.ts` wiring the dep over `PrismaRecommendationStore`; pause-lifecycle world + tests.
- **ad-optimizer, schemas, dashboard, chat**: zero diff. No migration (`db:check-drift` clean). No new route (`check-routes` clean; baseline verified before the change).

**Doctrine note for the PR body.** The transition is not a new mutating surface: the work unit already entered through `PlatformIngress.submit()`, was governance-gated, parked for mandatory approval, and dispatched; the rec-row status update is a post-execution truth record on that same governed flow, exactly like the WorkTrace write beside it ("outcome-linked traces matter" is a listed architecture invariant). The operator path (`applyAct`) is the same class of store write invoked from its governed surface.

## 7. Test doctrine

- **db (mocked Prisma, CI has no Postgres; mirror `prisma-workflow-store.test.ts`):** transition happy path pins the exact conditional WHERE (id + org + status pending + intent prefix; mocked Prisma cannot evaluate predicates, so the WHERE-shape assertions ARE the cross-org and non-recommendation-row pins, stated as named tests) and the write (acted, resolvedAt = executedAt, resolvedBy = given sentinel, merged params preserving BOTH `__recommendation` siblings (`note`/`riskContract`) AND top-level parameter siblings (e.g. `source`) while adding `executedWorkUnitId`); `count === 0` returns not_pending with NO audit write and no throw; missing row returns not_found; DB errors propagate; audit row written only on success with snapshot carrying `executableWorkUnitId`. Candidate projection: machine-acted row projects `executableWorkUnitId`; operator row projects null.
- **core:** `attributeOneRecommendation` passes `executableWorkUnitId` through to the row (machine and operator variants). Lives beside the 4d corroboration tests (the original outcome-attribution test file sits at the eslint ceiling).
- **apps/api executor unit suite (existing file, split if it nears the test-file line ceiling):** success leg calls the dep exactly once with `{ organizationId, recommendationId, executableWorkUnitId: workUnit.id, executedAt }`; EVERY non-success leg never calls it (one pin per leg, the per-leg never-transition table, the already-paused pin named explicitly); dep no-op legs record `not_pending` and `not_found` DISTINCTLY in outputs; dep throw preserves the success result, records `recommendationTransition: "error"`, and emits the greppable console.error.
- **apps/api end-to-end (the #860 lesson: gate-decision tests do not cover the post-approval leg).** Extend the pause lifecycle world with an in-memory recommendation row honoring the same conditional contract, then trace submit -> park -> approve -> execute -> transition: row flips to acted with resolvedAt = execution clock (the e2e approves 20h after the park; the unit suite pins the 47h case), resolvedBy = sentinel, stash = the trace's workUnitId with an explicit row/trace agreement assertion (`row.executedWorkUnitId === trace.workUnitId === lifecycle.actionEnvelopeId`), and the row now satisfies the candidates predicate (acted + resolvedAt + intent prefix). Negative legs: reject leaves the row pending; Meta-write failure -> recovery -> retry transitions exactly once; operator-preempted row no-ops while the pause still completes.
- **Eval gates:** `pnpm eval:riley` (12+10+6) and `pnpm eval:governance` (26) byte-identical; the outcome path has no import contact with `evals/` (4c/4d proved the property; re-run anyway).

## 8. Scope fence

ONE PR. In: the db method + sentinel constant + executor dep/success-leg call + outputs enrichment + bootstrap wiring + candidate/threading change + the test doctrine above + this spec + the implementation plan. Out: any UI (the deferred results dashboard stays deferred), any migration, any new env var or flag, refresh_creative or any second action class, observed-paused readback, the 4e late-interval read, resume/rollback machinery, trust-weighting consumers, backfill of historical executed pauses (none exist in production: flags are OFF everywhere, so there is nothing to backfill), un-expiring the lazy-expired corner (recorded residue, decision 3.7), the manual-preempt `resolved_by_external` cleanup rule (recorded residue, decision 3.2), richer not_pending cause discrimination (operator-acted vs dismissed vs expired; a future hardening option, the contract deliberately stays coarse).

**Claim discipline for the PR body.** This slice makes executed Riley pauses ATTRIBUTABLE and LINKABLE to the executing work unit (steps: rec becomes acted -> candidate found -> outcome row carries executableWorkUnitId). It does NOT itself prove executed pauses earn `corroborated`: that remains the 4d predicate's call over the resulting rows (windows, account spend, booking floors). The PR body must say the former and never the latter; `not_pending` is documented as "first writer won; the pause still executed".

**Adversarial-review hardening note (recorded, not exploitable today).** The stash is trustworthy because its ONLY writer couples it atomically to the status flip; the theoretical hole is a future emitter placing `__recommendation.executedWorkUnitId` at creation time, which `applyAct`'s sibling-preserving merge would carry onto an operator-acted row (fabricated provenance). Not realizable now (emit.ts never sets it; `parameters` are not externally influenced). If the emit surface ever accepts externally-influenced parameters, strip or reject this key at emit time.
