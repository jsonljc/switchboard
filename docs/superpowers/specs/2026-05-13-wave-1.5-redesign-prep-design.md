# Wave 1.5 — redesign prep (editorial conventions doc + agentKey route split)

**Status:** design spec
**Date:** 2026-05-13
**Author:** Jason (with Claude)
**Target branches:** `main` for both deliverables (per CLAUDE.md branch doctrine — specs/plans land on `main` via focused PRs)
**Implementation plan:** to be written after this spec is approved

## Why this batch exists

Six editorial dashboard redesigns are about to fork into parallel worktrees: `/activity`, `/reports`, `/approvals`, `/mission`, `/contacts`, and Alex+Riley home pages. Each has its own design prompt under `docs/design-prompts/` and its own locked HTML+JSX mockup under `docs/design-prompts/locked/switchboard/project/`. Two risks block clean parallel work:

1. **Visual drift.** Six worktrees each make ~10 micro-decisions (badge weight, mono usage, hairline density, empty-state copy, focus rings, motion timing, table density) the others can't see. Six divergent agent-home pages and approvals/mission/reports/activity/contacts pages would ship.
2. **Source-file collision.** Today Alex and Riley both render through one file: `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`. Wave 2 has Alex home and Riley home in separate worktrees that would race on this file.

Wave 1.5 is two small PRs to `main` that close both risks before the wave-2 fork:

- **PR 1 — Editorial conventions doc.** Codifies the implicit conventions in the locked mockups so six agents converge on the same patterns.
- **PR 2 — Route split.** Replaces the `[agentKey]` dynamic route with explicit `/alex` and `/riley` pages backed by a shared shell component.

Both PRs are surgical. Neither introduces visual, token, behavioral, or backend change. PR 1 lands first so wave-2 worktrees fork against locked conventions.

## Out of scope

- Any wave-2 visual change.
- Any new `--*` token or change to existing token values.
- Any change to `useAgent*` hooks, block components, query keys, or API routes.
- Mercury Tools conventions. Mercury surfaces already shipped with their own doctrine; this batch is editorial-register only (see PR 1 §1).
- Mira's home page. Mira remains an `AgentKey` in `@switchboard/schemas` (canonical names locked per project memory), but `/mira` continues to return Next 404 — adding her route is a future PR.

## PR 1 — Editorial conventions doc

### 1.1 What it is, in one paragraph

A single Markdown file at `docs/design-prompts/editorial-conventions.md` that locks the shared design decisions for editorial-register surfaces (Alex home, Riley home, /mission, /approvals, /reports, /activity, /contacts). Every rule cites a locked-mockup source. Wave-2 worktrees read this doc before designing; any new convention discovered during a redesign comes back here, not into the per-surface PR.

### 1.2 Register scope

This doc is **editorial-register only**.

Switchboard's design doctrine splits into two registers that are never mixed within a surface:

- **Editorial** — agent homes, marketing site, Decision Card, /reports. Source Serif 4 prose, JetBrains Mono folios, warm cream + ink + hairline tokens (`--cream`, `--ink`, `--ink-2..4`, `--hairline`, `--hair`, `--hair-soft`, `--editorial-accent`).
- **Mercury / Tools tier** — currently /activity, /contacts, /automations, /reports's previous incarnation. Uses `--mercury-*` tokens.

Mercury Tools surfaces already shipped with their own conventions. This doc does not re-litigate Mercury and is not the source of truth for Mercury work.

### 1.3 Register boundaries (the two surfaces that are migrating)

Two of the six wave-2 surfaces are crossing the register boundary from Mercury into editorial:

- **/activity** — currently at `apps/dashboard/src/app/(auth)/(mercury)/activity/`. Locked design is `locked/switchboard/project/activity-v2/Activity.html`.
- **/contacts** — currently at `apps/dashboard/src/app/(auth)/(mercury)/contacts/`. Locked design is `locked/switchboard/project/agent-home-v3/Pipeline.html` (shares the agent-home-v3 shell — the strongest possible signal that contacts is editorial now).

The doc names these two explicitly so their wave-2 worktrees know they are no longer Mercury and must not consume `--mercury-*` tokens. The route-group migration (moving the directories out of `(mercury)/`) is a wave-2 decision, not a wave-1.5 one — this doc only states the register intent.

### 1.4 Authoring rules (binding on the implementer)

1. **Derive every rule from a locked-mockup citation.** Format: ``locked/switchboard/project/<dir>/<file>:<line-range>``. The locked mockup is the source of truth; opinion isn't.
2. **When the locked mockups disagree across surfaces, adjudicate explicitly.** Name the disagreement in one sentence, pick a winner, give a one-line rationale. Example acceptable form: *"approvals-v2 uses 1px hairlines on row dividers, agent-home-v3 uses 0.5px on hairline-soft surfaces. Agent-home-v3 wins because cockpit prose density is higher; approvals can adopt 0.5px without visible loss. (approvals-v2/Approvals.html:NN; agent-home-v3/cockpit.jsx:NN.)"* Never paper over disagreement.
3. **Prescriptive, not exhaustive.** Lock the decisions that vary across the six worktrees. Don't re-specify per-surface designs.
4. **Doc is derived, not edited.** Header notes: "This file is derived from the lock at `docs/design-prompts/locked/`. Re-derive when the lock refreshes; don't hand-edit individual rules." New conventions found mid-redesign come back here as a PR to `main`, not as inline comments in a wave-2 surface PR.

### 1.5 Required sections

Each section is one short subsection, ~5–15 bullets, with citations on each rule.

1. **Token registry.** Inventory the four overlapping token namespaces in `apps/dashboard/src/app/globals.css`:
   - `--sw-*` — Stone & Weight marketing-site tokens (Stone & Weight register, **not** editorial).
   - `--mercury-*` — Mercury Tools tier (not editorial).
   - `--cream`, `--ink`, `--ink-2..4`, `--hairline`, `--hair`, `--hair-soft`, `--editorial-accent`, `--serif`, `--mono`, `--col`, `--col-wide`, `--gap-section*` — editorial register.
   - shadcn HSL components (`--background`, `--foreground`, `--accent`, `--operator`, `--positive`, `--caution`, `--negative`, etc.) — neutral, consumed by both registers via Tailwind utilities.
   - Rule for new tokens: a value reused across two editorial surfaces must be promoted to a token in the PR that introduces the second use; name conflicts with existing tokens are forbidden; new tokens land in the same PR that consumes them.
2. **Type stacks.** When Instrument Sans (`--font-display`) vs Inter (`--font-sans`) vs Source Serif 4 (`--font-serif`) vs JetBrains Mono (`--font-mono-editorial`). Editorial surfaces lead with serif for prose, mono for folios/numerics, sans for body/UI, display sparingly for hero. Derive from agent-home-v3 cockpit + reports-v2 + approvals-v2.
3. **Badges and pills.** Shape, density, weight, color discipline. Status pills (Alex/Riley state colors), risk badges, agent-name chips. Pull from cockpit pills + approvals risk badges in the locked files.
4. **Accent-color discipline.** When `--editorial-accent` is appropriate; when Alex amber (`#B8782E`) or Riley clay (`#B86C50`) is owned by an agent context; when shadcn `--operator` amber applies. Anti-list (where accent is NEVER used — success states, links, focus rings unless explicitly noted).
5. **Empty / loading / error / stale states.** Copy register, hairline density, placement of the freshness chip. Stale-data marker pattern (already present in `freshness.dataSource`).
6. **Motion timings and easings.** Use existing `--duration-*` and `--ease-*`. State where new motion lives (`globals.css` for shared, component-local CSS for surface-specific).
7. **Focus rings.** Default ring, hover-vs-focus distinction, ring color on cream-on-cream surfaces.
8. **Table density.** Row height, hairline weight, column-header treatment, hover affordance. Source: reports-v2 + activity-v2 tables.
9. **Anti-patterns.** Concrete list, each with one-line *because*. Examples (final list comes from the locked mockups during implementation):
   - "No `--mercury-*` tokens on editorial surfaces — registers don't mix."
   - "No shadcn `Card` shadow on editorial surfaces — hairlines, not shadows."
   - "No `--sw-accent` outside marketing routes — it's a marketing token."
   - "Agent accent (Alex amber / Riley clay) doesn't leave its own agent context."

### 1.6 Length and shape

Target ~400–600 lines including code-fenced examples. The doc is the canonical reference an implementer reads once at the start of their worktree session; readability matters more than completeness.

### 1.7 Acceptance for PR 1

- File exists at `docs/design-prompts/editorial-conventions.md`.
- Header names register scope, the locked-mockup derivation rule, and the disagreement-adjudication rule.
- Section 1.3's two register-boundary surfaces (/activity, /contacts) are named explicitly.
- All nine required sections present.
- Every rule has at least one `locked/switchboard/project/<dir>/<file>:<line-range>` citation.
- Where mockups disagreed, the disagreement is named and resolved with rationale.
- Markdown lint passes.
- No code change in this PR.

## PR 2 — Route split (replace `[agentKey]` with explicit `/alex` and `/riley`)

### 2.1 Goal and constraint

Eliminate the file-level race that wave-2 Alex and Riley home worktrees would have on `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`. Surgical: no visual change, no user-visible behavioral change (see §2.5 for the one internal difference: bogus slugs now hit Next's default 404 instead of a custom one), no token change, no hook change.

### 2.2 Approach — option (c) hybrid

Three options were considered:
- **(a)** Keep `[agentKey]`, extract per-agent config files. Rejected: both worktrees still touch the same client file.
- **(b)** Two fully duplicated pages with no shared code. Rejected: throws away the shared-shell model that the locked design and current implementation validate; near-certain visual drift.
- **(c)** Shared shell component + thin per-agent route shells. **Selected.** Matches the locked design's mental model (one `app.jsx` shell + per-agent `alex-config.jsx` / `riley-config.jsx`), eliminates the `[agentKey]` foot-gun, and aligns with the canonical-names doctrine.

### 2.3 File operations

1. **Move and rename the shell.**
   - From: `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`
   - To: `apps/dashboard/src/components/agent-home/agent-home-shell.tsx`
   - Export rename: `AgentHomeClient` → `AgentHomeShell`.
   - Internal logic: unchanged. Same five blocks, same hooks, same `agentKey` prop, same loading guard.
   - Test moves alongside: `__tests__/agent-home-client.test.tsx` → `apps/dashboard/src/components/agent-home/__tests__/agent-home-shell.test.tsx`. Rename references; assertions unchanged.

2. **Create `/alex/page.tsx`.**
   - Path: `apps/dashboard/src/app/(auth)/alex/page.tsx`.
   - Server component. Calls `fetchEnabledAgentsServer()`; if `alex` is not enabled, `notFound()`.
   - Renders `<EditorialAuthShell><AgentHomeShell agentKey="alex" /></EditorialAuthShell>`.
   - Sibling `__tests__/page.test.tsx` covers two cases: enabled → renders shell with `agentKey="alex"`; disabled → `notFound()`.

3. **Create `/riley/page.tsx`.**
   - Path: `apps/dashboard/src/app/(auth)/riley/page.tsx`.
   - Same structure as `/alex/page.tsx`, hard-coded to `"riley"`.
   - Sibling `__tests__/page.test.tsx` mirrors Alex's two cases.

4. **Delete `[agentKey]/`.**
   - Remove the directory in full: `page.tsx`, `agent-home-client.tsx`, `__tests__/page.test.tsx`, `__tests__/agent-home-client.test.tsx`, `__tests__/route-allowlist.test.ts`.
   - The route-allowlist test's purpose was to guard concrete top-level routes from being claimed by `[agentKey]`. With no dynamic segment, the guard is obsolete. (If desired, the implementer may add a one-line assertion in an existing dashboard route test that `/alex` and `/riley` resolve via concrete directories — optional.)

### 2.4 Things intentionally untouched

- `AGENT_KEYS` and `AgentKey` type in `@switchboard/schemas`. Hooks still take `AgentKey` for type safety.
- All `useAgent*` hooks (`useAgentGreeting`, `useAgentWins`, `useAgentMetrics`, `useAgentPipeline`).
- All five block components under `apps/dashboard/src/components/agent-home/`.
- `apps/dashboard/src/components/layout/app-shell.tsx`'s `SHELL_OWNED_PREFIXES` and `ONBOARDING_GATE_EXEMPT_EXACT` — both already enumerate `/alex` and `/riley` explicitly.
- `apps/dashboard/src/components/dev/dev-panel.tsx`'s hard-coded `/alex` and `/riley` entries.
- All API routes under `/api/dashboard/agents/[agentId]/*` — server-side `[agentId]` is unrelated to the editorial URL.
- Query keys in `apps/dashboard/src/lib/query-keys.ts`.

### 2.5 Behavior parity

| URL | Before | After |
|---|---|---|
| `/alex` | `[agentKey]/page.tsx` with `agentKey="alex"`; renders shell | `/alex/page.tsx` renders shell with `agentKey="alex"` |
| `/riley` | Same with `"riley"` | `/riley/page.tsx` renders shell with `"riley"` |
| `/mira` | `[agentKey]` gate calls `notFound()` (mira not enabled) | Next standard 404 (no file) |
| `/bogus` | `[agentKey]` calls `notFound()` (not in `AGENT_KEYS`) | Next standard 404 |
| Mercury routes | Concrete dirs beat `[agentKey]` (guarded by test) | Concrete dirs (no dynamic to compete) |

User-visible behavior is identical for valid URLs and for Mira. Bogus agent slugs (e.g., `/foo`) go from a custom 404 to Next's default 404; same response code, same lack-of-content.

### 2.6 Verification

Run from the worktree:

- `pnpm --filter @switchboard/dashboard typecheck`
- `pnpm --filter @switchboard/dashboard test`
- `pnpm --filter @switchboard/dashboard build` (per CLAUDE.md memory: dashboard `next build` is **not** in CI; `.js`-extension import regressions slip past — must run locally)

Manual:
- `pnpm dev`, hit `/alex` and `/riley` in browser, confirm rendered output matches pre-split visually.
- Hit `/mira` and `/bogus`, confirm 404.

### 2.7 Acceptance for PR 2

- `apps/dashboard/src/app/(auth)/[agentKey]/` directory does not exist.
- `apps/dashboard/src/app/(auth)/alex/page.tsx` and `apps/dashboard/src/app/(auth)/riley/page.tsx` exist, each with sibling test.
- `apps/dashboard/src/components/agent-home/agent-home-shell.tsx` exists and exports `AgentHomeShell`.
- All three verification commands pass.
- Manual `/alex` and `/riley` smoke test matches pre-split.
- PR diff contains only file moves, two new `page.tsx` files, and the renamed test — no other change.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Conventions doc prescribes rules the locked mockups don't actually encode | Authoring rule §1.4.1: every rule cites a locked file; reviewer rejects un-cited rules |
| Conventions doc papers over real disagreements across surfaces | Authoring rule §1.4.2: name disagreements explicitly with rationale |

| Mira gets silently re-introduced | Out-of-scope §, behavior-parity table §2.5, reviewer treats `/mira` as out-of-scope |
| Conventions doc tries to span Mercury and editorial registers | §1.2 scope explicit; §1.3 names the two boundary surfaces |
| Mockups under `agent-home-v3` apply to Pipeline (=/contacts), which is migrating from Mercury | §1.3 calls this out so the /contacts worktree knows it's now editorial |
