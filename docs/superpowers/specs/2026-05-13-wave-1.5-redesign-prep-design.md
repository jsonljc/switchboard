**Status:** design spec

## Why this batch exists

Six editorial dashboard redesigns are about to fork into parallel worktrees: `/activity`, `/reports`, `/approvals`, `/mission`, `/contacts`, and Alex+Riley home pages. Each has its own design prompt under `docs/design-prompts/` and its own locked HTML+JSX mockup under `docs/design-prompts/locked/switchboard/project/`. Two risks block clean parallel work:

1. **Visual drift.** Six worktrees each make ~10 micro-decisions (badge weight, mono usage, hairline density, empty-state copy, focus rings, motion timing, table density) the others can't see. Six divergent agent-home pages and approvals/mission/reports/activity/contacts pages would ship.
2. **Source-file collision.** Today Alex and Riley both render through one file: `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`. Wave 2 has Alex home and Riley home in separate worktrees that would race on this file.

Wave 1.5 is two small PRs to `main` that close both risks before the wave-2 fork:

- **PR 1 ΓÇË Wave-1 surface conventions doc.** Codifies the shared visual/interaction language across the six surfaces so they converge on the same paper-and-ink patterns, without forcing any surface to migrate its route group or token base.
- **PR 2 ΓÇË Route split.** Replaces the `[agentKey]` dynamic route with explicit `/alex` and `/riley` pages backed by a shared shell component.

Both PRs are surgical. Neither introduces visual, token, behavioral, or backend change. PR 1 lands first so wave-2 worktrees fork against locked conventions.

## Out of scope

- Any wave-2 visual change.
- Any new `--*` token or change to existing token values.
- Any change to `useAgent*` hooks, block components, query keys, or API routes.
- **Route-group migrations and token-namespace migrations.** The conventions doc covers shared visual/interaction language only. Whether a surface moves out of `(mercury)/` or stops aliasing `--mercury-*` tokens is each surface's own decision and ships as its own PR.
- Mira's home page. Mira remains an `AgentKey` in `@switchboard/schemas` (canonical names locked per project memory), but `/mira` continues to return Next 404 ΓÇË adding her route is a future PR.

## PR 1 ΓÇË Wave-1 surface conventions doc

### 1.1 What it is, in one paragraph

A single Markdown file at `docs/design-prompts/wave-1-surface-conventions.md` that locks the shared visual and interaction language across the six wave-2 dashboard surfaces (Alex home, Riley home, /mission, /approvals, /reports, /activity, /contacts). The doc separates two layers: **Layer 1 ΓÇË shared visual/interaction language** (typography, hairlines, badge geometry, empty states, motion, focus rings, table density, surface-specific governance conventions, anti-patterns) ΓÇË applies to all six surfaces; and **Layer 2 ΓÇË surface token ownership** ΓÇË names which token base each surface currently uses *without forcing any migration*. Wave-2 worktrees read this doc before designing; any new convention discovered during a redesign comes back here, not into the per-surface PR.

### 1.2 Doc scope ΓÇË visual language vs token ownership

This doc covers shared visual and interaction conventions for the six wave-2 surfaces. It does **not** replace per-surface token ownership and it does **not** authorize route-group migrations.

- **Layer 1 ΓÇË Visual/interaction language.** Typography stacks, prose cadence, hairline density, badge/pill geometry, empty/loading/error/stale patterns, motion timings, focus rings, table density, surface-specific governance conventions (e.g., /approvals confirmation codes), anti-patterns. Binding on all six surfaces.
- **Layer 2 ΓÇË Surface token ownership.** Names the token base each surface currently consumes. Mercury-routed surfaces (`/approvals`, `/activity`, `/contacts`, and `/reports` via aliases) continue to alias from `--mercury-*` until a separate, explicitly-approved migration. Agent homes (`/alex`, `/riley`) consume editorial tokens directly. `/reports` keeps its muted-operator-amber accent exception (per its own surface spec ΓÇË `/reports` reads as a printed renewal statement, not an operational action surface).

A Mercury-routed surface may adopt Layer-1 conventions while continuing to alias `--mercury-*`. The two layers are independent. **This doc never instructs a surface to change its route group or token base.**

### 1.4 Authoring rules (binding on the implementer)

1. **Every rule cites either a locked-mockup file/line or an approved surface spec section.** Format: `locked/switchboard/project/<dir>/<file>:<line-range>` *or* `docs/design-prompts/<surface-spec>.md#<section-anchor>`. Production-constraint rules (e.g., "Source Serif 4 not Cormorant for prose", "no second sticky topbar", "backend emits dollars not cents", "no raw agent IDs in operator-visible copy", "no new global tokens without promotion") come from surface specs and production code, not from mockup lines ΓÇË both citation forms are valid.
2. **When sources disagree across surfaces, adjudicate explicitly.** Name the disagreement in one sentence, pick a winner, give a one-line rationale. Example: *"approvals-v2 uses 1px hairlines on row dividers, agent-home-v3 uses 0.5px on hairline-soft surfaces. Agent-home-v3 wins because cockpit prose density is higher; approvals can adopt 0.5px without visible loss. (approvals-v2/Approvals.html:NN; agent-home-v3/cockpit.jsx:NN.)"* Never paper over disagreement.
3. **Prescriptive, not exhaustive.** Lock the decisions that vary across the six worktrees. Don't re-specify per-surface designs.
4. **No route or token migrations.** This doc never instructs a surface to move out of `(mercury)/` or to stop aliasing `--mercury-*` tokens. Surface-specific token migrations land as their own PRs.
5. **Doc is derived, not edited.** Re-derive when the lock or surface specs refresh; don't hand-edit individual rules. New conventions discovered mid-redesign come back here as a focused PR to `main`.

### 1.5 Required sections

The doc has 10 sections. Sections 1ΓÇô8 are Layer-1 shared visual language. Section 9 is Layer-1 surface-specific conventions. Section 10 is Layer-1 anti-patterns. (Layer-2 surface token ownership is captured in ┬¯1.3 of this spec and reproduced once in the doc as a reference table; it is not a section of binding rules.) Each section is one short subsection, ~5ΓÇô15 bullets, with citations.

1. **Token registry (reference, not prescription).** Inventory the four overlapping token namespaces in `apps/dashboard/src/app/globals.css`:
   - `--sw-*` ΓÇË Stone & Weight marketing-site tokens.
   - `--mercury-*` ΓÇË Mercury Tools tier; currently aliased into `/approvals`, `/activity`, `/contacts`, `/reports`.
   - `--cream`, `--ink`, `--ink-2..4`, `--hairline`, `--hair`, `--hair-soft`, `--editorial-accent`, `--serif`, `--mono`, `--col`, `--col-wide`, `--gap-section*` ΓÇË editorial; currently consumed directly by agent homes.
   - shadcn HSL components (`--background`, `--foreground`, `--accent`, `--operator`, `--positive`, `--caution`, `--negative`, ΓÇ³) ΓÇË neutral, consumed by both registers via Tailwind utilities.
   Rule for new tokens: a value reused across two surfaces must be promoted to a token in the PR that introduces the second use; name conflicts with existing tokens are forbidden; new tokens land in the same PR that consumes them. Do not prescribe which surfaces consume which namespace ΓÇË that's the ┬¯1.3 table.
2. **Type stacks.** When Instrument Sans (`--font-display`) vs Inter (`--font-sans`) vs Source Serif 4 (`--font-serif`) vs JetBrains Mono (`--font-mono-editorial`). All six surfaces lead with serif for prose, mono for folios/numerics, sans for body/UI, display sparingly for hero. Derive from agent-home-v3 cockpit + reports-v2 + approvals-v2.
3. **Badges and pills.** Shape, density, weight, color discipline. Status pills (Alex/Riley state colors), risk badges, agent-name chips. Pull from cockpit pills + approvals risk badges in the locked files.
4. **Accent-color discipline.** When `--editorial-accent` is appropriate; when Alex amber (`#B8782E`) or Riley clay (`#B86C50`) is owned by their respective agent context; when shadcn `--operator` amber applies; **and the /reports muted-operator-amber exception** with rationale (renewal-statement surface). Anti-list: accent is NEVER used for success states, links, or focus rings unless explicitly noted.
5. **Empty / loading / error / stale states.** Copy register, hairline density, placement of any freshness chip. Stale-data marker pattern (already present in `freshness.dataSource` returned by `useAgent*` hooks).
6. **Motion timings and easings.** Use existing `--duration-*` and `--ease-*` from `globals.css`. State where new motion lives (`globals.css` for shared across two+ surfaces, component-local CSS for surface-specific).
7. **Focus rings.** Default ring, hover-vs-focus distinction, ring color on cream-on-cream surfaces.
8. **Table density.** Row height, hairline weight, column-header treatment, hover affordance. Sources: reports-v2 + activity-v2 tables.
9. **Surface-specific conventions.** Sub-subsections for the three surfaces with governance vocabulary the other surfaces don't share:
   - **/approvals ΓÇË governed action surface.** Confirmation-code language (operator vocabulary, not engineering); amber CTA reserved for mutating commit only; risk-graded confirmation visual; no inline approve outside `/approvals`. Cite the /approvals surface spec + approvals-v2 mockup.
   - **/reports ΓÇË renewal statement.** Muted operator amber accent; "printed statement" framing; no real-time freshness chip on monthly figures. Cite the /reports surface spec + reports-v2 mockup.
   - **/activity ΓÇË Tools-tier paper-and-ink convergence.** Adopts editorial visual language while continuing to live under `(mercury)/activity/` and alias existing tokens. Cite the /activity surface spec + activity-v2 mockup.
10. **Anti-patterns.** Bullets in the form `**Don't [X].** Because [Y].` Required entries:
    - Don't use this doc to make route-group migrations or to force a surface to drop its current token aliases ΓÇË those are separate PRs with explicit approval.
    - Don't put a shadcn `Card` shadow on any wave-2 surface ΓÇË paper-and-ink uses hairlines, not shadows.
    - Don't use `--sw-accent` outside marketing.
    - Don't use Alex amber (`#B8782E`) outside Alex's agent context.
    - Don't use Riley clay (`#B86C50`) outside Riley's agent context.
    - Don't use the editorial or operator accent for success states.
    - Don't use the editorial or operator accent for links unless explicitly noted in ┬¯4.
    - Don't introduce a new `--duration-*` value inline ΓÇË promote to a token in `globals.css` first.
    - Don't write rules without a citation ΓÇË un-cited rules are reviewer-rejected.
    Plus 2ΓÇô4 surface-specific anti-patterns derived from the locked mockups and surface specs during this task.

## PR 2 ΓÇË Route split (replace `[agentKey]` with explicit `/alex` and `/riley`)

### 2.1 Goal and constraint

Eliminate the file-level race that wave-2 Alex and Riley home worktrees would have on `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`. Surgical: no visual change, no user-visible behavioral change (see ┬¯2.5 for the one internal difference: bogus slugs now hit Next's default 404 instead of a custom one), no token change, no hook change.

### 2.2 Approach ΓÇË option (c) hybrid

Three options were considered:
- **(a)** Keep `[agentKey]`, extract per-agent config files. Rejected: both worktrees still touch the same client file.
- **(b)** Two fully duplicated pages with no shared code. Rejected: throws away the shared-shell model that the locked design and current implementation validate; near-certain visual drift.
- **(c)** Shared shell component + thin per-agent route shells. **Selected.** Matches the locked design's mental model (one `app.jsx` shell + per-agent `alex-config.jsx` / `riley-config.jsx`), eliminates the `[agentKey]` foot-gun, and aligns with the canonical-names doctrine.

### 2.3 File operations

1. **Move and rename the shell.**
   - From: `apps/dashboard/src/app/(auth)/[agentKey]/agent-home-client.tsx`
   - To: `apps/dashboard/src/components/agent-home/agent-home-shell.tsx`
   - Export rename: `AgentHomeClient` Γ¶Ê `AgentHomeShell`.
   - Internal logic: unchanged. Same five blocks, same hooks, same `agentKey` prop, same loading guard.
   - Test moves alongside: `__tests__/agent-home-client.test.tsx` Γ¶Ê `apps/dashboard/src/components/agent-home/__tests__/agent-home-shell.test.tsx`. Rename references; assertions unchanged.

2. **Create `/alex/page.tsx`.**
   - Path: `apps/dashboard/src/app/(auth)/alex/page.tsx`.
   - Server component. Calls `fetchEnabledAgentsServer()`; if `alex` is not enabled, `notFound()`.
   - Renders `<EditorialAuthShell><AgentHomeShell agentKey="alex" /></EditorialAuthShell>`.
   - Sibling `__tests__/page.test.tsx` covers two cases: enabled Γ¶Ê renders shell with `agentKey="alex"`; disabled Γ¶Ê `notFound()`.

3. **Create `/riley/page.tsx`.**
   - Path: `apps/dashboard/src/app/(auth)/riley/page.tsx`.
   - Same structure as `/alex/page.tsx`, hard-coded to `"riley"`.
   - Sibling `__tests__/page.test.tsx` mirrors Alex's two cases.

4. **Delete `[agentKey]/`.**
   - Remove the directory in full: `page.tsx`, `agent-home-client.tsx`, `__tests__/page.test.tsx`, `__tests__/agent-home-client.test.tsx`, `__tests__/route-allowlist.test.ts`.
   - The route-allowlist test's purpose was to guard concrete top-level routes from being claimed by `[agentKey]`. With no dynamic segment, the guard is obsolete. (If desired, the implementer may add a one-line assertion in an existing dashboard route test that `/alex` and `/riley` resolve via concrete directories ΓÇË optional.)

### 2.4 Things intentionally untouched

- `AGENT_KEYS` and `AgentKey` type in `@switchboard/schemas`. Hooks still take `AgentKey` for type safety.
- All `useAgent*` hooks (`useAgentGreeting`, `useAgentWins`, `useAgentMetrics`, `useAgentPipeline`).
- All five block components under `apps/dashboard/src/components/agent-home/`.
- `apps/dashboard/src/components/layout/app-shell.tsx`'s `SHELL_OWNED_PREFIXES` and `ONBOARDING_GATE_EXEMPT_EXACT` ΓÇË both already enumerate `/alex` and `/riley` explicitly.
- `apps/dashboard/src/components/dev/dev-panel.tsx`'s hard-coded `/alex` and `/riley` entries.
- All API routes under `/api/dashboard/agents/[agentId]/*` ΓÇË server-side `[agentId]` is unrelated to the editorial URL.
- Query keys in `apps/dashboard/src/lib/query-keys.ts`.
