# `/contacts/[id]` (Slice D1.5) — Design Spec

_2026-05-09 · part of the agent-first redesign track · Phase D, surface 1.5 (Tools tier, Mercury register) · follows D1 (`docs/superpowers/specs/2026-05-08-contacts-d1-design.md`)_

> **Reading posture:** D1.5 inherits the Mercury vocabulary, production-gate pattern, and surface-agnostic backend rules already locked by D1. This spec describes only what is new: the read-only detail page, its composite endpoint, and the route-availability flip. Decisions ledger §2.0 is binding.

---

## 1. Problem & scope

### 1.0 One-line scope

`/contacts/[id]` is a read-only inspection surface for a single `Contact`. It is the natural follow-on to D1's browse list, closes the existing dead-end `/contacts/:id` link, and flips `ROUTE_AVAILABILITY.contact = true`.

### 1.1 What this slice ships

- New route `apps/dashboard/src/app/(auth)/contacts/[id]/page.tsx` rendering a Mercury single-column detail page.
- New backend endpoint `GET /api/dashboard/contacts/:id` (Next.js proxy) → Fastify → new core projection `getContactDetail({orgId, contactId}, deps)` → existing stores (`ContactStore`, `OpportunityStore`, `ThreadStore`, `RecommendationStore`, `HandoffStore`, `LifecycleRevenueEventStore`).
- New shared types in `packages/schemas/src/contacts.ts` for the composite payload (`ContactDetailResponse` and section row shapes).
- A flip of `ROUTE_AVAILABILITY.contact` from `false` → `true` in `apps/dashboard/src/lib/agent-home/resolve-link.ts`. D1's row links and decision-card `/contacts/:id` links go live in the same PR.
- Fixture-mode parity with the D1 list: detail-page fixtures keyed by the three known list fixture ids; unknown id → 404 even in fixture mode.

### 1.2 What this slice does **not** ship

- `/contacts/:id/conversations` and `/contacts/:id/conversations/:threadId` sub-routes (`recommendation-adapter.ts:48`, `handoff-adapter.ts:22`). They stay dead and are closed by a follow-on slice.
- Any **mutation** on a contact: no edit, no archive, no merge, no tag, no assign, no add-note.
- Full decision lineage (acted, dismissed, escalated, completed). D1.5 surfaces only **open** decisions for the contact.
- Owner tasks. No current consumer points at `/contacts/:id` for owner-task context; defer until a use-site requests it.
- Shared Mercury header chrome consolidation. `/contacts/[id]` reuses the `ContactsHeader` shipped in D1 verbatim (no per-contact breadcrumb or back link in v1; the brand cluster + nav is enough orientation).
- Editorial-shell `Tools ▾ / Contacts` link expansion. Operators reach `/contacts/[id]` from D1's list, decision-card links, or direct URL.
- Server-side rendering of the detail payload. D1.5 is a client component using TanStack Query, matching D1's read pattern.

### 1.3 Why this surface, why now

- **D1 already links here.** Every row in D1's list assembles a `detailHref` of `/contacts/:id` (`packages/core/src/contacts/list.ts:63`) but renders the row as `aria-disabled` because `ROUTE_AVAILABILITY.contact === false`. The same is true for `apps/dashboard/src/lib/agent-home/resolve-link.ts:22`. Both are dead today; D1.5 lights them up in one PR.
- **One pattern validates two more.** D2 (`/automations`) and D3 (`/activity`) are list-only Mercury surfaces. D1.5 settles the **detail-page** half of the Mercury vocabulary so D2/D3 don't have to re-litigate it if/when they grow detail surfaces of their own.
- **Open decisions need an inspection home.** When a decision card mentions "Maya R. is asking about pricing," the operator needs a place to land that says "here's Maya — open opportunities, recent threads, what else is pending on her." D1.5 is that landing surface.

### 1.4 Out-of-scope decisions inherited from prior specs

- Mercury design tokens, register split, surface-agnostic backend rule, production-gate pattern — all inherited from D1 §1.4.
- Canonical agent names Alex / Riley / Mira (`memory/project_canonical_agent_names.md`).
- D1's locked answer to OPEN-3 (`ROUTE_AVAILABILITY.contact = false` for D1) is **explicitly superseded by D1.5**. Flipping the flag is the slice's reason for existing.

## 2. Decisions

### 2.0 Decisions ledger (locked 2026-05-09)

| #   | Question                                            | Locked answer                                                                                                                                                                                                                      |
| --- | --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | v1 ambition                                         | **Read-only detail page.** No mutations of any kind in v1.                                                                                                                                                                         |
| 2   | Route coverage                                      | **Only `/contacts/:id` ships in D1.5.** `/contacts/:id/conversations` and `/contacts/:id/conversations/:threadId` stay unavailable and are closed by a follow-on slice.                                                            |
| 3   | Sections rendered                                   | Six, in fixed reading order: **Header → Profile facts → Opportunities → Conversation threads → Open decisions → Revenue events.** No sidebar, no tabs.                                                                             |
| 4   | Layout                                              | **Single-column long-scroll** Mercury read layout. Same vertical rhythm as `/reports` and `/contacts`. Mono-caps section labels via the existing `.section-label` class.                                                           |
| 5   | Backend shape                                       | **One composite endpoint.** `GET /api/dashboard/contacts/:id` returns `{ profile, opportunities, threads, openDecisions, revenueEvents }`. One round-trip, one React Query key, one loading boundary.                              |
| 6   | Internal modularity inside the composite projection | Five **internally separate** builders (`buildContactDetailProfile`, `…Opportunities`, `…Threads`, `…OpenDecisions`, `…RevenueEvents`) composed in `getContactDetail()`. Public API stays composite.                                |
| 7   | "Decision history" framing                          | Renamed **"Open decisions."** Surfaces only **pending** recommendations and handoffs for this contact. Full lineage (acted/dismissed/escalated/completed) is reserved for a future slice.                                          |
| 8   | Open-decisions data source                          | Reuse existing pending-only stores (`recommendationStore.listBySurface({surface:"queue", status:"pending"})` + `handoffStore.listPending(orgId)`) already used by `apps/api/src/routes/decisions.ts`.                              |
| 9   | Open-decisions filtering risk                       | The contactId match must be **schema-guarded and fail closed**: malformed or missing `targetEntities` entries are ignored, not surfaced. No silent type coercion. No partial matches.                                              |
| 10  | Conversation threads behaviour                      | Render as **inert tiles** (`aria-disabled="true"`, no link), same gating pattern D1 list rows use. Tiles show assigned-agent, summary, and last-message time. Lit up by the follow-on slice that ships `/conversations/:threadId`. |
| 11  | Auth + missing/cross-org behaviour                  | Backend returns **404** for missing or cross-org contactId (no info leak about whether the id exists). Page calls Next.js `notFound()`.                                                                                            |
| 12  | Production gate                                     | Same `NEXT_PUBLIC_CONTACTS_LIVE` flag as D1. Off → fixtures; on → live.                                                                                                                                                            |
| 13  | Fixture mode                                        | Detail fixtures **keyed by the three known D1 list fixture ids**, fully populated. Unknown id → 404 even in fixture mode. Lets list → detail clicks work without a live db.                                                        |
| 14  | Route-availability flip timing                      | `ROUTE_AVAILABILITY.contact = true` flips **in the D1.5 PR itself**. D1's row links and the `/contacts/:id` links from `resolve-link.ts:22` go live alongside the page.                                                            |
| 15  | Header chrome                                       | Reuse D1's `ContactsHeader` verbatim. No per-contact breadcrumb, no back-to-list link, no mutation buttons in v1.                                                                                                                  |
| 16  | Render model                                        | **Client component + TanStack Query**, matching D1's `useContactsList` pattern. New `useContactDetail(contactId)` hook with the new query key namespace `contacts.detail(id)`.                                                     |

## 5. Backend (surface-agnostic boundary)

### 5.1 Layered structure

```
Next.js page (client component, useContactDetail)
        │  fetch
        ▼
apps/dashboard/.../api/dashboard/contacts/[id]/route.ts   (Next proxy)
        │  forward
        ▼
apps/api/src/routes/dashboard-contact-detail.ts           (Fastify)
        │  call
        ▼
packages/core/src/contacts/detail.ts
  → getContactDetail({ orgId, contactId }, deps)
        │  composes
        ▼
  buildContactDetailProfile(contact)
  buildContactDetailOpportunities(opportunityStore, …)
  buildContactDetailThreads(threadStore, …)
  buildContactDetailOpenDecisions(recStore, handoffStore, …)
  buildContactDetailRevenueEvents(revenueEventStore, …)
```

### 5.2 Composite projection contract

```ts
export class ContactNotFoundError extends Error {
  readonly code = "CONTACT_NOT_FOUND";
}

export interface ContactDetailDeps {
  contactStore: ContactStore;
  opportunityStore: OpportunityStore;
  threadStore: ThreadStore;
  recommendationStore: RecommendationStore;
  handoffStore: HandoffStore;
  revenueEventStore: LifecycleRevenueEventStore;
}

export async function getContactDetail(
  input: { orgId: string; contactId: string },
  deps: ContactDetailDeps,
): Promise<ContactDetailResponse>;
```

`getContactDetail` first fetches the `Contact` (org-scoped). If it's missing, it throws `ContactNotFoundError`, which the Fastify route maps to **404**. No info leak: cross-org access path is identical to missing-id.

### 5.3 Open-decisions filtering — schema-guarded, fail-closed

The `Recommendation` and `HandoffPackage` records embed `contactId` inside JSON columns (`targetEntities` for recs, `leadSnapshot.leadId` for handoffs). The contactId match in `buildContactDetailOpenDecisions` MUST:

1. Pull org-scoped pending records via the existing store methods (no new SQL JSON predicates in v1).
2. Filter in-memory with **strict** type guards. For recommendations: `typeof targetEntities?.contactId === "string"` and the value equals the requested contactId. For handoffs: `typeof leadSnapshot?.leadId === "string"` and the value equals the requested contactId. Anything else (missing key, wrong type, null, malformed JSON) → record is **silently excluded**, never surfaced.
3. Never coerce. Never trim. Never partial-match. A record either matches exactly or it does not appear.
4. Be covered by tests that pass deliberately-malformed `targetEntities` and `leadSnapshot` shapes (missing key, null, number-where-string, nested object, empty string) and assert exclusion in every case.

This guard prevents sloppy JSON access from producing noisy or cross-contact results. Performance is acceptable because the existing endpoints already cap the org-scoped pending fetch (recs at 50, handoffs at the natural pending count). If the cap is ever hit and the contact's decisions fall outside the slice, we add a store-level `listByContactId` method then — not now.

### 5.4 Fastify route

```ts
// apps/api/src/routes/dashboard-contact-detail.ts
app.get("/api/dashboard/contacts/:id", async (req, reply) => {
  const orgId = await requireOrganizationScope(req, reply);
  if (!orgId) return;

  const { id } = req.params as { id: string };

  try {
    const response = await getContactDetail(
      { orgId, contactId: id },
      {
        contactStore: app.contactStore,
        opportunityStore: app.opportunityStore,
        threadStore: app.threadStore,
        recommendationStore: app.recommendationStore,
        handoffStore: app.handoffStore,
        revenueEventStore: app.lifecycleRevenueEventStore,
      },
    );
    return response;
  } catch (e) {
    if (e instanceof ContactNotFoundError) {
      return reply.code(404).send({ error: "CONTACT_NOT_FOUND" });
    }
    throw e;
  }
});
```

### 5.5 Next.js proxy

Mirrors the D1 list proxy verbatim (`apps/dashboard/src/app/api/dashboard/contacts/route.ts`), at path `apps/dashboard/src/app/api/dashboard/contacts/[id]/route.ts`. Forwards auth headers, returns upstream body unchanged, propagates 404 status.

Each section component takes its own slice of `ContactDetailResponse` and renders Mercury vocabulary. Sections render their own internal empty state (e.g., "No opportunities yet.") rather than collapsing the section heading — the section labels are a navigational map of the page.

### 6.2 `useContactDetail`

```ts
export function useContactDetail(contactId: string) {
  const isLive = process.env.NEXT_PUBLIC_CONTACTS_LIVE === "true";
  const queryKey = useScopedQueryKeys()?.contacts.detail(contactId);

  if (!isLive) {
    const fixture = CONTACT_DETAIL_FIXTURES[contactId];
    if (!fixture) notFound();
    return { data: fixture, isLoading: false, error: null, refetch: () => {} };
  }

  return useQuery({
    queryKey,
    queryFn: async () => {
      const res = await fetch(`/api/dashboard/contacts/${contactId}`);
      if (res.status === 404) notFound();
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return ContactDetailResponseSchema.parse(await res.json());
    },
  });
}
```

The query key namespace appended in `apps/dashboard/src/hooks/use-query-keys.ts`:

```ts
contacts: {
  list: (q: object) => [...base, "contacts", "list", q] as const,
  detail: (id: string) => [...base, "contacts", "detail", id] as const,
}
```

### 6.3 Threads section is inert

Until a follow-on slice ships `/contacts/:id/conversations/:threadId`, every thread tile renders with `aria-disabled="true"`, no link wrapper, and a `title="Conversation view coming next"` tooltip. Same gating shape D1's row gating uses, mediated by `ROUTE_AVAILABILITY.thread` (new key, default `false`). Adding a `thread` key to `ROUTE_AVAILABILITY` is part of D1.5.

### 6.4 Empty / loading / error

| State           | Trigger                                      | Treatment                                                                              |
| --------------- | -------------------------------------------- | -------------------------------------------------------------------------------------- |
| Loading         | First fetch in flight                        | Header skeleton + 5 hairline section skeletons. No tabular content placeholders.       |
| Not found (404) | Backend returned 404                         | `notFound()` → standard `/not-found` route. No bespoke "missing contact" page in D1.5. |
| Error (5xx/net) | Any non-404 failure                          | Inline `<EmptyState variant="error">` at the page level with `[Try again]` → refetch.  |
| Section empty   | Section data is empty (no opps, no threads…) | "No opportunities yet." (etc.) inside the section, label still visible.                |

## 7. Production gate + fixture mode

Same `NEXT_PUBLIC_CONTACTS_LIVE` flag as D1. Off → fixtures; on → live API.

Fixture file `fixtures.ts` defines `CONTACT_DETAIL_FIXTURES: Record<string, ContactDetailResponse>` keyed by the three list-page fixture ids. Each fixture is fully populated (≥ 1 opportunity, ≥ 1 thread, ≥ 1 open decision, ≥ 1 revenue event) so screenshot review covers every section.

Unknown ids in fixture mode → `notFound()`. The fixture mode does NOT pretend every id resolves; otherwise broken navigation goes uncaught during design review.

## 8. Route-availability flip (the headline)

In the same PR:

```diff
// apps/dashboard/src/lib/agent-home/resolve-link.ts
-export const ROUTE_AVAILABILITY = { contact: false, thread: false } as const;
+export const ROUTE_AVAILABILITY = { contact: true, thread: false } as const;
```

(Adding `thread` is the new key D1.5 introduces; defaults `false` per §6.3.)

Effect:

- D1 list rows become live `<Link>` (one branch in `contact-row.tsx`).
- Decision-card `/contacts/:id` links from `resolve-link.ts:22` become live.
- `/contacts/:id/conversations*` links remain dead (gated by `ROUTE_AVAILABILITY.thread`).

The `disabledNotice` D1 renders above its table ("Browse-only for now. Contact detail is coming next.") is **removed** in the same PR, since the notice was conditioned on `!ROUTE_AVAILABILITY.contact`.

## 9. Tests

Three layers, mirroring D1:

- **Schemas** (`packages/schemas/src/__tests__/contact-detail.test.ts`): every section schema parses a happy-path fixture; rejects bad enum / missing required / wrong-type cases.
- **Core projection** (`packages/core/src/contacts/__tests__/detail.test.ts`): each builder tested in isolation against `vi.fn()` store mocks (per `memory/feedback_api_test_mocked_prisma.md`); composite `getContactDetail` tested for happy path, `ContactNotFoundError` on missing/cross-org, and the schema-guarded fail-closed cases for open decisions (malformed `targetEntities`, missing key, wrong type, null).
- **API** (`apps/api/src/__tests__/api-contact-detail.test.ts`): happy 200, 404 for unknown id, 404 for cross-org, 401 without org scope.
- **Frontend** (per-component `__tests__/`): each section renders its slice; `useContactDetail` covers fixture branch + live branch + 404 → `notFound()`; page-level test covers loading / error / not-found state transitions.
