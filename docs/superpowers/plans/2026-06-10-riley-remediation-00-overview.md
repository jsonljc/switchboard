# Riley Capability Audit Remediation: Master Overview

> **This is the orchestration index, not an executable task list.** The six tier plans linked below are the executable plans (each carries the standard agentic-worker header and bite-sized TDD tasks). Read this file first: it holds the verified-findings table, the answered open decisions, the cross-workstream dedup, the dependency graph, and the cross-slice integration review that every tier PR must pass.

**Goal:** Turn Riley from "shipped but production-inert" into an agent that a real pilot clinic can credential, that survives first contact with a live Meta account, whose brain stops missing the worst case, and whose revenue loop closes, without scaling the act-leg past the guards the audit found load-bearing.

**Source audit:** `docs/audits/2026-06-10-riley-capability-audit/` (FINDINGS.md + `domains/D1..D10`). Baseline `origin/main @ 84083f0c`. All findings below were **re-verified at file:line against current `main`** on 2026-06-10 (see status column); nothing is planned on stale evidence.

**Branch doctrine:** these plans land on `main` as a focused docs PR. Implementation branches then _consume_ these plans; they do not re-accumulate planning docs. Each tier is an independently shippable arc.

---

## 1. The one theme

The audit's meta-finding is **shipped, but production-inert**: nearly every headline capability reads config that no product surface writes, governance policies only the dev seed creates, or credentials no production path can provision. The repo's own memory predicted this class (`feedback_safety_gate_needs_producer_population`, `feedback_per_slice_review_misses_cross_slice_seams`): each slice shipped its consumer with tests against fixture producers, and "who populates this for a real org?" had no owner.

**Therefore the load-bearing rule for every PR in this plan set is:** _the producer ships in the same PR as the consumer it feeds, with a test that runs from the real producer's default output_, not from a fixture. This is non-negotiable; it is the exact rule whose violation created this audit.

## 3. Open decisions (audit §9): resolved

1. **Tier 0 packaging, DECIDED: one arc, folded into the production-provisioning-runbook.** All three dedup agents independently converged: pilot-spine F-16, Alex F3, and Riley D4-5/D5-1/D6-2 are the _same_ missing org-provisioning step. One seeder, one runbook section. Tier 0 owns it; F-16 and F-02 are closed in Tier 0's acceptance criteria.
2. **Credential unification, DECIDED: resolver fallback now, unify post-pilot.** Riley's `getDeploymentCredentials` gains a fallback to the org-level `Connection` (`serviceId="meta-ads"`) so an operator can credential Riley today through the existing Settings UI, with a deprecation note. The canonical single-store migration is deferred to post-pilot. (The OAuth self-serve path is fixed in parallel for _security_, not because the pilot needs it, see decision-adjacent note below.)
3. **D8-1 zombie widget, DECIDED: remove now.** A broken affordance on every authed page is anti-trust. Conversational Riley (D8-2) is a Tier-4 design item that deserves a real design, not a resurrected single-tenant relic.
4. **Spec-1B gate, DECIDED: yes, Tier 5 (D5-2, D9-2, D4-6) are explicit entry criteria.** The audit found the act-leg _template_ safe only because it is human-gated and tiny; autonomy multiplies every thin spot. No Spec-1B implementation PR merges until Tier 5 is green. The existing strategy spec (`docs/superpowers/specs/2026-06-05-close-the-revenue-loop-design.md`) predates the audit and lacks the blast-radius contract. Tier 5's D4-6 task amends it.
5. **Learning-loop pilot scope, DECIDED: D7-2 is pilot-window-eligible, not pilot-blocking.** Plan it in Tier 3, sequenced after Tier 0; it can ship _during_ the pilot because the data accrues either way and it is the lightest "gets better over time" proof. It does not gate go-live. (The user may pull it earlier or later; default is "ready to ship as soon as Tier 0 is live.")

**Decision-adjacent clarification on the credential path (important, the audit conflated two routes under D10-3):**

- The **listing route** `GET /api/connections/facebook/:deploymentId/accounts` (`facebook-oauth.ts:154-200`) is reachable **today** with any tenant Bearer key and has no org check → cross-tenant credential read **exploitable now** (this is security-audit **F2**). Fix in Tier 0 immediately, independent of the pilot.
- The **authorize/callback** legs (`:41,50,76`) are unreachable today (the Bearer wall 401s Facebook's browser redirect, which is _why_ D10-1's producer is dead). Their unsigned-`state`/no-binding flaw is **latent** until that wall is lifted. So Tier 0 must lift the wall (auth-exempt the redirect legs) **and** add HMAC-signed state + org-binding **in the same PR**, or unbreaking the flow opens a CSRF hole.

No decision in §9 was a genuine product-behavior fork lacking a default, so none was escalated to the user; all five are resolved above with rationale, per the "work autonomously" directive.

## 7. Final cross-slice integration review (run before declaring the arc done)

The audit exists because per-slice review missed cross-slice seams (`feedback_per_slice_review_misses_cross_slice_seams`). After each tier's PRs merge, run an integration pass that **pins every new producer→consumer seam with a `ConsumerSchema.safeParse(producerOutput)` test driven by the real producer's default output**. The seams this plan creates:

1. **Seeder → governance gate / weekly cron / handoff.** Seeded pause + handoff policies' `actionType` regexes match what `GovernanceGate.evaluate` tests for `adoptimizer.campaign.pause` and the handoff intent; seeded `inputConfig.targetCostPerBooked` (number) and `conversionActionType` (string) satisfy the strict `typeof` gates at `inngest-functions.ts:194,195`; seeded `roster.config.targetCpbCents` satisfies `metrics-riley.ts:114` and `mission.ts:210`. Test: provision a fresh non-`org_dev` org in a seed test, then assert the gate executes/parks correctly and the cron config is fully populated.
2. **Resolver fallback → weekly cron + pause executor.** The org `Connection` (serviceId `meta-ads`) decrypts to the **same** `{accessToken, accountId}` shape the `DeploymentConnection` path returns. Test both stores through one resolver assertion.
3. **OAuth signed-state → callback verifier.** Same HMAC alg + secret on both sides; a forged/altered `state` is rejected; a deployment not owned by the caller's org is 403'd on the listing route.
4. **Booked-value producer (D3-1) → `value:{gt:0}` queries + corroboration + trueROAS.** A real booking with a stamped per-service value flows through `queryBookedValueCentsByCampaign` and un-darkens `deriveCorroboration`. Test from the booking tool's real output, not a seeded ConversionRecord.
5. **Response-aware submitter (D6-1) → `SubmitWorkResponse` union.** All three arms (`ok:false`, bare `ok:true`, `approvalRequired:true`) + `null` are handled; assert the deny and ungated-execute arms now alarm.
6. **Zero-conversion rule (D1-1) → eval fixtures.** New conversions=0/spend>0 fixture pins a pause/abstain, not silence.
7. **Idempotency marker (D5-3/D4-1) → submitter branch order.** A replayed parked submit reconstructs `approvalRequired:true` and the submitter reports `parked:true`, no false alarm.
8. **alert flip (D2-9) → OperatorAlerter.** A forced weekly-audit failure and a zero-output run both raise exactly one operator alert.
