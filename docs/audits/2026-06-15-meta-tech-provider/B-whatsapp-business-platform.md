# Meta Tech Provider Audit — B: WhatsApp Business Platform (server-side)

Date: 2026-06-15
Auditor scope: Embedded-Signup token exchange + WABA onboarding, inbound webhook, outbound messaging (session + template), data-deletion callback. Server logic only (dashboard click-through is a sibling audit).
Method: read-only, every claim cites `file:line` against the working tree on branch `feat/receipted-booking-quality-report`.

---

## 0. Verdict (TL;DR)

**Could we screencast a real Embedded Signup + a live inbound→outbound round-trip TODAY? NO — "yes-but-major-rework".**

There are two SEPARATE onboarding code paths and neither is a complete Tech-Provider Embedded-Signup flow:

1. The dedicated ESU route `POST /api/whatsapp/onboard` exists and makes REAL Graph calls, but (a) it never exchanges an OAuth **code** for a token — it expects the caller to already hand it an `esToken` (`whatsapp-onboarding.ts:74-93`); (b) its `createConnection` writes the **WABA id into the credentials `token` field** instead of a real access token (`bootstrap/routes.ts:198-203`), and hardcodes `organizationId: ""` (`bootstrap/routes.ts:207`) — so the row it persists can never drive the live inbound runtime, which reads `creds.token` as a Bearer access token (`runtime-registry.ts:166-176`). The onboarding write is effectively a dead-end record.
2. The live messaging runtime is **BYOT (bring-your-own-token)**, not Tech-Provider central-token. The standard provision path (`organizations.ts:283-291`) stores an operator-pasted per-tenant `token`, and the inbound adapter + health checker authenticate with that stored per-tenant token (`runtime-registry.ts:167-172`, `health-checker.ts:96-110`). The central `META_SYSTEM_USER_TOKEN` is used ONLY by the management/read + send-test + template-create routes (`whatsapp-management.ts:227`, `whatsapp-send-test.ts:165`, `whatsapp-template-create.ts:101`) and by the onboard route's Graph calls — NOT by the inbound→reply path.

So the **read/management surface and the send-test surface are demo-able with a real WABA** (they use the system token and call real Graph endpoints), but a true **Embedded-Signup → auto-provision → inbound message → Alex auto-reply** loop is not wired end-to-end. The phone `/register` step uses a hardcoded PIN `"000000"` with no two-step-verification handling (`whatsapp-onboarding.ts:129-132`). Credit-line sharing / `assigned_users` credit allocation is absent (only a `MANAGE`-task `assigned_users` call is made).

## 3. Inbound webhook (questions 2)

**GET verify handshake — correct.** `managed-webhook.ts:51-67` routes by `webhookId`, looks up the gateway entry, and delegates to `adapter.handleVerification(query)`. `whatsapp.ts:108-122` does a constant-time compare of `hub.verify_token` against the per-channel `this.verifyToken`, returns `hub.challenge` on `hub.mode==="subscribe"`, else 403. Caveat: `verifyToken` is per-tenant from creds; if a channel was created without it (the ESU path, §2 step 3b), this returns 403.

**POST signature verify — correct and fail-closed (security audit F9 lineage).** `managed-webhook.ts:69` opts into `rawBody`. The route: (a) fails closed if the adapter has no `verifyRequest` (`:87-93`); (b) fails closed if `request.rawBody` is not a string (`:102-109`) — no re-serialize fallback; (c) computes `sha256=HMAC(appSecret, rawBody)` and `timingSafeEqual`s with a length-guard (`whatsapp.ts:95-103`). Missing `appSecret` → fail closed with a warn (`whatsapp.ts:88-93`). This is the canonical correct implementation.

**Message + status parsing — REAL and broad.** `parseIncomingMessage` (`whatsapp.ts:124-153`) handles interactive (button/list/nfm_reply flow responses), media, unsupported, and text; CTWA referral metadata is extracted (`whatsapp-parsers.ts:23-35`) and CTWA-tagged inbounds are forwarded through the lead-intake front door (`managed-webhook.ts:149-169`). `parseStatusUpdate` (`whatsapp.ts:318-367`) extracts id/recipient/status/timestamp/errors/pricing.

**Gap:** status updates are routed ONLY to the test-send bridge (`managed-webhook.ts:115-126` → `main.ts:357-366` → `whatsapp-test-send-status-bridge.ts`), which updates `WhatsAppTestSend.lastWebhookStatus`. The general-purpose `PrismaWhatsAppStatusStore` (`prisma-whatsapp-status-store.ts`) is **never called from the inbound path** — production message delivery/read receipts for real conversations are dropped on the floor. Also, a status-only webhook short-circuits and returns before dedup/CTWA/message handling (`managed-webhook.ts:118-125`), which is correct, but means a status callback for a non-test-send message is silently discarded.

## 4. Outbound send (question 3)

**Session/free-form: REAL** — `whatsapp.ts:155-168` → `sendMessage` → `POST /{phoneId}/messages` with `type:text`, `withRetry` (429/5xx), 80 msg/s token-bucket limiter (`whatsapp.ts:19-48`). Token = `this.token` (per-tenant). This is the path the inbound reply uses (`managed-webhook.ts:180-182` builds a `ReplySink` calling `adapter.sendTextReply`).

**Template: REAL at two layers** — (a) adapter `sendTemplateMessage` (`whatsapp.ts:174-196`) for the conversational path; (b) the dedicated `POST /api/dashboard/whatsapp/send-test` route (`whatsapp-send-test.ts:91-244`) which gates on a `testRecipients` allowlist (`:142-150`) and refuses non-APPROVED templates (`:186-197`), using the **system token** (`:165`). The workflows `meta-lead-greeting` / `conversation-followup-send` / `conversation-reminder-send` send templates directly via `WHATSAPP_ACCESS_TOKEN` (per-tenant) (`meta-lead-greeting-workflow.ts:12-43`, `conversation-followup-send-workflow.ts:87-121`).

**24-hour window gating:**

- `isWithinWhatsAppWindow` / `canSendWhatsAppTemplate` are pure helpers, fail-closed on null inbound (`whatsapp-window.ts:9-32`). USED by the proactive eligibility evaluator in the follow-up/reminder workflows (`conversation-followup-send-workflow.ts:57-69`).
- The skill-runtime `WhatsAppWindowGateHook` (`whatsapp-window-gate.ts`) substitutes an approved template for the free-form reply when outside the window — but it is **DARK by default**: it no-ops unless `governanceConfig.whatsappWindow` is present AND `enabled:true` (`:79,102`), and only blanks/substitutes in `mode:"enforce"` (`:119,244,279,336`). Default posture is off/observe.
- **Crucially, the adapter's own `sendTextReply`/`sendMessage` does NOT consult the window at all** (`whatsapp.ts:155-168`). The window is enforced only if the gate hook is wired into the skill pipeline with an enforce-mode config. So a free-form reply outside the 24h window would still be POSTed to Graph and rejected by Meta at send time, not pre-empted.

**Template registry + "WA template gate" — current state:** all 10 registry entries are `approvalStatus:"draft"` (`whatsapp-registry.ts:60-217`). In enforce mode the gate substitutes ONLY `"approved"` entries (`whatsapp-window-gate.ts:195-215`), so today every outside-window template-substitution attempt falls through to block+handoff with sub-cause `template_not_approved`. Marketing templates are additionally blocked unless `allowMarketingTemplateSubstitution:true` (default false, `whatsapp-window-gate.ts:217-240`). Net: the template gate is **default-off and would block everything even if turned on**, because no registry entry is marked approved and none of these template names have been submitted/approved at Meta.

## 5. Token model (question 4)

Two token regimes coexist; they are NOT the same and the live messaging path uses the **per-tenant** one:

- **Central system token** `META_SYSTEM_USER_TOKEN` (a.k.a. SUAT): read by `whatsapp-management.ts:227`, `whatsapp-send-test.ts:165`, `whatsapp-template-create.ts:101`, and the onboard route (`bootstrap/routes.ts:186` → `whatsapp-onboarding.ts:49,90,112,...`). Powers READ/management, send-test, template-create, and the onboard Graph calls.
- **Central app token** `WHATSAPP_GRAPH_TOKEN` + `WHATSAPP_APP_SECRET`: read by the standard provision webhook auto-registration in `organizations.ts:379-380` (used as `appToken` for `debug_token` and `verify_token` for `subscribed_apps`).
- **Per-tenant token** `WHATSAPP_ACCESS_TOKEN` (env, single-tenant) used by the greeting/follow-up/reminder workflows (`meta-lead-greeting-workflow.ts:12`, `conversation-followup-send-workflow.ts:87`, `conversation-reminder-send-workflow.ts:82`); and per-tenant `creds.token` (stored on `Connection`/`DeploymentConnection`) used by the **inbound→reply runtime** (`runtime-registry.ts:167`, `health-checker.ts:97`) and every `WhatsAppAdapter` send method (`whatsapp.ts:404-417` uses `this.token` from `creds.token`).

**Where client WABA tokens are stored & encrypted:** `Connection.credentials` / `DeploymentConnection.credentials`, AES-encrypted via `encryptCredentials` (`organizations.ts:283-291`, `bootstrap/routes.ts:198-203`) and decrypted via `decryptCredentials` (`runtime-registry.ts:7,92,122`, `whatsapp-management.ts:296`, `organizations.ts:395`). Encryption key = `CREDENTIALS_ENCRYPTION_KEY` (enforced in prod, `chat/startup-checks.ts:45-51`).

**The mismatch that breaks Tech-Provider mode:** in a real Tech-Provider setup the inbound runtime would authenticate with the **central** system-user token scoped to each client WABA's phone-number-id. Here the inbound runtime instead expects a **per-tenant** `creds.token`. The standard provision path satisfies this only because the operator manually pastes a per-tenant token (BYOT). The ESU path is supposed to supply it automatically but writes the WABA id there instead (`bootstrap/routes.ts:199`). There is no code path where the inbound runtime uses `META_SYSTEM_USER_TOKEN` to send on a client's behalf.

## 7. Data Deletion callback (question 6) — COMPLETE & CORRECT

`POST /api/meta/deletion` (`meta-deletion.ts:79-188`):

- Auth-bypassed in middleware by **exact path** (`auth.ts:135-136`), integrity by signed_request HMAC.
- `parseAndVerifySignedRequest` (`meta-signed-request.ts:47-88`) splits `sig.payload`, computes `HMAC-SHA256(appSecret, payloadEncoded)`, constant-time compares, base64url-decodes the JSON, requires `user_id`. Fails closed on empty/malformed/bad-sig/non-JSON/missing-user_id.
- On verify: matches `Contact.phone` against both `userId` and `±"+"` variants (`:122-124`), cascade-erases each via `eraseContactFully` (cancels external calendar events too), records a `DataDeletionRequest` with `confirmationCode`, returns `{ url, confirmation_code }` per Meta spec (`:179-186`).
- `GET /api/meta/deletion/status?code=` returns status/timestamps/count (`:191-230`).
- Rate-limited 10/min (`:85-90`). Errors are PII-scrubbed before logging (`describeDeletionError`, `:46-58`).

**This is the cleanest, most review-ready piece of the whole surface.** Requires `META_APP_SECRET` set and the callback URL registered in the Meta App dashboard (config-only).

## 8. "Looks wired but inert" findings (ruthless)

1. **ESU onboard persists an unusable credential.** `bootstrap/routes.ts:199` stores `token: data.wabaId` (WABA id) as the access token and `organizationId: ""` (`:207`); no `verifyToken`/`appSecret`. The runtime adapter reads `creds.token` as a Bearer (`runtime-registry.ts:167`) → every outbound 401s and inbound POST verify fails closed (`managed-webhook.ts:87-93`, `whatsapp.ts:88`). The onboarding "success" response (`whatsapp-onboarding.ts:179-193`) is a phantom — Graph steps succeeded but the channel can't run.
2. **`/whatsapp/onboard` has no OAuth code exchange** — it requires the caller to pre-exchange the code into a token (`whatsapp-onboarding.ts:79`). The real exchanger is Ads-only (`facebook-oauth.ts:104`, `routes/facebook-oauth.ts:130`). If the dashboard posts the ESU code, `debug_token` fails.
3. **WhatsApp Flows endpoint is dead code.** `whatsappFlowsRoutes` (`whatsapp-flows.ts:52`) is never registered in `bootstrap/routes.ts` (grep: zero non-test references). The encryption is correct but the route is unreachable → Flow submissions 404. The adapter's `sendFlowMessage` (`whatsapp.ts:281-316`) can send a flow, but the data-exchange callback that would receive responses is DARK.
4. **WhatsApp-window gate + template registry are default-off AND unapprovable.** Gate no-ops unless `governanceConfig.whatsappWindow.enabled` (`whatsapp-window-gate.ts:79,102`); every registry template is `approvalStatus:"draft"` (`whatsapp-registry.ts:60-217`), so even in enforce mode it blocks all substitutions (`:195-215`). Memory's "WA template gate" is not a live sender.
5. **Stale Graph API versions in core notifiers.** `WhatsAppApprovalNotifier` defaults `v18.0` (`core/notifications/whatsapp-notifier.ts:17`) and `ProactiveSender` hardcodes `v18.0` (`proactive-sender.ts:147`). Everything else is `v21.0`. Owner-approval-over-WhatsApp and proactive owner notifications ride the old version — works today but is drift, and the approval notifier sends a free-form interactive message with no window check.
6. **General-purpose status store never wired.** `PrismaWhatsAppStatusStore` (`prisma-whatsapp-status-store.ts`) is only constructed in tests; the live inbound status path only feeds the test-send bridge (`main.ts:357-366`). Real conversation delivery/read receipts are discarded.
7. **`assigned_users` task array uses single-quote literal** `tasks=['MANAGE']` (`whatsapp-onboarding.ts:112`) — Graph expects a JSON array `["MANAGE"]`; unverified against live Graph, plausibly rejected.
8. **`subscribed_apps` verify_token = app secret** (`whatsapp-onboarding.ts:153`) but the auto-provisioned channel stores no `verifyToken`, so the subsequent GET handshake can't echo it → 403 (`whatsapp.ts:113-121`).
9. **Workflows assume a single global per-tenant token.** `WHATSAPP_ACCESS_TOKEN`/`WHATSAPP_PHONE_NUMBER_ID` are process-global env (`meta-lead-greeting-workflow.ts:12-13`), so in a multi-tenant Tech-Provider deployment every org's greeting would send from ONE number — not per-client. Silently "skips" if unset (returns `completed`, `:14-20`), so a misconfig looks like success.

## 9. Send/receive round-trip status

**What works (with a real WABA + per-tenant token + the standard provision path):** operator pastes a per-tenant WhatsApp Cloud API token + phone-number-id in Settings → `organizations.ts` provisions a Connection+ManagedChannel+DeploymentConnection, best-effort registers the webhook via `subscribed_apps`, health-probes the number, and notifies chat to load the adapter. An inbound message then verifies (GET + POST HMAC), parses, and `handleIncoming` drives Alex; the reply goes out via `sendTextReply` on the same per-tenant token. **This BYOT loop is plausibly demo-able** if the env (`WHATSAPP_GRAPH_TOKEN`, `WHATSAPP_APP_SECRET`, `CHAT_PUBLIC_URL`, `INTERNAL_API_SECRET`) and a publicly reachable chat webhook are configured. Memory flags it "unproven-locally" — the blocker is purely operational (public webhook URL + real creds), not code.

**What's missing to prove inbound→outbound via TECH-PROVIDER Embedded Signup:** the ESU path (`/whatsapp/onboard`) must (a) exchange the code for a real business-system-user token, (b) store that real token (+ appSecret + a real verifyToken) on the Connection with the correct `organizationId`, and (c) the inbound runtime must authenticate sends with the per-client system-token. None of (a)-(c) is implemented; (b) is actively wrong.

## 10. Demo-ability verdict (concrete)

- **Real WhatsApp Embedded-Signup onboarding screencast: NO.** The route doesn't exchange the code, doesn't share a credit line, hardcodes the registration PIN, and persists an unusable credential. A reviewer following the actual ESU SDK flow would not get a working channel out the other side.
- **Real `whatsapp_business_management` in-product demo (read WABA + list templates + CREATE a template, the explicit App-Review target per `2026-05-29-whatsapp-template-create-design.md`): YES**, provided `META_SYSTEM_USER_TOKEN` + a real connected WABA seed (the spec's throwaway `Connection`+`ManagedChannel`). These routes call real Graph v21.0.
- **Real `whatsapp_business_messaging` send demo: YES (constrained)** via `POST /send-test` to an allowlisted recipient with an APPROVED template, on the system token. A live conversational free-form reply is demo-able only on the BYOT provision path with a real per-tenant token + public webhook.
- **Data Deletion callback demo: YES** — register the URL, POST a signed_request, observe `{url, confirmation_code}` and the status endpoint.

**Top blockers to a clean Tech-Provider story, in priority order:**

1. Implement WA ESU **code→token exchange** and store the resulting token (not the WABA id) with the right `organizationId`, `appSecret`, and a real `verifyToken` (`whatsapp-onboarding.ts:79`, `bootstrap/routes.ts:197-217`).
2. Decide the live-runtime token model: either (a) store per-client system-scoped tokens on the Connection, or (b) refactor the adapter to send with `META_SYSTEM_USER_TOKEN` + per-client phone-number-id (`whatsapp.ts:404`, `runtime-registry.ts:166`).
3. Add **credit-line sharing** to onboarding (`whatsapp-onboarding.ts` after step 4).
4. Handle phone `/register` **2-step PIN** conflicts / number migration (`whatsapp-onboarding.ts:129`).
5. Register `whatsappFlowsRoutes` if Flows are in the App-Review surface (`bootstrap/routes.ts`).
6. Bump core notifier Graph versions off `v18.0` (`core/notifications/whatsapp-notifier.ts:17`, `proactive-sender.ts:147`).
7. Approve+mark the registry templates and/or wire real conversation status persistence if the demo includes delivery receipts.
