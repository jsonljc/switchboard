# D2: UGC Engine

Domain auditor d2, 2026-06-10, branch docs/mira-capability-audit. Scope: the UGC generation engine in `packages/creative-pipeline/src/ugc/` plus its provider clients, QA stack, SSRF posture, consent substrate, and reachability from Mira surfaces.

## Scope and method

Read every file under `packages/creative-pipeline/src/ugc/` (including `phases/`), `util/safe-url.ts`, the Kling/HeyGen/ElevenLabs/Whisper clients in `stages/`, the ConsentRecord model and store in `packages/db`, the UGC seams in `apps/api` (submit workflow, decision workflow, render-spend, mira-brief route, concept-draft and self-brief workflows, bootstrap wiring), and the core read-model status mapper. Reconciled against `docs/superpowers/specs/2026-06-04-mira-slice3-generator-quality-design.md` and `docs/superpowers/specs/2026-06-03-mira-roadmap.md`. Verified the slice-3 PR chain on main via `git log` (#896, #897, #898, #903, #907). Every claim below cites live code in this worktree. Prior-session hypotheses were re-verified, two confirmed, two refuted, one partially confirmed (details inline).

Headline: slice 3 shipped essentially as specced. Real frame QA, direction-faithful prompts, end-to-end UGC reachability from the desk, durable assets, mode-aware governance seams, and a real HeyGen client are all on main with their consumers. The remaining gaps are concentrated in three places: the consent/likeness gate exists only as storage, the default UGC output is silent video, and no Riley/Alex signal reaches any UGC creative decision. Quality control is real at the integrity layer (frame QA) and human at the taste layer; taste is measurable only at structure granularity.

## Findings

### D2-F1 (P1) Likeness consent is stored but enforced nowhere; the approved flag on creators is also decorative (verified)

Claim: ConsentRecord (personName, mediaTypes, recordingUri, revoked) and CreatorIdentity.consentRecordId exist, with `attachConsentRecord` on the creator store, but no code path checks consent or `approved` before a creator is cast, rendered, or published. The creator pool query is `where: { deploymentId, isActive: true }` (`packages/db/src/stores/prisma-creator-identity-store.ts:43-48`), so an unapproved creator with a revoked consent record is fully castable. Repo-wide grep: ConsentRecord is referenced only in `packages/db`, `packages/schemas`, and their tests; `PrismaConsentRecordStore` has zero consumers beyond the `packages/db/src/index.ts:126` export.

Evidence: `packages/db/prisma/schema.prisma:2228-2248`; `prisma-creator-identity-store.ts:43-48` ("findMany({ where: { deploymentId, isActive: true } })"); `ugc/scene-caster.ts:60-89` (castCreators scores personality/repetition only); `phases/production.ts` and the publish preconditions contain no consent reference.

Impact: today the only creator is the synthetic House Creator ("Not a real person; no likeness rights involved", `seed-mira-creative-deployment.ts:168-170`), so the gap is latent. The moment a real-person creator lands (a clinic owner's HeyGen avatar via `identityRefIds: ["heygen:..."]`, or the soul_id tier the schema anticipates), nothing prevents generating and publishing that person's face and voice after consent revocation. For SG/MY clinics under PDPA this is a structural legal gate that must not depend on operator memory.

Recommendation: filter or assert at the casting boundary: a creator with a non-stock qualityTier or non-empty identityRefIds requires `approved === true` and a linked consent record that is not revoked or expired; enforce in `preloadContext` or `castCreators` and add a pinning test. Tag: extends (the schema planned consent; enforcement was never specced). Effort: M.

### D2-F2 (P2) Consent store revoke()/getById() are cross-tenant unscoped (hypothesis confirmed, latent) (verified)

Claim: `revoke(id)` mutates by bare id with an explicit deferral comment, and `getById(id)` reads by bare id with no orgId despite ConsentRecord carrying orgId. All CreatorIdentity mutations carry the same documented deferral (org reachable only via FK, needs migration).

Evidence: `packages/db/src/stores/prisma-consent-record-store.ts:42-48`: "route-governance: store-mutation-deferred — unscoped Prisma mutation ... tracked for Round-3 tenant-isolation sweep in #643"; same marker on `prisma-creator-identity-store.ts:54,62,70,78,86`.

Impact: latent (no live callers), but the first consumer of this store inherits a cross-tenant primitive: a route bug could revoke or read another org's consent record. Consent is exactly the record class where tenant bleed is unacceptable.

Recommendation: fold into the #643 sweep before any consumer ships: `revoke(orgId, id)` via updateMany with a `count === 0` guard (per the updateMany no-match-abort rule), orgId on getById. Tag: planned (#643). Effort: S.

### D2-F3 (P1) The default UGC path produces silent video; the creator voice fields are decorative (verified)

Claim: Kling text2video receives only the composed visual prompt (`ugc/video-provider.ts:56-71`); no UGC code path synthesizes speech. ElevenLabs is constructed exclusively in the polished pro tier (`stages/run-stage.ts:154-156`). The seeded House Creator carries an ElevenLabs voiceId (`seed-mira-creative-deployment.ts:171-178`) that nothing in the UGC pipeline reads, and has `identityRefIds: []`, so `providersAllowed` is always `["kling"]` (`phases/scripting.ts:207`). HeyGen-native TTS exists but only fires for talking_head specs whose creator carries an explicit `heygen:` ref AND when HEYGEN_API_KEY is configured.

Evidence: `phases/scripting.ts:168-169,191,207`; `ugc/video-provider.ts:60-67` (Kling request: prompt, duration, aspect, image, negative, camera; no audio field); grep: no ElevenLabs reference under `src/ugc/`.

Impact: the spec's own division (3.5: "v1 speech is HeyGen-native TTS", ElevenLabs lipsync deferred at `elevenlabs-client.ts:53`) is documented, but the out-of-box configuration ships a "talking_head" ad with no voice and no operator-facing warning. Spoken-word UGC is the format's entire mechanism on Meta; a silent clip is not a competitive creative, which blunts the generate-publish-learn loop at its first step. The desk QA line cannot catch it (frames carry no audio by design).

Recommendation: shortest honest fix: surface "no audio track" on the UGC draft card (the read model already parses production outputs); product fix: provision a default HeyGen avatar ref for the house creator at install, or wire the deferred ElevenLabs lipsync leg behind the named R2 TODO. Tag: planned (spec 3.5 deferral) for the audio leg, new for the operator-facing disclosure. Effort: M.

### D2-F4 (P1) No Riley or Alex signal reaches any UGC creative decision; UGC is reachable from the desk only (verified)

Claim: `preloadContext` hardcodes `funnelFrictions: []` ("SP8 adds real friction store") and `providerCapabilities: []` (`ugc/ugc-job-runner.ts:236-237`), so `translateFrictions([])` always returns empty weights and the complete 8-rule friction translation table (`ugc/funnel-friction-translator.ts:13-62`) has no producer. The planning phase passes `performanceMemory: { structureHistory: {}, creatorHistory: {} }` (`ugc-job-runner.ts:130`) and `recentStructureIds: []` / `recentCastings: []` (`phases/planning.ts:81,92`), so structure selection degenerates to static platform affinity (the performance and fatigue dimensions of `selectStructures`, `structure-engine.ts:198-228`, are always 0): every job for the same platforms ranks the same three structures. Separately, both agent-initiated entry points are hardwired polished: the Riley handoff workflow creates via the polished `jobStore.create` (`creative-concept-draft-workflow.ts:126`) and the self-brief submit request has no mode parameter (`mira-self-brief-request.ts:75-90`), so only a human on the desk can start a UGC job.

Impact: the north star is "turn Riley diagnoses and Alex funnel reality into winning creative". For the UGC engine, zero bits of Riley or Alex reality reach any creative decision: not structure choice, not script constraints, not hooks, not casting. The synergy moat does not exist on this leg; UGC output is a function of the brief text and templates alone.

Recommendation: smallest real step: a FunnelFriction producer (Riley diagnosis categories map cleanly onto the existing frictionType vocabulary: low_trust, price_shock, weak_hook, low_urgency) persisted per deployment and read in `preloadContext`; then add `mode` to the concept-draft parameters so a Riley handoff can request real-talk creative. Tag: planned (runner SP8 comments; roadmap slice-2 framing) for frictions; extends for delegation mode. Effort: L.

### D2-F5 (P2) A UGC job whose production generated zero assets completes as draft_ready, never failed (verified, one citation corrected)

Claim: `allUgcAssetsRejected` requires `assets.length > 0` (`status-mapper.ts:41-44`). When every spec exhausts on generation errors (provider outage, client not configured) or hits `no_allowed_provider` / budget-exceeded, `processSpec` returns no asset and `executeProductionPhase` returns `{ assets: [], failedSpecs: [...] }` as a successful phase output (`phases/production.ts:247-250,295-297`). The runner advances through delivery to `ugcPhase: "complete"` and emits `assetsProduced: 0`; the status mapper then returns `draft_ready` (`status-mapper.ts:58`) with `deriveUgcDraft` undefined. The all-REJECTED case correctly reads failed; the all-ERRORED case does not.

Impact: an operator sees a "ready" creative containing nothing; the desk read model hides it (the gate is `hasVideo = typeof job.draft?.videoUrl === "string"`, `creative-read-model/desk-model.ts:59`; corrected during verification from the earlier "isReviewable" name, same mechanism), so the job silently evaporates rather than reading as the failure it is. This is the same silent-empty class the spec's 3.3e creator seeding fixed for empty pools, surviving via a different route.

Recommendation: extend the failure rule: production output present AND assets empty maps to failed (one condition next to `allUgcAssetsRejected`, plus a test). Tag: extends (spec 3.1 consumer-4 rule covers only the rejected case). Effort: S.

### D2-F6 (P2) Provider generation errors are swallowed with zero diagnostics; the ugc.\* events have no consumers (verified)

Claim: the per-attempt catch in `processSpec` is bare (`phases/production.ts:202-206`: `catch { if (attempt === attemptsFor - 1) break; }`): no log, no persisted error, and the final `failedSpecs` reason is the generic "all providers exhausted, no reusable asset". Additionally, `creative-pipeline/ugc.failed`, `ugc.stopped`, and `ugc-phase.completed` are emitted by the runner but a repo-wide grep finds no subscriber. The dead-letter doctrine is satisfied at the Inngest level (the runner is registered with a `makeOnFailureHandler` in `bootstrap/inngest.ts:1128-1136`), but an in-phase generation failure never reaches any ledger or log line.

Impact: diagnosing a Kling outage versus a misconfigured client versus an SSRF rejection requires re-running the job with a debugger. For a paid-render pipeline this is an operations blind spot, and combined with D2-F5 a provider outage degrades to a quiet empty "draft_ready".

Recommendation: capture the last error message per provider into the failedSpecs entry and `console.warn` each generation failure (matching the QA degrade's existing logging at `realism-scorer.ts:299`). Tag: new. Effort: S.

### D2-F7 (P2) shouldRequireApproval fails open on a non-finite trustLevel (NaN-blind gate) (verified)

Claim: the gate is `return ctx.trustLevel < threshold` (`ugc/approval-config.ts:32`). A NaN trustLevel makes every comparison false, so shouldRequireApproval returns false for all four phases: the job runs planning through delivery with no waits, which also bypasses the spend park (spend attaches to the approve-into-production gesture; with no gate there is no gesture). trustLevel originates as `deployment?.listing?.trustScore ?? 0` (`ugc-job-runner.ts:234`), a DB float, so NaN requires data corruption: low likelihood, but the failure direction is wrong (gate evaporates rather than locks).

Evidence: `approval-config.ts:22-33`; `ugc-job-runner.ts:332-363`. Verification also checked the one escape hatch: the `alwaysRequireApproval` short-circuit precedes the comparison but `DEFAULT_APPROVAL_CONFIG.alwaysRequireApproval` is `[]` (`approval-config.ts:19`), so it cannot save the NaN path. The repo has prior art for exactly this class (memory: NaN-blind comparison gates, caught in #939). The vision-QA numeric path, for contrast, is structurally NaN-safe (JSON.parse cannot produce NaN and zod's z.number() with min/max bounds rejects it, `realism-scorer.ts:95-104`), refuting the prior-session hypothesis for the QA gates.

Recommendation: `if (!Number.isFinite(ctx.trustLevel)) return true;` before the comparison, one test. Tag: new. Effort: S.

### D2-F8 (P2) ugcConfig.retryConfig is schema-valid input that nothing reads (verified)

Claim: `UgcConfigSchema.retryConfig` exists (`packages/schemas/src/ugc-job.ts:52-57`) and the submit workflow could carry it, but the runner hardcodes `retryConfig: { maxAttempts: 3, maxProviderFallbacks: 2 }` (`ugc-job-runner.ts:188`). The spec documents this honestly (3.3d: "setting it changes nothing").

Impact: a config knob that silently no-ops is a future operator trap.

Recommendation: wire it (`ugcConfig.retryConfig ?? default`) or drop the field. Tag: planned (spec names it). Effort: S.

### D2-F9 (P2) Scene direction is constant: the variety seed exists and is never used (verified)

Claim: `pickFrom(arr, fallback, seed = 0)` is always called without a seed (`ugc/ugc-director.ts:65-69,165-171`), so lighting is always "natural" (first of UGC_LIGHTING), environment and hairState are always the first array entries, wardrobe is always the first two palette entries, and `DEFAULT_IMPERFECTIONS` plus forbiddenFraming are constants. Within a job and across jobs, every clip from the same creator carries an identical scene sentence; creative variety comes only from structure choice and LLM script sampling.

Impact: for a generator whose pitch is volume-with-variation, the deterministic direction layer collapses to one fixed look per creator, weakening creative fatigue resistance (the thing UGC volume exists to beat).

Recommendation: pass a deterministic per-spec seed (casting index or a hash of specId) through `generateDirection`; the seed parameter already exists. Tag: new. Effort: S.

### D2-F10 (P1) Claims safety for UGC scripts is neither library-backed nor prompt-guarded; claimsPolicyTag is dead (verified)

Claim: the roadmap's cross-cutting invariant 4 requires "the agent composes only from a pre-approved claim library and never invents claims. This is a hard gate, not a tier." The UGC script prompt contains no health/medical-claim guardrail at all: its FORBIDDEN list is generic ad cliches ("limited time offer", "act now", `ugc-script-writer.ts:73`), and its output format requests only text and language (`:76-83`), so the optional `claimsPolicyTag` (`:43`) is effectively never populated; nothing downstream reads it (`phases/scripting.ts:181` stores it; no consumer). No claim classifier runs on creative content anywhere in the pipeline or at publish. Verification hunted for a refuter and found none: the banned-phrases corpus (including `governance/banned-phrases/my.ts`) and its scanner are wired only into the skill-runtime chat-outbound hook (`skill-runtime/hooks/deterministic-safety-gate.ts`); zero scanner or banned-phrase references exist in `packages/creative-pipeline` or the creative workflows.

Impact: for SG/MY aesthetic clinics (HSA/PHMC-regulated advertising), a hallucinated efficacy claim ("removes wrinkles permanently") rides straight to a one-click Keep. The mandatory human approval and paused-only publish keep this from being P0, but the entire regulated-claim screen rests on the operator's eyeball with zero machine assistance, in the vertical where the roadmap calls claim review a hard gate.

Recommendation: minimal: add an explicit medical/efficacy-claim prohibition plus a bounded claimsPolicyTag enum to the prompt and surface the tag on the draft card; real: per-org pre-approved claim library injected as script constraints (the `scriptConstraints` channel already exists and is empty today). Tag: planned (roadmap invariant 4, unimplemented). Effort: M.

### D2-F11 (P2) ProviderPerformanceTracker is built-but-unwired; the ranking history bonus is dead code in the live path (verified)

Claim: `ProviderPerformanceTracker` and `emptyPerformanceHistory` are exported (`packages/creative-pipeline/src/index.ts:59-60`) and `rankProviders` implements a pass-rate/latency bonus (`provider-router.ts:163-174`), but no code constructs a tracker and the runner builds `ProductionInput` without `providerHistory` (`ugc-job-runner.ts:185-206`), so the bonus never executes outside tests.

Impact: provider routing cannot learn that, say, HeyGen has been failing QA; combined with D2-F6 the pipeline neither records nor reacts to provider quality. Violates the roadmap's own "nothing ships built-but-unwired" rule (it predates the rule, but it is precisely the named anti-pattern).

Recommendation: persist per-provider pass rates from the qaResults the production phase already computes (the data is in hand at `phases/production.ts:302`) and thread them into the next job's `providerHistory`. Tag: extends. Effort: M.

### D2-F12 (OPP) UGC taste is measurable only at structure granularity, and taste never feeds back into UGC generation (verified)

Claim: the shipped taste vocabulary keys Keep/Pass on the leading spec's structureId (`creative-descriptor.ts:51-58`, both call sites mode-correct: `services/cron/creative-taste-sweep.ts:224-225`, `services/workflows/creative-performance-history.ts:41-42`), and frame QA is integrity-only by design (`realism-scorer.ts:131-147` pins the aesthetics prohibition). So the answer to "is UGC taste measurable or vibes" is: measurable, but only as "operator kept ugc creatives with <structure> structure"; no per-creator, per-scene-style, or per-hook signal exists, and the taste provider injects into the polished runner only (the spec 3.4 deferral: feeding an empty block to UGC scripting was correctly rejected until buckets accumulate).

Recommendation: when UGC taste buckets have real sourceCounts, thread the existing creativeTasteProvider into `runUgcScriptWriter` constraints (the named wiring point) and consider widening the descriptor with creatorId. Tag: planned (spec 3.4 names both the trigger and the seam). Effort: M.

### D2-F13 (P2) The production budget gate only checks between specs; a single spec can overshoot the job budget (corrected)

Claim: `costTracker.total > input.budget.totalJobBudget` is evaluated before each spec (`phases/production.ts:278`), while attempts accrue inside `processSpec` (`:145`, success-only: generation errors do not bill), so the final spec can exceed the cap by up to its own worst case. Corrected during verification: the tracker accrues the router's `ESTIMATED_COST` table, which pins kling at `KLING_COST_PER_5S` ($0.35) regardless of clip duration (`provider-router.ts:109-111`, `cost-estimator.ts:10`), so the tracked worst case is 3 x 0.35 + 1 x 1.00 = $2.05 (heygen is capped at one attempt), not the $3.10 originally stated; $3.10 is reachable only as real money for >7s clips ($0.70 actual vs $0.35 tracked), which is a second, smaller finding inside this one: the budget tracker undercounts long-clip kling spend by 2x. The spec's stated division (governance parks expected 1x spend; the job budget caps the tail) holds, with this bounded overshoot. Budget inputs are NaN-safe: `UgcConfigSchema.budget.totalJobBudget` is `z.number()` parsed at submit (`ugc-job.ts:43-51`, `creative-job-submit-workflow.ts:80`); note the submit site deliberately omits budget, so the runner default of 50 applies (`ugc-job-runner.ts:190`).

Recommendation: check the tracker inside the attempt loop, or document the one-spec overshoot in the runbook. Tag: extends. Effort: S.

## What is sound

- The honest-stub QA discipline survived de-stubbing: `deriveApprovalState` still gates on `qaStatus === "evaluated"` (`realism-scorer.ts:231-238`), every infrastructure shortfall degrades to human review with the reason persisted in notes (`:297-302`), and the vision verdict is zod-validated with the aesthetics prohibition pinned in the exported prompt (`:131-147`). Frame-only soft scores are renormalized over present dimensions and audioNaturalness is structurally never fabricated (`:159-178,289`).
- SSRF posture is consistent and thorough: one policy (`util/safe-url.ts`) used by the frame extractor, the durable-upload download, and the polished assembler; `redirect: "error"` at every fetch of provider bytes; streaming size caps; IPv4-mapped-IPv6 and NAT64 bypass forms rejected (`safe-url.ts:194-202`). DNS rebinding is explicitly documented out of scope.
- The slice-3 cross-slice seams were actually closed: the decision workflow guard, render-spend gate, publish preconditions, and status mapper are all mode-aware on the same three terminal flags (ugcPhase complete, ugcFailure, stoppedAt), refuting the staleness half of the 3-flags hypothesis: the flags still exist, but every checked consumer handles them (`creative-job-decision-workflow.ts:27-44`, `creative-render-spend.ts:28-35`, `creative-publish-preconditions.ts:58-73`, `status-mapper.ts:46-67`).
- Inngest discipline holds: live clients ride deps, never step-memoized context (`ugc-job-runner.ts:73-75,193-205`); the runner has an onFailure handler (`bootstrap/inngest.ts:1128-1136`); the jobId-only wait match is documented polished parity.
- Per-attempt persistence is honest: every generated attempt persists its own AssetRecord row with snapshot-copied qaHistory; a real evaluated fail re-enters the retry loop instead of returning; durable upload is deliberately outside the generation try/catch so storage failures propagate to failUgc rather than masquerade as generation errors (`phases/production.ts:167-201,305-313`).
- The provider cost tables are aligned to shared constants with the parity intent documented (`provider-router.ts:102-114` importing from `cost-estimator.ts`).
