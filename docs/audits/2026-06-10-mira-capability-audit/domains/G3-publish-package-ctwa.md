# G3: Can the publish draft package feed Alex? (CTWA package audit)

Gap: no prior domain evaluated whether the paused Meta draft package Mira publishes can start an Alex WhatsApp conversation at all. The critic asserted it verifiably cannot as shaped. Audited 2026-06-10 against worktree HEAD 84083f0c. References: R2 platform-policy research (docs/audits/2026-06-10-mira-capability-audit/research/R2-platform-policy.md, cited as R2 with line numbers), the publish spec (docs/superpowers/specs/2026-06-02-mira-publish-paused-ad-design.md), and the north-star loop (SG/MY clinic, conversions happen inside Alex WhatsApp conversations). Extends D4-F6 (SG-only targeting, non-currency-aware budget) and D8-F6 (Connection.status ignored); does not restate them.

## Verdict

Confirmed, with one important correction to the critic's framing. The package as shaped cannot feed Alex: it is an instant-form lead ad skeleton (OUTCOME_LEADS objective, LEAD_GENERATION optimization goal, LEARN_MORE CTA, placeholder web link) with no WhatsApp destination anywhere in the create chain, and R2 line 48 plus R2 line 136 put a CTWA ad's bar at an Engagement or Leads objective with a WhatsApp conversation destination, a Send Message or Chat on WhatsApp CTA, and a verified page-to-WABA binding. Worse than incomplete, it is not operator-finalizable: Meta campaign objectives are immutable after creation, so "finalize in Ads Manager" means rebuilding a new campaign outside the persisted metaCampaignId, which severs the attribution join the receipted-bookings loop depends on. The correction: the inbound half is much closer to real than the critic's "cannot at all" framing implies. The CTWA webhook referral capture, lead.intake ingress, Contact attribution, booked-event payload, and campaign-grouped booked stats all exist and connect; the single missing link inbound is that the ad-to-campaign resolver hook (resolveCampaignId) is never wired at construction, so ConversionRecord.sourceCampaignId stays null and the booked leg can never credit a Mira campaign. Three P1 fixes (CTWA-shape the package, add the Meta-required create params, wire the resolver) plus the precondition and copy hardening below make the loop closable end to end.

## Findings

### G3-F1 (P1, verified) The draft package is lead-form shaped, carries no WhatsApp destination, and cannot be converted into a CTWA ad by the operator

Claim: every destination-bearing constant in the package points away from WhatsApp. apps/api/src/services/creative-publish-function.ts:16-21 declares `DRAFT_OBJECTIVE = "OUTCOME_LEADS"`, `DRAFT_OPTIMIZATION_GOAL = "LEAD_GENERATION"`, `DRAFT_CTA = "LEARN_MORE"`, `DRAFT_LINK_PLACEHOLDER = "https://switchboard.example/finalize-in-ads-manager"`. The ad set create body (packages/ad-optimizer/src/meta-ads-client.ts:291-297) sends only campaign_id, name, targeting, optimization_goal, status; there is no `destination_type` and no `promoted_object`. The creative (meta-ads-client.ts:320-334) wires `call_to_action: { type: ..., value: { link: params.linkUrl } }`, a web-link CTA to a placeholder domain. Grep for `destination_type` across apps/api returns nothing; the only reader is the advisory learning-inputs path (meta-ads-client.ts:194,226).

Policy bar (R2): "Supported objectives: Engagement (WhatsApp conversation location) and Leads ... CTA should be 'Send Message' or 'Chat on WhatsApp'; mismatched CTAs like 'Learn More' hurt review and performance" (R2:48). "objectives restricted to Engagement/Leads with a WhatsApp conversation location, CTA limited to Send Message / Chat on WhatsApp" (R2:136). R2:134 adds that CTWA campaigns optimizing on conversations are the policy-stable design under Meta's health-and-wellness optimization restrictions, "which conveniently matches the WhatsApp booking funnel"; LEAD_GENERATION optimization against a health destination is exactly the funnel R2 warns is being restricted.

Counter-evidence hunted: the in-code framing says placeholders are intentional, "Placeholder ad content the operator finalizes in Ads Manager (the locked 'parked draft' framing)" (creative-publish-function.ts:12-14), and the workflow summary promises "review & activate in Ads Manager" (apps/api/src/services/workflows/creative-publish-workflow.ts:5). That defense holds for budget and link text. It does not hold for objective and destination: Meta does not allow changing a campaign objective or an ad set's destination type after creation (external-API claim, consistent with the spec's own admission that the wire format is unverified live, spec section 11). So an operator who wants the CTWA ad the north star requires must create a fresh campaign by hand, and the persisted metaCampaignId (creative-publish-function.ts:127) then refers to a dead draft. The attribution sweep joins booked stats on exactly that id (apps/api/src/services/cron/creative-attribution.ts:288 `bookedStats.get(job.metaCampaignId)`), so the rebuilt, actually-running campaign is invisible to Mira's loop credit. The publish spec never planned a WhatsApp destination: its linkUrl comment reads "destination (booking link)" (spec line 192) and the section 11 go-live blocker list (durable storage, page-id setter, async hardening, wire-format validation, currency-aware budget) contains no CTWA item.

Impact: for the SG/MY wedge where conversions happen in Alex WhatsApp conversations, a Mira-published ad as shaped can never start one, and the workaround destroys attribution. The loop's act leg dead-ends at a placeholder URL.

Recommendation: reshape the package builder around a CTWA destination: campaign objective OUTCOME_ENGAGEMENT (or OUTCOME_LEADS kept, per R2:48 both are supported, but optimization moved to conversations), ad set `destination_type: "WHATSAPP"` plus `promoted_object: { page_id }`, optimization_goal CONVERSATIONS, creative `call_to_action: { type: "WHATSAPP_MESSAGE", value: { app_destination: "WHATSAPP" } }` replacing the placeholder link. Exact enum values must be validated against a live sandbox account in the same pass the spec already mandates for the upload wire format. Tag: new (no spec coverage). Effort: M.

### G3-F2 (P1, verified in code; one live-API caveat) The create chain omits Meta-required create parameters, so the first live publish likely fails before any draft exists, and the failure is alert-silent

Claim: three parameters Meta requires at create time are absent from the bodies. createDraftCampaign (meta-ads-client.ts:273-286) sends name, objective, status, bid*strategy, daily_budget and nothing else; `special_ad_categories`, mandatory on POST /act*\*/campaigns since Marketing API v5.0 (2019) even when the correct value is the empty array, appears nowhere in the repo (grep `special_ad_categories|specialAdCategor` across packages and apps returns zero hits). createDraftAdSet (meta-ads-client.ts:291-297) sends no `billing_event`, required on ad set creation, and no `promoted_object`, required when optimization_goal is LEAD_GENERATION (a lead-gen ad set must promote a page). These are external-API requirements I cannot prove from the worktree; the in-repo anchor is the spec's own go-live blocker: "Meta wire format (validate before go-live): uploadCreativeAsset's base64-JSON body is unverified against live Meta; proven here only via mock" (spec section 11), an admission that the entire chain has only ever passed against mocks; the publish tests (creative-publish-gate.test.ts, creative-publish-route.test.ts) inject AdsClientLike mocks and cannot catch missing Graph params.

Impact: the publish chain is dead on arrival against the live Graph API at step create-draft-campaign, after consuming a mandatory human approval. The Meta-call failure path retries then dead-letters with `alert: false` (creative-publish-function.ts:56-61, Class B), so a 100 percent reproducible failure surfaces only as a job that never reaches parked_paused.

Recommendation: add `special_ad_categories: []` (with a comment recording the medspa decision: cosmetic services are not a Meta special ad category; the health constraints arrive via the health-and-wellness data-source categorization instead, R2:37), `billing_event: "IMPRESSIONS"`, and the promoted_object from G3-F1. Then run the one live-sandbox validation pass the spec already requires. Tag: extends (spec section 11 wire-format blocker). Effort: S.

### G3-F3 (P1, verified) The inbound CTWA capture chain is real but the campaign-id leg is unwired, so the booked leg can never credit a Mira campaign

Claim: the inbound half exists end to end. The WhatsApp webhook parser extracts referral fields (apps/chat/src/adapters/whatsapp-parsers.ts:25-32: source_id to sourceAdId, ctwa_clid, source_url); the managed webhook fires the CTWA adapter per org (apps/chat/src/routes/managed-webhook.ts:136-156); buildCtwaIntake submits lead.intake with attribution.sourceAdId (packages/ad-optimizer/src/lead-intake/ctwa-adapter.ts:60-73) through the seeded system principal (apps/chat/src/gateway/ctwa-ingress-request.ts); the handler persists sourceAdId, sourceCampaignId, sourceAdsetId onto the Contact (packages/core/src/intents/lead-intake-handler.ts:73-75); calendar-book stamps the booked outbox event with `sourceCampaignId: conversion.sourceCampaignId` from contact attribution (packages/core/src/skill-runtime/tools/calendar-book.ts:365-378, booked-conversion-payload.ts:34). But the CTWA webhook referral carries only the ad id, never a campaign id, and the adapter's resolution hook is optional and unwired: `resolveCampaignId?: (adId: string) => Promise<string | null>` (ctwa-adapter.ts:39, used at :91-104) is not passed at the only construction site (apps/chat/src/main.ts:168-181 passes ingress and now only), and the obvious resolver `getAdCampaignId` (meta-ads-client.ts:384-399) has zero production callers (grep: definition and dist only). Downstream, the booked join is strictly campaign-keyed: queryBookedStatsByCampaign groups ConversionRecords by sourceCampaignId with `sourceCampaignId: { in: campaignIds }` (packages/db/src/stores/prisma-conversion-record-store.ts:270-279) and the sweep matches `bookedStats.get(job.metaCampaignId)` (creative-attribution.ts:288). The reports contract is explicit that the leg fires only on a non-null sourceCampaignId: "ctwa_captured: the booking's ConversionRecord carried a non-null sourceCampaignId" (packages/schemas/src/reports/v1.ts:81-88).

Impact: even after F1 and F2 are fixed and a real CTWA ad starts a real Alex conversation that books, sourceCampaignId is null at intake, the ConversionRecord groupBy drops the row (`if (!row.sourceCampaignId) continue;`, prisma-conversion-record-store.ts:290), every paid visit renders as campaign_missing, and Mira's receipted-booking credit (the G2 booked leg) never fires. The whole prove-leg of the north star is one missing constructor argument away from blind.

Recommendation: two independent fixes, do both. (1) Wire resolveCampaignId in apps/chat/src/main.ts using getAdCampaignId with a fresh MetaAdsClient per call (the client rate limiter is 60s per instance, meta-ads-client.ts:10; a shared instance would delay lead intake by up to a minute) and the org's meta-ads Connection credentials; resolution is already non-blocking on failure (ctwa-adapter.ts:101-103). (2) Make the booked join ad-id tolerant: ConversionRecord already persists sourceAdId (schema.prisma:2058), the job persists metaAdId at the terminal checkpoint (creative-publish-function.ts:168-171), and the spec's P3 note already names ad-level join keys (spec section 11, "metaVideoId/metaAdId ... is the join key"; creative-attribution.ts:160 comment "move to ad-level insights on join.metaAdId"). Pin with a seam test that runs the real main.ts wiring shape, not an adapter unit test with the dep injected (the per-slice-review gotcha: the adapter passes its own contract while the wiring drops the leg). Tag: extends (adapter hook and spec P3 anticipated it; the wiring is the missing slice). Effort: S.

### G3-F4 (P2, verified) Health-vertical constraints are unpinned in the package: 18+ rides a Meta default, the special-ad-category decision is unrecorded, and AI-disclosure state is carried nowhere

Claim: DRAFT_TARGETING is geo-only, `{ geo_locations: { countries: ["SG"] } }` (creative-publish-function.ts:19); no age_min is sent anywhere in the chain. R2:23 is unambiguous for this vertical: "Ads for ... cosmetic procedures and surgeries must be targeted 18+". Counter-evidence: Meta's ad set default minimum age is 18, so today's omission likely passes review; but the mandatory-for-the-vertical invariant exists only as an upstream default, nothing in the package or its tests pins it, and any operator targeting edit or platform default change silently breaks a hard policy rule. Separately, no AI-disclosure state exists on the job or in the package: CreativeJob carries productDescription and targetAudience as brief fields (schema.prisma CreativeJob model) but no field recording that the creative contains a photorealistic synthetic human, and the publish chain passes no disclosure signal. R2:15-17 documents that Meta auto-labels AI-generated photorealistic humans next to the Sponsored tag, treats detected-but-undisclosed synthetic media as a disclosure failure under 2026 enforcement (secondary-sourced, directional), and that "the safe posture is to keep provenance intact and self-disclose where the flow offers it". The special-ad-category question resolves to "none for medspa" but the decision should be recorded where the field is set (mechanical fix in G3-F2).

Impact: latent policy breaks for the exact vertical the product serves; the disclosure gap also blinds the operator runbook (the human who activates in Ads Manager has no signal that self-disclosure applies to this asset).

Recommendation: pin `age_min: 18` in the package targeting with an R2 citation comment and a test asserting it; add an aiDisclosure marker to CreativeJob (set by the UGC pipeline whenever an avatar asset is used) and surface it in the publish route response and operator copy. Tag: new. Effort: S.

### G3-F5 (P2, verified) The ad's primary text is ungoverned brief input, and the CTWA pre-filled opener (governed copy per R2) does not exist in the package

Claim: the creative message is the raw brief field: `message: pre.job.productDescription` (creative-publish-function.ts:151), and the campaign name embeds the same string (draftName, :63-65). productDescription is operator INPUT to the pipeline, not reviewed ad copy; the Keep decision that gates publishability (creative-publish-preconditions.ts:65) reviews the rendered video, and no copy field exists on the job. R2:48 notes primary text over roughly 125 characters truncates in feed; nothing guards length. For CTWA specifically, R2:136 requires "the pre-filled opener treated as governed copy (it is part of the reviewed ad)"; the package has no welcome-message concept at all (grep page_welcome across apps/api: nothing), so when F1 lands there will be no governed source for the opener that becomes the first message of the Alex conversation.

Impact: the one piece of text a patient sees in feed, and the first message Alex receives, are either ungoverned or missing; for a regulated vertical (R2:67-78 SG HCSA, R2:91 MY MAB) ad text is the compliance surface.

Recommendation: add primaryText and welcomeMessage to the brief/job, render them in the review surface so Keep covers them, length-guard primaryText, and feed both into the package (message and the CTWA welcome message field). Tag: new. Effort: M.

### G3-F6 (P2 latent, becomes P1 once F1 lands; extends D8-F6) Publish pre-flight verifies a pageId string exists but never the page-to-WABA binding or the org's WhatsApp channel health, although the org's WABA identity is already stored

Claim: assertPublishable's only destination check is string presence: `creds["pageId"]` else META_PAGE_NOT_CONFIGURED (apps/api/src/services/creative-publish-preconditions.ts:102-109). R2:47 makes binding health a hard requirement: "the WhatsApp number must be verified and connected to the Facebook Page and Business Manager. Unverified or incorrectly linked numbers prevent publication; wrong-number assignment silently routes ad conversations to the wrong inbox", and R2:136 elevates it to a publish precondition ("Number-to-Page-to-WABA binding must be verified before publish"). The system already holds everything needed to check: the embedded-signup onboarding persists wabaId, phoneNumberId, verifiedName, displayPhoneNumber into the WhatsApp connection (apps/api/src/routes/whatsapp-onboarding.ts:135-140) and the chat runtime reads credentials.phoneNumberId per org (apps/chat/src/managed/runtime-registry.ts:168-172); ManagedChannel carries a per-org whatsapp status field (schema.prisma:520-538). The pre-flight consults none of it. D8-F6 already covers ignoring Connection.status on the meta-ads connection; this is the adjacent, distinct axis: even a connected ads connection says nothing about whether the configured page is bound to THIS org's WABA.

Impact: latent today (F1 means no CTWA ad is created), but the moment the package is CTWA-shaped this is the misroute scenario R2 names: a clinic's paid ad conversations landing in another inbox, or publication blocked at Meta review, both after the human approval is spent.

Recommendation: extend assertPublishable with (a) the D8-F6 status fix, (b) require an active ManagedChannel whatsapp row for the org, and (c) a Graph probe that the configured pageId's connected WhatsApp number matches the org's stored phoneNumberId, with a typed CTWA_BINDING_UNVERIFIED failure code. Tag: extends (D8-F6, R2:136). Effort: S.

### G3-F7 (OPP) Corrected package proposal: one builder, every field sourced

The package builder should become a pure function `buildCtwaDraftPackage(ctx)` (new module beside creative-publish-function.ts, which currently inlines all constants) with every field traceable to a system source. Campaign: objective OUTCOME_ENGAGEMENT, status PAUSED (unchanged), special_ad_categories [] (vertical constant, comment citing R2:37), bid_strategy unchanged, daily budget floor derived from getAccountSummary().currency (meta-ads-client.ts:264; planned, D4-F6 and spec section 11). Ad set: destination_type WHATSAPP, promoted_object { page_id } from meta-ads Connection credentials.pageId (creative-publish-preconditions.ts:103), optimization_goal CONVERSATIONS, billing_event IMPRESSIONS, targeting { geo_locations.countries: [org country from org profile, per D4-F6], age_min: 18 (R2:23) }, status PAUSED. Creative: page_id as above; video_id from the existing upload checkpoint; message from the new governed primaryText brief field (G3-F5) with a length guard; call_to_action { type: WHATSAPP_MESSAGE, value: { app_destination: WHATSAPP } }; CTWA welcome message from the new governed welcomeMessage brief field. Ad: unchanged, PAUSED. Preconditions feed the builder: accessToken and accountId from meta-ads Connection credentials (preconditions:89-94, plus the D8-F6 status fix); wabaId, phoneNumberId, displayPhoneNumber from the whatsapp Connection credentials written at onboarding (whatsapp-onboarding.ts:135-140) for the F6 binding probe; aiDisclosure from the job (G3-F4). Inbound mirror so the loop closes: resolveCampaignId wired in apps/chat/src/main.ts (fresh MetaAdsClient per call) and the ad-level booked-join fallback on ConversionRecord.sourceAdId (schema.prisma:2058) against job.metaAdId. All four create payloads validated once against a live sandbox account, which the spec already lists as a go-live blocker. Tag: new. Effort: M.

## Seams and non-duplication

D4-F6 owns geo-country derivation and the currency-aware budget floor; this report inherits both and adds only age_min to the same targeting object. D8-F6 owns Connection.status; G3-F6 adds the binding axis on top of it. G2's booked leg consumes the sourceCampaignId this report shows is never populated (G3-F3); the fix lands in chat wiring and the conversion-store join, not in G2's read path. The CAPI dispatcher is keyed on sourceAdId and currently hardcodes currency SGD (apps/api/src/bootstrap/conversion-bus-wiring.ts:11,20); it is off per the moat scorecard and out of G3 scope, noted here only because the same resolver fix feeds it.
