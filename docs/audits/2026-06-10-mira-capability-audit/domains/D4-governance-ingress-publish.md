# D4. Governance, ingress, and the publish spine

Domain auditor: d4. Baseline: branch docs/mira-capability-audit (HEAD 84083f0c). All file paths relative to the worktree root. All claims verified against live code in this worktree; specs cited only for reconciliation.

## Scope and method

This domain enumerates every Mira-related intent registration, traces every Mira mutating path to PlatformIngress.submit(), and audits the publish spine end to end: approval policy and binding hash, the creative spend gate (p2a-iii), system_auto_approved short-circuit ordering, the PAUSED structural ceiling, the page-id setter, WorkTrace coverage, and idempotency. Method: read the registrars and ingress first, then follow every submitter (routes, crons, Inngest functions, delegation tools, workflow handlers) and every candidate bypass (mira-decision route, taste sweep, attribution, direct Prisma writes), then reconcile against the three publish-adjacent specs and the 2026-06-03 roadmap. Pinning tests were located for each safety claim rather than re-run.

An independent adversarial verification pass (2026-06-10) re-opened every citation in this report against the worktree, hunting for refuting counter-evidence (guards in callers, fixes elsewhere, pinning tests, ingress-side validation). All nine findings survived; none were refuted; two carry in-place corrections (D4-F2 gains test-enforced-doctrine counter-context, D4-F4 fixes a client method name).

## Findings

### D4-F1 (verified). P1. The full creative governance install exists only for org_dev; every real pilot org gets a dark Mira

Claim: seedMiraCreativeDeployment is the single function that installs the creative deployment, the creative.job.\* allow policy, the publish mandatory-approval policy, the handoff allow+approval policies, the compose allow policy, the autonomous+spendAutonomy posture, the $15 threshold, and the house creator. Its only production caller is the dev seed for org_dev. The shipped pilot runbook covers only OrgAgentEnablement.

Evidence: packages/db/prisma/seed.ts:611 ("await seedMiraCreativeDeployment(prisma, \"org_dev\")"); packages/db/src/seed/creative-governance.ts:20-24 ("until that lands, a pilot org needs seedMiraCreativeDeployment(org) run explicitly or creative.job.\* default-denies"); docs/runbooks/2026-05-29-mira-pilot-enablement.md runs only seedMiraPilotOrgs. The provisioning runbook commit 9f213538 (docs/runbooks/provisioning.md) is on a different branch, not on main, and covers infra, not this seed.

Impact on the north star: fail-closed but capability-dark. An enabled pilot org sees the /mira feed, then every brief 409s (creative_deployment_not_provisioned, mira-brief.ts:115-121) or default-denies; publish, spend gate, handoffs, and self-brief are all inert. The act leg of the loop cannot start for any paying clinic.

Recommendation: add a one-shot provisioning script + runbook step that runs seedMiraCreativeDeployment for the pilot org (it is already idempotent), and a smoke assertion (submit a draft, expect execute; submit publish, expect park). Tag: planned (the seed NOTE names this the pending pilot-enablement workstream). Effort: S.

### D4-F2 (verified, corrected). P1. The governed publish seam has no product surface, and the absence is doctrine-enforced by guard tests

Claim: nothing in the dashboard can trigger POST /creative-jobs/:id/publish or read publish state. Grep for "publish", "metaPublishStatus", and "parked_paused" across apps/dashboard/src returns zero functional matches (only landing copy and the desk empty-state line "Nothing is published without you", mira-config.ts:25). The marketplace api-client has submit/approve/estimate methods but no publish method (apps/dashboard/src/lib/api-client/marketplace.ts:331-371), and there is no Next proxy route for it (per the dashboard-needs-proxy-route convention, a missing route.ts means the endpoint is unreachable from the browser).

Evidence: route exists and is correct at apps/api/src/routes/creative-pipeline.ts:273-321; zero dashboard consumers (proxy route inventory: creative-jobs list/get/approve/estimate and mira desk/brief/creatives/decision only; no publish route.ts).

Correction note (verification pass): the gap is not merely an unpicked-up slice; it is actively pinned as M1 draft-only doctrine by guard tests. apps/dashboard/src/components/cockpit/**tests**/mira-copy-hygiene.test.tsx:42 forbids the standalone word "Publish" across Mira cockpit surfaces (only "not published" reassurance copy is allowed), and the creative detail page test rejects affirmative publish/launch CTAs (apps/dashboard/src/app/(auth)/mira/creatives/[id]/**tests**/creative-detail-page.test.tsx:140-156). The claim (no product surface, act leg curl-only) verified exactly; the correction is that closing it requires a deliberate doctrine revision, not just wiring.

Impact: the publish-and-learn loop (Keep a winner, park a PAUSED draft, finalize in Ads Manager, attribute) is operable only by curl/API key. The publish-paused spec declared dashboard UI a non-goal of PR B (spec section 2 non-goals), but no later slice picked it up; the roadmap closes slice 1 without it.

Recommendation: a Publish action on the Mira desk for kept+complete creatives: one client method, one proxy route, reuse the 202 PENDING_APPROVAL contract; surface the parked approval in the existing Inbox. This must land together with an amendment to the copy-hygiene and detail-page guard tests (allow a governed Publish CTA on kept+complete creatives) so the doctrine change is explicit, not accidental. Tag: extends. Effort: M.

### D4-F3 (verified). P1. creative.job.submit trusts a client-supplied deploymentId and ignores the ingress-resolved DeploymentContext

Claim: the submit workflow binds the new AgentTask and CreativeJob to input.deploymentId / input.listingId taken verbatim from the request body, with no check that the deployment belongs to workUnit.organizationId and without using workUnit.deployment (which DeploymentResolver resolved org-scoped at ingress, doctrine 5). The sibling handlers do this correctly: the concept-draft workflow re-reads the deployment and conflates org mismatch with not-found (creative-concept-draft-workflow.ts:96-104), and the decision workflow checks job.organizationId (creative-job-decision-workflow.ts:20).

Evidence: apps/api/src/services/workflows/creative-job-submit-workflow.ts:6-11 (cast, no parse), :29-35 (taskStore.create with input.deploymentId), :57-89 (job create); packages/db/src/stores/prisma-creative-job-store.ts:71-89 (create persists deploymentId unchecked); route input SubmitBriefInput at apps/api/src/routes/creative-pipeline.ts:16-21 carries deploymentId/listingId from the client.

Impact: a request can create org A jobs bound to org B's (or a non-creative) deployment id. Every read path I traced is org-scoped first, so no cross-org read or spend authority leak was found, but the tenant linkage of a governed, spend-bearing record is silently corruptible, and downstream per-deployment artifacts (taste buckets, performance history filters, ugc trust reads) key off that field. The mira-brief route is safe because it resolves the deployment itself (mira-brief.ts:112-121); the older /creative-jobs route is the exposed one.

Recommendation: in buildCreativeJobSubmitWorkflow, take deploymentId from workUnit.deployment and verify org ownership via deploymentStore (mirror the concept-draft pattern); treat the body fields as display hints or drop them. Tag: new. Effort: S.

### D4-F4 (verified, corrected). P2. Operator-facing creative submits carry no ingress idempotency key, and the marketplace client sends no key at all

Claim: none of the three creative routes pass idempotencyKey into platformIngress.submit (creative-pipeline.ts:87-94, 198-205, 292-299; mira-brief.ts:126-140; a grep for idempotencyKey across both route files returns zero matches), so the D1 claim-first replay guard never engages for operator submits. The replay protection that exists is the HTTP middleware response cache, which (a) only works when the client sends an Idempotency-Key header, and (b) stores the response in onSend, leaving a concurrency window where two in-flight requests with the same key both pass the preHandler check and execute (apps/api/src/middleware/idempotency.ts:89-141, no claim leg). The Mira desk proxy generates a key per request (apps/dashboard/src/app/api/dashboard/agents/mira/brief/route.ts:21), but the marketplace client's submitCreativeBrief sends no header at all (marketplace.ts:318-335), and the shared request helper adds only Content-Type and Authorization (apps/dashboard/src/lib/api-client/core.ts:7-16).

Correction note (verification pass): the client method is submitCreativeBrief, not createCreativeJob as originally written; every other leg verified as stated.

Impact: a retried or double-submitted brief creates two CreativeJobs and two paid render pipelines. Bounded by the spend gate only for over-$15 renders on provisioned orgs. By contrast every system path is keyed (self-brief weekly keys, handoff handoff:riley:..., delegate delegate:<wu>:<intent>:<hash>).

Recommendation: thread the validated Idempotency-Key header into the ingress submit (the trace lookup is already org-scoped), and add the header to the marketplace client. Tag: extends (doctrine 6). Effort: S.

### D4-F5 (verified). P2. The system_auto_approved short-circuit structurally precedes the spend lever; the no-spend invariant is comment-enforced only

Claim: GovernanceGate returns execute for approvalMode "system_auto_approved" at governance-gate.ts:100-108, before policy load and before applySpendApprovalThreshold at :178-185, so any such intent can never be parked by the spend gate (the #931 pattern). Today the only Mira use is creative.concept.draft, which I verified is genuinely no-spend (handler never imports creative-pipeline, never emits an event, creative-concept-draft-workflow.ts:140) and internal-trigger-only (contained-workflows.ts:406). The guard that spend-bearing intents must not use this mode lives in comments (contained-workflows.ts:383, 394-399) and review discipline, not in code.

Impact: latent. One future registration mistake (a "draft" intent that grows a paid side effect, or a financial operator intent copied from operator-intents.ts:110-121) silently re-opens the gap.

Recommendation: a registration-time assertion or boot-time lint: system_auto_approved requires budgetClass "cheap" and a registered no-spend marker, or at minimum a pinning test enumerating every system_auto_approved registration and asserting none is spend-bearing. Tag: extends (#931 pattern). Effort: S.

### D4-F6 (verified). P2. Publish draft package constants are not pilot-complete: SG-only targeting and a non-currency-aware minimum budget

Claim: the paused draft hardcodes DRAFT_TARGETING { geo_locations: { countries: ["SG"] } }, a 500-minor-unit daily budget, and a placeholder link (creative-publish-function.ts:15-21). The publish spec itself flags the currency-aware minimum budget as a go-live refinement (spec sections 6.6 and 11) and the code comment repeats it (creative-publish-function.ts:13-15).

Impact: for the SG/MY wedge, an MY clinic's parked draft arrives with SG geo targeting and a possibly Meta-rejected budget constant in the wrong currency scale; rejection surfaces as a retried-then-dead-lettered Meta error rather than a crisp precondition. All spend-safe (PAUSED), but it erodes the "operator finalizes a sensible draft" promise.

Recommendation: derive country from the org profile (pdpaJurisdiction already exists on contacts; org config is the right home) and the budget floor from getAccountSummary currency, per spec section 11. Tag: planned (publish spec 6.6/11). Effort: S.

### D4-F7 (verified, corrected). P2. Decorative registration fields invite misreading: approvalPolicy, parameterSchema, and timeoutMs have no enforcement at dispatch

Claim: the policy engine never reads IntentRegistration.approvalPolicy (the real gate is the seeded policy's policyApprovalOverride, policy-engine.ts:327-332, 421-422; grep for approvalPolicy across policy-engine.ts, platform-ingress.ts, and governance-gate.ts returns nothing); parameterSchema is {} on every workflow intent (contained-workflows.ts:489) and validated nowhere at ingress or in the gate. timeoutMs is one hop less dead than first stated: the constraint resolver maps registration.timeoutMs into ExecutionConstraints.maxRuntimeMs (packages/core/src/platform/governance/constraint-resolver.ts:45), but no execution mode enforces that constraint: workflow-mode and operator-mutation-mode receive `_constraints` unused (modes/workflow-mode.ts:45, modes/operator-mutation-mode.ts:57), and skill-mode reads only constraints.trustLevel (modes/skill-mode.ts:93). The contained-workflows registrations document this honestly in comments (:379-389, :428-432), but the registry surface still reads like a contract.

Correction note (verification pass): the original "zero consumers" wording missed constraint-resolver.ts:45; the constraint is produced but never consumed by any dispatch path, so the conclusion (decorative at dispatch) stands with sharper evidence.

Impact: a future intent author who sets approvalPolicy "always" and believes it gates (the exact trap the publish spec section 4.1 warns about) ships an ungated action; a hung compose LLM call is bounded not by the registered 30s but by the skill executor's own ExecutionPolicy budget (default maxRuntimeMs 120s, skill-executor.ts:362-364) plus the Inngest function timeout, two surfaces the registration author never touched.

Recommendation: either enforce (a registry lint rejecting approvalPolicy values stronger than what a seeded policy backs; a dispatch-level timeout) or rename/annotate the fields as documentation-only in the type itself. Tag: new (confirms the prior-session decorative-timeoutMs hypothesis). Effort: M.

### D4-F8 (verified). P2. Dead pipeline registrar still exports governed-looking creative intents

Claim: registerPipelineIntents (creative.produce, creative.ugc.produce, approvalPolicy "threshold", triggers including chat) is exported from the platform index but has no caller anywhere in apps or packages outside its own file.

Evidence: packages/core/src/platform/pipeline-intent-registrar.ts:24-45; caller grep returns only the index export (packages/core/src/platform/index.ts:127).

Impact: none at runtime today; the risk is accidental future wiring of intents whose decorative threshold policy would mean default-deny at best and confusion at worst, plus map noise for auditors.

Recommendation: delete the registrar (mind the build-type-checks-dead-files gotcha: remove the index export in the same change). Tag: new. Effort: S.

### D4-F9 (verified). P2. Keep/Pass review verdict is a sanctioned ingress bypass with an entitlement asymmetry

Claim: POST /agents/mira/creatives/:id/decision writes CreativeJob.reviewDecision directly via prisma.creativeJob.updateMany (mira-decision.ts:59-63), bypassing ingress entirely: no WorkTrace, no governance, no billing entitlement check. The bypass is deliberately allowlisted ("draft-only, reversible, no cross-agent writes", route-allowlist.yaml:145-146) and the taste sweep treats the review columns as the durable event record (creative-taste-sweep.ts:193-199), so there is a durable record. But reviewDecision === "kept" is also the publish precondition (creative-publish-preconditions.ts:65) and the taste-learning input, so the gesture sits on the spend path's critical chain while being the one Mira mutation an unentitled org can still perform.

Impact: low. Publish itself re-parks under the mandatory policy and re-checks entitlement at ingress, so claim and spend safety hold; the asymmetry is mostly an audit-trail and consistency wrinkle.

Recommendation: keep the route as is; add the entitlement check (or an ActivityLog row) if Keep ever drives an automated downstream action beyond eligibility. Tag: extends. Effort: S.

## What is sound

The spine itself is in genuinely good shape, with real tests pinning each safety property:

- One front door, consistently used. Every spend-capable Mira path submits through PlatformIngress: both brief routes, the approve route, the publish route, the Riley handoff closure, the Alex delegate tool, and both self-brief submits (contained-workflows.ts:109-134, 503-581; delegation-submitter.ts:48-62). The only direct writes are the allowlisted draft-only review verdict and derived-data learning projections.
- The publish gate composes correctly in the engine. Allow does not short-circuit, require_approval sets a sticky override that beats identity tolerance, deny wins over everything, and an unmatched intent default-denies (policy-engine.ts:309-348, 588-589). The anchored publish rule cannot leak onto continue/stop. All pinned by creative-publish-gate.test.ts:134-152 and mira-brief-compose-gate.test.ts:150-186 against the REAL GovernanceGate and the real SKILL.md-derived registration.
- Mandatory survives autonomy. The spend lever engages only for an explicitly autonomous deployment with spendAutonomy true, relaxes only approvalLevel "standard", never touches deny, and escalates over-threshold executes (spend-approval-threshold.ts:47-105). extractSpendAmount is Number.isFinite-guarded (spend-limits.ts:19-25), closing the NaN-blind class.
- Activation is structurally unreachable. PAUSED is hardcoded at all three creation sites and updateCampaignStatus throws on ACTIVE (meta-ads-client.ts:276, 296, 348, 355-360); the publish chain never calls it, and the step checkpoints make retries reuse persisted meta ids rather than recreating objects (creative-publish-function.ts:109-173).
- Approve ends in dispatch-or-recovery with payload authority. bindingHash is required and revision-matched at approve (approvals.ts:239-244, respond-to-parked-lifecycle.ts:212-214); the frozen payload is written to the WorkTrace before dispatch and execution reads from the trace; failures transition to recovery_required; governance is evaluated exactly once per work unit (the post-approval dispatch synthesizes an execute decision rather than re-running the gate, platform-lifecycle.ts:374-383).
- Crons are governance-literate. The self-brief worker reads its flag per invocation, fails closed on enablement and deployment resolution, keys every submit per ISO week, branches on approvalRequired before touching outputs, and names entitlement and in-flight-claim skips (mira-self-brief.ts:140-236). The handoff enrichment composes before the park so the human approves exactly what dispatches (handoff-brief-enrichment.ts:31-38). All seven Mira-adjacent Inngest functions carry onFailure handlers (inngest.ts:969-1071, 1176; creative-publish-function.ts:189).
- The runtime identity seams are guarded. Boot throws unless skills/mira/SKILL.md declares slug "creative" distinct from alex (skill-mode.ts:140-144); the compose executor is registered under that slug with zero tools and zero conversation hooks, with claim safety placed at the SKILL.md source, the desk review, and the mandatory publish approval (skill-mode.ts:667-691); the SKILL.md body contains no dotted-triple tool tokens.
