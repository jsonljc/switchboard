# D1: Polished Pipeline Engine

Domain auditor: d1. Baseline: branch docs/mira-capability-audit (tracks origin/main). Date: 2026-06-10.

## Scope and method

This domain covers the polished CreativeJob pipeline: stage flow (trends, hooks, scripts, storyboard, production), model and provider selection, cost estimation and its relationship to governance spend, stage failure and retry behavior, quality gates, output and durable asset landing, and the sufficiency of brief context for medspa-quality output. Every claim below was verified by reading live code in this worktree. Model id validity was checked against the Anthropic model catalog (claude-api reference, cached 2026-05-26). Specs reconciled: 2026-06-03-mira-roadmap.md, 2026-06-04-mira-slice3-generator-quality-design.md, 2026-06-03-creative-durable-asset-storage-design.md. UGC pipeline internals are out of scope except where shared infrastructure (call-claude, cost constants) or contrast (budget enforcement) is load-bearing.

The actual stage list is five stages, not seven: trends, hooks, scripts, storyboard, production (`packages/creative-pipeline/src/stages/run-stage.ts:52`). Image generation is an optional sub-step of storyboard; video generation, voiceover, captions, and assembly are all inside the single production stage.

A creative director's honest inventory: as designed, the pipeline can produce competent but generic short-form ad strategy (angles, hooks, 15-30s scripts with timing), scene-level storyboards, and 5-10s Kling b-roll clips stitched into one 9:16 video with a synthetic voiceover and burned-in captions. It cannot produce on-brand visuals (it never sees brand assets or images), real clinic or product footage, human presenters in polished mode, per-variant assets (F5), per-platform aspect variants in one run (only platforms[0] is used, video-producer.ts:105), or compliance-safe medical-aesthetics copy. As shipped, with F1 and F2 live, a polished job produces nothing usable against the live API.

## Findings

### D1-F1 (P0): every LLM stage calls a nonexistent Anthropic model id; the pipeline cannot run live (verified)

Claim: `DEFAULT_MODEL = "claude-sonnet-4-5-20250514"` (call-claude.ts:5) is not a real model id. Claude Sonnet 4.5's dated id is `claude-sonnet-4-5-20250929`; the `20250514` date belongs to `claude-sonnet-4-20250514` (Sonnet 4). Unknown model ids return HTTP 404 not_found_error from the Messages API. No caller overrides the model: `model: options.model ?? DEFAULT_MODEL` (call-claude.ts:77,102) and a repo-wide grep shows no call site passing `model` (the only other model literal is `"dall-e-3"` in image-generator.ts:25). This breaks trends, hooks, scripts, and storyboard in polished mode, plus the UGC script writer and frame-QA, which share callClaude/callClaudeWithImages.

Evidence: call-claude.ts:5 `const DEFAULT_MODEL = "claude-sonnet-4-5-20250514";`; consumers per grep: trend-analyzer.ts, hook-generator.ts, script-writer.ts, storyboard-builder.ts, ugc/ugc-script-writer.ts, ugc/frame-qa-deps.ts. Unchanged since #212; #897 touched the file and kept the constant. All tests mock the Anthropic SDK, so nothing pins the id against the live API.

Impact on the north star: Mira's entire generative leg is dead on arrival in any live environment; nothing downstream (publish, learning loop, taste) can ever receive a real polished asset. This also confirms the prior-session hypothesis that governance model constraints never reach pipeline executors: model selection is a constant, with no config or governance path at all.

Recommendation: replace with a valid alias (for example `claude-sonnet-4-6`), thread model selection through injected config per stage, and add one cheap live-canary test (skipped in CI without a key) that asserts the configured model id resolves via the Models API. When governance model-tier constraints exist on a deployment, honor them here. Tag: new. Effort: S.

### D1-F2 (P0): the operator's production-tier choice never reaches the render; pro tier, assembly, and polished publishability are unreachable (verified)

Claim: the runner loads the job exactly once in a memoized Inngest step before any approval (`step.run("load-job", ...)`, creative-job-runner.ts:63) and reads `productionTier: job.productionTier ?? "basic"` from that snapshot (creative-job-runner.ts:123). The tier is first written by the decision workflow at the scripts gate (`action === "continue" && job.currentStage === "storyboard"` then `updateProductionTier(..., input.productionTier ?? "basic")`, creative-job-decision-workflow.ts:47-53), which happens after the snapshot. Inngest replays memoize step output as JSON, so on every resume load-job returns the original row, where `productionTier` is null (`productionTier String?`, schema.prisma:1386; creation writes no tier, creative-job-submit-workflow.ts:57-69). Therefore production always runs basic tier: run-stage.ts:153-166 never constructs ElevenLabs, Whisper, or the VideoAssembler; video-producer.ts:144-149 returns raw clips; durableAssetUrl is never set; creative-publish-preconditions.ts:72-77 always fails with CREATIVE_ASSET_NOT_DURABLE for polished jobs.

A second wrinkle compounds it: spend is priced at the production gate from the request's claimed tier (`computeRenderSpend(job, parsed.data.productionTier)`, apps/api/src/routes/creative-pipeline.ts:193), while persistence happens only at the earlier scripts gate, and execution honors neither. The governance record can say the operator approved a pro render at pro cost while the system renders basic.

Evidence: creative-job-runner.ts:63,123; creative-job-decision-workflow.ts:47-53; schema.prisma:1386; run-stage.ts:142,153-166; creative-publish-preconditions.ts:72-77. The roadmap (section 2) states go-live blocker 1 cleared "at the polished video-producer.ts assembly seam"; the seam exists but no polished run can reach it. Runner tests use pass-through mock steps with no replay memoization, so the defect is untested.

Impact on the north star: the publish leg of the loop is structurally dead for polished creatives, and approval state (the doctrine says approval is lifecycle state) diverges from execution. Even after F1 is fixed, no polished creative can ever be published as a paused Meta draft.

Recommendation: re-read the tier in a fresh, distinctly-named step after the storyboard-gate wait resolves (or carry the tier on the stage.approved event payload and consume it from the wait result), and align the gate at which tier is collected with the gate at which spend is priced. Add a pinning test whose mock step tools memoize first-run values. Tag: new. Effort: S.

### D1-F3 (P1): the cost estimator double-counts scripts; the approved spend overstates the executed render by roughly the script count (verified)

Claim: estimateCost flattens scenes across ALL storyboards and then multiplies by scriptCount again: `allScenes = storyboard.storyboards.flatMap(...)`, `totalScenes = allScenes.length * scriptCount`, `klingCost = (sum over allScenes) * scriptCount` (cost-estimator.ts:87-101). But the storyboard builder creates one storyboard per script ("Create one storyboard per script", storyboard-builder.ts:65) and production renders each scene exactly once (video-producer.ts:108-141). computeCreativeEstimates passes `scriptCount = scripts.length` with the full multi-storyboard output (creative-render-spend.ts:62-66). For N scripts the spend signal and the estimate readback are about N times the real cost. The unit tests pin single-storyboard fixtures and only assert monotonicity, so they cannot catch this (packages/creative-pipeline/src/stages/**tests**/cost-estimator.test.ts:6-60; the second fixture at src/**tests**/cost-estimator.test.ts:18 is also single-storyboard).

Impact: the governance spend threshold fires on inflated numbers (operators decline renders they would approve; any ledger entry recorded from spendAmount is wrong by N times). Conservative direction, but the doctrine goal is correct governance numbers, not merely safe ones. Note also that for polished the estimate is purely an approval signal: nothing enforces any ceiling at execution time (contrast UGC production's budget cap, ugc/phases/production.ts:278-279).

Recommendation: remove the scriptCount multiplier (cost is the sum over flattened scenes), or estimate per storyboard; add a two-storyboard fixture test asserting parity with the production loop's call count. Tag: new. Effort: S.

### D1-F4 (P1): production completes as success with zero clips; a missing KLING_API_KEY guarantees it; the dead-letter contract is inert for generation failures (verified)

Claim: every per-scene Kling failure is swallowed into an errors array (video-producer.ts:132-139) and the stage returns success even when `clips` is empty (video-producer.ts:144-149 for basic; the assembly block is simply skipped for pro). The stage never throws, so Inngest retries and the wired onFailure dead-letter handler (inngest.ts:1093-1101) never engage for generation failure: the job advances to complete and appears in the review feed with no playable asset, with errors visible only inside the stageOutputs JSON. The failure mode is guaranteed when KLING_API_KEY is unset because run-stage constructs its own client with an empty key (`new KlingClient({ apiKey: process.env.KLING_API_KEY ?? "" })`, run-stage.ts:144), bypassing the bootstrap guard that only constructs a Kling client when the key exists (inngest.ts:234-235, injected to the UGC runner only). An empty key yields a non-transient 401, which kling-client.ts:105-106 throws immediately for every scene.

Impact: silent zero-output completions burn operator trust and approval cycles; transient Kling outages permanently degrade a job instead of retrying; two construction regimes for the same provider dependency drift.

Recommendation: throw when clips.length === 0 (or below a scene-coverage threshold) so retries and onFailure fire; inject the shared bootstrap klingClient into the polished path; consider per-scene checkpointing (step-per-scene or persisted partial results) so a retry does not re-buy already-rendered clips. Tag: extends (2026-05-25 Inngest failure contract). Effort: S-M.

### D1-F5 (P1): pro-tier assembly concatenates every script into one video with one voiceover reading all scripts back-to-back (verified)

Claim: the voiceover joins all scripts (`input.scripts.scripts.map((s) => s.fullScript).join("\n\n")`, video-producer.ts:156) and assembly stitches the flat clip list across ALL storyboards into a single mp4 (video-producer.ts:206-231), with text overlay timing cursoring across storyboards as if they were one timeline (video-producer.ts:208-221). The script writer produces up to 5 scripts (script-writer.ts:66), each with its own storyboard, so the only assembled deliverable is one long video containing every variant end to end with a voiceover that reads all five scripts serially. The durable-asset spec line 190 ("the polished pipeline produces exactly one assembled video") documents the behavior without addressing that the one video is a concatenation of variants.

Impact on the north star: the loop's whole point is testing variants against measured outcomes (Riley reallocates toward what pays). The polished pipeline structurally cannot emit per-variant assets, so creative-level attribution and taste learning have nothing meaningful to compare even if publish worked.

Recommendation: assemble per storyboard with that script's own voiceover and upload N durable assets under per-variant keys (or as a minimal v1, render only scripts[0] and say so). Tag: new. Effort: M.

### D1-F6 (P2): brief context is too thin for medspa-quality output; brand voice and images barely enter the pipeline (verified)

Claim: the brief is productDescription, targetAudience, platforms, brandVoice, references, productImages (creative-job-submit-workflow.ts:18-27); there are no vertical, offer, locale, or compliance fields, so medspa context exists only if a caller writes it into free text. brandVoice reaches only the scripts stage (run-stage.ts:104-114); trends, hooks, and storyboard prompts never see it. productImages are interpolated as URL strings into prompt text ("Product Images (incorporate into visual direction)", storyboard-builder.ts:96-98); no vision call is made, so the model cannot see the product, the clinic, or any brand asset. Stage prompts are generic DTC performance-ad framing (trend-analyzer.ts:21, hook-generator.ts:25-31) with no medical-aesthetics claim guardrails anywhere.

Impact: for SG/MY aesthetic clinics the output will be generic wellness-ad copy with model-imagined visuals, off-brand and compliance-risky (medical claims, before-after rules). This blunts the "Mira turns diagnoses into winning creative" leg even after the P0s are fixed.

Recommendation: thread brandVoice through all four LLM stages; extend the brief with vertical, offer, and compliance-notes fields rendered as a prompt block; pass productImages as image content blocks (callClaudeWithImages already exists) in the storyboard stage. Tag: extends (roadmap slice 3 generator quality; handoff brief enrichment workstream). Effort: M.

### D1-F7 (P2): the trend stage fabricates trend signals; there is no live trend source (verified)

Claim: the trends stage asks the model to produce "Current trend on this platform" with no retrieval, tool, or data feed (trend-analyzer.ts:39-46); output is labeled trendSignals and consumed downstream as fact (hook-generator.ts:83-84). These are model priors as of its training cutoff presented as current platform trends.

Impact: hooks and scripts get anchored on invented trends; for a system whose pitch includes reading the market, this is an honesty gap a pilot customer can catch.

Recommendation: either rename and reframe the stage as strategy angles (cheap), or wire a real signal source (Meta ad library, platform trend APIs) behind an injected provider. Surface provenance in the review UI. Tag: new (OPP if reframed as roadmap work). Effort: M.

### D1-F8 (P2): DALL-E reference image URLs expire before production consumes them across human-paced approval gates (verified)

Claim: DalleImageGenerator returns the OpenAI-hosted URL (image-generator.ts:24-41), which is a short-lived signed link (on the order of an hour). The storyboard stage stores it in scene.referenceImageUrl (storyboard-builder.ts:137), and production uses it as the Kling image2video first frame (video-producer.ts:120) only after the storyboard approval gate, which can wait up to 24h (creative-job-runner.ts:151-155). By render time the URL is likely dead, so image2video fails per scene and is swallowed per F4. Reference image generation is also sequential per scene and absent from estimateCost (no DALL-E line item, cost-estimator.ts).

Recommendation: upload reference images through assetStorage at generation time (the client is already injected into the runner) and store durable URLs; add an image-generation line to the estimate when generateReferenceImages is true. Tag: extends (durable-asset spec covers only the final MP4). Effort: S-M.

### D1-F9 (P2): four sequential human gates with silent 24h timeout-stops; only one gate carries spend (verified)

Claim: the runner waits for an approval after every stage (creative-job-runner.ts:148-163); timeout or stop calls jobStore.stop and returns, with no notification or resume path; a stalled operator kills the job silently after 24h at any of four gates. Only the approve-into-production gate carries a spend signal (creative-render-spend.ts:44-67 returns null before the storyboard exists); the trends, hooks, and scripts gates govern free Claude output.

Impact: heavy operational drag against the north-star shape (autonomous production with one meaningful human gate); jobs die quietly mid-funnel.

Recommendation: auto-advance the free stages (flag-gated), keep the spend gate mandatory; on timeout-stop emit an escalation or notification event instead of silence. Tag: OPP. Effort: M.

### D1-F10 (P2): stage approval events match on jobId only; a duplicate continue can release the wrong gate, including the paid one (verified)

Claim: every wait uses `match: "data.jobId"` (creative-job-runner.ts:151-155) and the decision workflow emits `{ jobId, action }` with no stage field (creative-job-decision-workflow.ts:74-77). Inngest matches events against waits registered at delivery time, so a duplicate or delayed continue event that lands after the next wait registers releases that next gate. Because spendAmount is computed at request time from the then-existing storyboard (apps/api/src/routes/creative-pipeline.ts:190-194), a continue approved at the scripts gate carries no spend signal, and if its duplicate releases the production gate the paid render proceeds without a spend-gated approval. The window is narrow (the duplicate must arrive after the next stage completes) but the bypassed gate can be the paid one.

Recommendation: include the awaited stage in the stage.approved payload and match on it (ids, not string filters, per the Inngest contract); have the decision workflow reject a continue whose requested stage does not match the job's current gate. Tag: new. Effort: S.

## What is sound

- The async failure contract is honored at registration: mode dispatcher and job runner both carry retries 3 plus makeOnFailureHandler dead-letter wiring (inngest.ts:1077-1103).
- Durable upload uses deterministic per-job keys so stage retries overwrite rather than orphan objects, and a storage failure propagates loudly instead of fake-succeeding with a temp path (video-producer.ts:259-277 and the comment block above it).
- The spend signal is computed server-side so the operator cannot understate cost, and it is Number.isFinite-guarded (creative-render-spend.ts:79-97), which refutes the prior-session NaN hypothesis for the money path; Kling's duration parse is also NaN-safe (`parseFloat(...) || 5`, kling-client.ts:88). The one unguarded numeric, ElevenLabs `parseFloat(header ?? "0")` (elevenlabs-client.ts:51), is informational only.
- Inter-stage contracts are parse-don't-cast: every stage Zod-parses its predecessors' outputs (run-stage.ts:83,102-103,120,140-141), and the creative descriptor resolves hookRef with explicit NaN and bounds guards (creative-descriptor.ts:25-33).
- The assembler is SSRF-hardened via a safe-URL policy and body-size limits (video-assembler.ts:6-12,41-45).
- Approval is lifecycle-governed: stage.approved is emitted only from the post-governance decision workflow, with mode-aware stop semantics and a not-awaiting-approval guard (creative-job-decision-workflow.ts:28-78).
- Slice-3 items verifiably shipped where claimed: Whisper receives the OpenAI key, never the Anthropic one (run-stage.ts:157-164, spec 3.6), and UGC video prompts compose SceneStyle/UgcDirection rather than raw script text (ugc/video-prompt.ts:53 with SceneStyleSchema.safeParse), settling that hypothesis as fixed for UGC; the polished optimizer remains template-based by documented decision (slice-3 spec 3.2 "Rejected: LLM prompt optimization").
- Slice-2 learning inputs are wired with degrade-to-nothing semantics: taste context loads in a step that returns [] on error and history feeds via safeParse (creative-job-runner.ts:75-99).

## Open questions

- Has any polished job ever completed end to end against live providers in any environment? F1 predicts no; run logs or the CreativeJob table would settle when the model id and tier defects were last masked by mocks.
- Does the pilot plan intend first publishes from polished or UGC assets? UGC kept assets do gain durableAssetUrl via the shared storage client, so F2's urgency depends on which mode the pilot leans on.
- The Kling client authenticates with a static Bearer token against api.klingai.com (kling-client.ts:114-119); Kling's public API has historically required JWT signing from an access-key/secret pair. Whether this contract matches the current vendor API cannot be verified from the repo and needs one live smoke test.
- Was estimateCost's scriptCount multiplier written for an earlier one-storyboard design (the test fixture suggests so), and are any persisted ledger entries already inflated by it?
- Brief enrichment (MIRA_HANDOFF_BRIEF_ENRICHMENT_ENABLED, self-brief cron) is D3's domain: once those flags flip, does enriched brief content land in fields the stage prompts actually render, or only in pastPerformance?
