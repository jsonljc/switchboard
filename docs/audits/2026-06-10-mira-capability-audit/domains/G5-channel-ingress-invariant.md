# G5. Channel-as-ingress invariant for Mira: settled

Gap question: `creative.respond` is auto-registered alongside compose with trigger "chat" allowed. Can any channel-binding path resolve managed inbound chat to the creative deployment today, what happens if it does, and what (if anything) pins the behavior? Is the invariant intact by construction, by policy default-deny, or only by accident of current seed data?

Method: every claim below was traced in live code in this worktree, with counter-evidence hunted (guards in callers, fallbacks in stores, fail-open branches, pinning tests). Extends D3 open question 1 (creative.respond unbound on every surface?), D4 open question 4 (silent deny trace), D4-F1 (org_dev-only creative install), and D4-F7 (decorative registration fields).

## 1. The registration is real and chat-triggerable

`registerSkillIntents` registers the declared intent (`creative.brief.compose`) and then a sibling `${slug}.respond` for every skill, explicitly so the ChannelGateway's `${resolved.skillSlug}.respond` submit does not hit intent_not_found (packages/core/src/platform/skill-intent-registrar.ts:48-62). Both registrations carry `allowedTriggers: ["chat", "api", "schedule", "internal"]` (skill-intent-registrar.ts:41). apps/api boots it with `[alexSkill, miraSkill]` where miraSkill is loaded from skills/mira/SKILL.md with frontmatter slug "creative" (apps/api/src/bootstrap/skill-mode.ts:134-150). So `creative.respond` exists in the live IntentRegistry, defaultMode "skill", executor `{ mode: "skill", skillSlug: "creative" }`, and chat is an allowed trigger. Because skills/mira/SKILL.md declares `tools: []`, deriveMutationClass returns "read" and approvalPolicy derives to "none" (skill-intent-registrar.ts:6-11, 29), so the registration itself attaches no approval requirement (this is the D4-F7 decorative-fields family: these derived fields would matter the moment a policy ever allowed the intent).

## 2. Enumeration: every channel-binding resolution path

The managed gateway submits `${resolved.skillSlug}.respond` with trigger "chat" and actor `{ id: sessionId, type: "user" }` after resolving the deployment from the channel token (packages/core/src/channel-gateway/channel-gateway.ts:149, 315-340). Bindings pin a deploymentId, never a skillSlug; the slug comes from the resolved AgentDeployment row (packages/core/src/platform/prisma-deployment-resolver.ts:85-104, 130). The resolver gates on deployment status "active", listing status "listed", and non-null skillSlug (prisma-deployment-resolver.ts:107-119); it has no notion of which slugs are conversational.

Path A, org provisioning (telegram/whatsapp/slack): POST /api/organizations/:orgId/channels creates Connection + ManagedChannel and upserts the DeploymentConnection against the deployment returned by `ensureAlexListingForOrg` (apps/api/src/routes/organizations.ts:312-339). Alex-pinned by construction. Cannot reach creative.

Path B, marketplace telegram binding: POST /api/marketplace/deployments/:id/connections/telegram checks only that the deployment exists and belongs to the caller's org, then creates a DeploymentConnection type "telegram" and registers the Telegram webhook at /webhook/managed/<connId> (apps/api/src/routes/marketplace.ts:567-656, org check at 584-586). No skillSlug restriction. An org-authed caller CAN bind a telegram bot to its creative deployment.

Path C, marketplace web widget: POST /api/marketplace/deployments/:id/connections/widget, same org-only check (marketplace.ts:519-565, check at 536-538), creates a "web_widget" DeploymentConnection and returns the public token. The chat app serves POST /widget/:token/messages straight into the managed gateway (apps/chat/src/endpoints/widget-messages.ts:17-18, 52-53). Also creative-bindable.

Liveness of B and C: the chat app loads EVERY active DeploymentConnection at boot, with adapters for telegram, whatsapp, and slack, and registers /webhook/managed/<connId> (apps/chat/src/managed/runtime-registry.ts:86-111, 151-187); the webhook handler passes the connection id as the resolution token (apps/chat/src/routes/managed-webhook.ts:175), exactly what `resolveByChannelToken("telegram", token)` matches on `{ id: token }` (prisma-deployment-resolver.ts:88-91). The hot provision-notify endpoint only handles ManagedChannel rows (apps/chat/src/main.ts:360-407), so a marketplace-created binding goes live at the next chat restart, but Telegram's webhook is registered immediately at creation (marketplace.ts:630-637).

Path D, single-tenant chat (/webhook/telegram, dev/no-DB): StaticDeploymentResolver pins skillSlug from env `SKILL_SLUG`, default "sales-pipeline" (apps/chat/src/single-tenant/static-deployment-resolver.ts:11; wired at apps/chat/src/main.ts:92-95). `SKILL_SLUG=creative` makes EVERY single-tenant inbound submit creative.respond. With the default `ORGANIZATION_ID` ("default") ingress dies at deployment resolution (no active creative deployment for that org); pointed at org_dev it reaches the same governance stage as B/C. A misconfig path, not a binding.

Path E, operator surfaces: the dashboard has no operator chat that submits respond intents (grep over apps/dashboard/src for `.respond`/ingress submit finds only approval respond plumbing); the chat approval bridge intercepts approval-shaped payloads terminally before any submit (channel-gateway.ts:169-188). Not a path.

Which orgs have a bindable creative target: only a SEEDED creative deployment is chat-resolvable, because the self-serve deploy route never sets skillSlug (marketplace.ts:229-239) and the resolver throws on null skillSlug (prisma-deployment-resolver.ts:117-119). `seedMiraCreativeDeployment` is the only writer of `skillSlug: "creative"` (packages/db/src/seed/seed-mira-creative-deployment.ts:54-76) and its only production caller is the dev seed for org_dev (packages/db/prisma/seed.ts:611); the backing listing is seeded "listed" (packages/db/prisma/seed-marketplace.ts:282 plus status block). So today resolution-to-creative is constructible on org_dev via Paths B and C, and on every future pilot org the moment D4-F1's provisioning workstream runs the same seed (the seed itself names this pending, seed-mira-creative-deployment.ts:148-151).

Answer to sub-question 1: yes, resolution is possible today. Two live product routes create the binding, the registry auto-serves it, the resolver returns skillSlug "creative", and the gateway submits `creative.respond` trigger "chat".

## 3. What happens: a three-layer deny, none of it structural

The chat-to-api hop preserves the request verbatim (org, actor, intent, targetHint) over INTERNAL_API_SECRET and submits through the same PlatformIngress (apps/chat/src/gateway/http-platform-ingress-adapter.ts:26; apps/api/src/routes/internal-ingress.ts:61-74). Inside `submit`: intent lookup succeeds (registered, section 1); the entitlement check is ORG-level billing only, `entitlementResolver.resolve(request.organizationId)` (packages/core/src/platform/platform-ingress.ts:196-209), so Mira enablement (OrgAgentEnablement) is NOT consulted at ingress and does not block; trigger "chat" validates (registration allows it); deployment resolution honors targetHint.skillSlug and re-resolves the active creative deployment org-scoped (apps/api/src/bootstrap/platform-deployment-resolver.ts:13-22). Then the GovernanceGate runs, and here the denials begin.

Layer 1, identity fail-closed (applies to ALL respond intents, not creative-specifically): `loadIdentitySpec` throws "Identity spec not found" for any actor without an IdentitySpec row (apps/api/src/app.ts:535-540; packages/db/src/storage/prisma-identity-store.ts:42-48 returns null, no fallback), and chat actors are raw session ids (channel-gateway.ts:317). The ingress catches and converts to a deny with reasonCode GOVERNANCE_ERROR, persisting a failed trace (platform-ingress.ts:252-279). This is the same fail-closed floor pilot-spine F-16 documented for fresh orgs (docs/audits/2026-06-07-pilot-spine-audit/index.md:190), and the same workstream that will REMEDIATE it for real orgs.

Layer 2, policy default-deny: if an identity resolves, no policy matches `creative.respond` for any org, and the engine's final decision is `policyResult.policyDecision ?? "deny"` (packages/core/src/engine/policy-engine.ts:588-589). Verified against every production policy writer: the creative allow policy matches `"creative.job.*"` (unanchored regex, requires the substring "job"; packages/db/src/seed/creative-governance.ts:47), publish and compose are anchored (`^creative\.job\.publish$`, `^creative\.brief\.compose$`; creative-governance.ts:77, 121), riley pause and handoff are anchored adoptimizer patterns (riley-pause-governance.ts:33; recommendation-handoff-governance.ts:31), the org_dev base allow_read pattern `"*.read|*.list|*.get"` is an invalid regex that evaluates false (packages/db/prisma/seed.ts:266; packages/core/src/engine/rule-evaluator.ts:115-119), and no production code seeds organizationId-null global policies (policy load is global+org, packages/db/src/storage/prisma-policy-store.ts:133-177). The trusted-behavior fast path that would bypass the default deny (policy-engine.ts:504, 582-585) never fires: every production IdentitySpec writer sets `trustBehaviors: []` (seed.ts, system-identity.ts:43 and 67, provision-dashboard-user.ts:68). No `approvalMode: "system_auto_approved"` short-circuit applies (governance-gate.ts:100-108): that mode is set only on operator intents (apps/api/src/bootstrap/operator-intents.ts:121, allowedTriggers ["api"]). The deployment's seeded `trustLevelOverride: "autonomous"` and spend lever cannot flip a deny (creative-governance.ts:32-35; governance-gate.ts:176-177, the spend post-processor "NEVER touches a deny").

Layer 3, execution would fail closed anyway, but ugly: there IS an executor for creative on the conversation surface. The single SkillMode instance that serves chat-triggered work units maps `executorBySlug: new Map([["creative", composeExecutor]])` and `builderRegistry.register("creative", miraBuilder)` (apps/api/src/bootstrap/skill-mode.ts:728-749). If a decision ever came back execute, SkillMode would resolve miraSkill and call miraBuilder with the chat parameters as `request` (packages/core/src/platform/modes/skill-mode.ts:45-58, 70, 85); miraBuilder safeParses MiraComposeRequestSchema, which requires `composeSource` (packages/schemas/src/mira-compose.ts:26-39), so chat parameters throw ParameterResolutionError "mira-compose-request-invalid" (packages/core/src/skill-runtime/builders/mira.ts:211-217), caught as a failed EXECUTION_ERROR result (skill-mode.ts:120-124). Mira has no conversational skill: skills/mira/SKILL.md is a compose brain (tools: [], required params BUSINESS_NAME, PERFORMANCE_CONTEXT, PIPELINE_STATE, TRIGGER_CONTEXT that only the compose builder supplies). In every layer's outcome the visitor sees the generic technical-failure copy "I'm having trouble right now. Let me connect you with the team." (channel-gateway.ts:136), and a failed/denied WorkTrace accumulates per message.

Answer to sub-question 2: governance default-denies creative.respond for every org including org_dev (no allow seeded, none global), the identity floor denies all chat-session actors before that, and even an allowed unit would error in the builder, not converse.

## 4. What pins this: nothing pins the deny; one test pins the opposite

Repo-wide, exactly one test references creative.respond: packages/core/src/platform/skill-intent-registrar.respond.test.ts:52-65 asserts `creative.respond` IS registered alongside compose (and :39-42 asserts the chat trigger is allowed on the slug-generic respond intent). That pins the dangerous direction as intended behavior. The creative-governance test asserts the publish regex rejects job siblings but never tests "creative.respond" against the ALLOW policy (packages/db/src/seed/**tests**/creative-governance.test.ts:8-25). No test asserts the gate denies creative.respond, no test asserts the marketplace connection routes restrict slugs, no test asserts channel resolution cannot return creative. The invariant currently survives on the CONTENT of three seeded regex strings plus an absent identity row, and the F-16 remediation workstream will be editing exactly those two tables (per-org policies and identity specs) for real pilot orgs. A single broad allow policy with a natural-looking pattern like `creative.*` (unanchored `matches`, so it DOES match "creative.respond") would silently open the path, and because the registration derives read-class/approvalPolicy "none", the allowed unit would go straight to execute with no approval parking.

Answer to sub-question 3 and exact pinning tests to add (see G5-F4 below for the full spec): (a) extend creative-governance.test.ts to assert every creative policy builder's pattern rejects "creative.respond"; (b) add a gate-level pin (packages/core/src/platform/governance/**tests**/creative-respond-default-deny.test.ts) that evaluates a chat-triggered creative.respond WorkUnit against the real seeded policy inputs and an empty-trustBehaviors identity and asserts outcome "deny", including with the autonomous+spendAutonomy deployment posture set.

## Verdict

The channel-as-ingress invariant for Mira is NOT intact by construction: the intent is registered with chat allowed by design, two live org-authed routes (marketplace telegram and web widget connections) can bind a channel to the creative deployment with no slug restriction anywhere in the binding, registry, resolver, or ingress chain, and the conversation-serving SkillMode carries a full executor+skill+builder for slug "creative". It holds today by policy default-deny reinforced by two accidents of current seed data: no seeded or global policy matches creative.respond (the three creative policies are job/compose/publish-shaped), and only the org_dev seed produces a chat-resolvable skillSlug="creative" deployment (self-serve deploys leave skillSlug null). The identity-spec fail-closed throw denies all chat-session actors today, but that is the F-16 floor the pilot-org remediation will deliberately relax, and the policy leg is one unanchored `creative.*` allow row away from flipping. Nothing pins the deny; the only existing test pins the opposite (registration plus chat trigger). Settlement: classify as policy-default-deny plus seed-data accident, then make it constructional with a conversational-surface allowlist at the registrar or binding routes, and pin it with the two tests specified in G5-F4.

## Findings

### G5-F1 (verified). P1. Channel bindings to the creative deployment are constructible through two live routes; no layer of the chain knows which slugs are conversational

Claim: POST /api/marketplace/deployments/:id/connections/telegram and /widget check only existence + org ownership (marketplace.ts:529-538, 577-586) before creating a DeploymentConnection on ANY deployment, including the seeded skillSlug="creative" one; the chat RuntimeRegistry auto-serves every active DeploymentConnection as a live webhook (runtime-registry.ts:86-111); the resolver returns whatever skillSlug the deployment row carries (prisma-deployment-resolver.ts:130); the gateway then submits creative.respond trigger chat (channel-gateway.ts:318, 328), which is a registered, chat-allowed intent (skill-intent-registrar.ts:41, 56-62). The only constructional accident limiting blast radius is that self-serve deploys never set skillSlug (marketplace.ts:229-239) so only seeded deployments resolve (org_dev today, every pilot org once D4-F1 provisioning lands).

Recommendation: make the invariant constructional. Preferred: a conversational-surface declaration in SKILL.md frontmatter (alex yes, mira no) consumed by registerSkillIntents so `creative.respond` is never registered and inbound resolution to creative returns intent_not_found (note the trade-off: intent_not_found returns before WorkUnit creation, platform-ingress.ts:182-192, so no trace is persisted; if audit-trail-on-attempt is wanted, instead drop "chat" from the auto-registered respond intent's allowedTriggers for non-conversational skills so the trigger_not_allowed deny is explicit). Secondary hardening: the two marketplace connection routes reject deployments whose skillSlug is not in a conversational allowlist. Tag: new. Effort: S.

### G5-F2 (verified). P1. The deny that holds today is seed-data-shaped, and the workstream that fixes F-16 is the exact place it can silently flip

Claim: creative.respond default-denies only because (a) no seeded or global policy matches it (creative-governance.ts:47, 77, 121 patterns verified non-matching; prisma-policy-store.ts:133-177 loads global+org and no production writer seeds global policies) and (b) chat-session actors have no IdentitySpec so the gate throws and ingress converts to a fail-closed GOVERNANCE_ERROR deny (app.ts:535-540; platform-ingress.ts:252-279), which is pilot-spine F-16's known floor slated for per-org remediation. The policy `matches` operator is an unanchored regex test (rule-evaluator.ts:64-65, 115-119), so a remediation-era allow policy written as `creative.*` would match creative.respond; with the registration derived read-class and approvalPolicy "none" (skill-intent-registrar.ts:6-11, 29, from SKILL.md `tools: []`), the allowed unit executes with no approval parking. Extends D4-F1 (the same seed installs the policies) and D4-F7 (derived registration fields become load-bearing the moment a policy matches).

Recommendation: when writing the F-16 per-org policy seed, keep every allow pattern anchored (the creative-governance.ts:70-75 comment already documents why) and add the non-match pin of G5-F4(a) in the same PR; land G5-F1's structural guard before pilot-org creative provisioning so the invariant stops depending on policy-table contents at all. Tag: extends. Effort: S.

### G5-F3 (verified). P2. If governance ever allowed it, Mira's conversation surface fails closed but ugly: a builder zod-reject per message, failure copy to the visitor, and trace noise

Claim: there IS an executorBySlug for creative on the conversation-serving SkillMode (apps/api/src/bootstrap/skill-mode.ts:748) and a registered "creative" builder; a chat-shaped parameters object fails MiraComposeRequestSchema (requires composeSource; packages/schemas/src/mira-compose.ts:26-39) so miraBuilder throws ParameterResolutionError (builders/mira.ts:211-217), caught as EXECUTION_ERROR (core skill-mode.ts:120-124); the visitor gets the generic trouble copy (channel-gateway.ts:136). skills/mira/SKILL.md is compose-only (tools: [], compose-shaped required params), so no conversational capability exists behind the intent. This is an error path, not a designed deny: each inbound message would burn a WorkTrace failed row and on-call noise, never spend.

Recommendation: none beyond G5-F1; this layer is acceptable as a last line precisely because it spends nothing. Record it in the runbook as the expected signature ("mira-compose-request-invalid" storms = a channel is bound to creative) until the structural guard lands. Tag: extends. Effort: S.

### G5-F4 (verified). P1. Nothing pins the deny; the only creative.respond test pins the opposite direction. Exact pinning tests specified

Claim: repo-wide, the only test naming creative.respond asserts it IS registered and that the respond intent allows the chat trigger (skill-intent-registrar.respond.test.ts:39-42, 52-65); creative-governance.test.ts:8-25 tests the publish regex against job siblings only, never "creative.respond"; no test covers the marketplace connection routes' slug behavior or asserts a gate deny for creative.respond.

Recommendation, the two pins to add: (a) packages/db/src/seed/**tests**/creative-governance.test.ts, new case "no creative policy matches creative.respond": for each of buildCreativeAllowPolicyInput, buildCreativePublishApprovalPolicyInput, buildCreativeBriefComposeAllowPolicyInput (orgId "org_x"), extract each rule condition with operator "matches" and assert `new RegExp(cond.value).test("creative.respond")` is false (mirrors the existing lines 22-25 style and survives pattern edits). (b) packages/core/src/platform/governance/**tests**/creative-respond-default-deny.test.ts: construct GovernanceGate with deps where loadPolicies returns the three creative policy inputs mapped to Policy shape, loadIdentitySpec returns a spec with trustBehaviors [] and standard riskTolerance, loadCartridge null, getGovernanceProfile null, evaluate(realEngineEvaluate) imported from the engine; evaluate a WorkUnit `{ intent: "creative.respond", trigger: "chat", actor: { id: "session-123", type: "user" }, organizationId: "org_x", parameters: { message: "hi", conversation: { messages: [], sessionId: "s" } }, deployment: { skillSlug: "creative", trustLevelOverride: "autonomous", spendAutonomyEnabled: true, deploymentId: "dep_1", trustScore: 0, trustLevel: "supervised" } }` with the live `creative.respond` registration produced by registerSkillIntents over a mira-shaped SkillDefinition (slug "creative", intent "creative.brief.compose", tools []); assert `decision.outcome === "deny"`. This pins both that the seeded autonomous/spend posture cannot flip the deny and that the policy set's non-match is what holds. If G5-F1's structural guard is chosen instead, replace (b) with a registrar test asserting `registry.lookup("creative.respond")` is undefined (or its allowedTriggers exclude "chat") for a skill declaring no conversational surface. Tag: new. Effort: S.
