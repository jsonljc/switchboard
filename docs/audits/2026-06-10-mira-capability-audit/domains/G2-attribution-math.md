# G2: Creative-Attribution Measurement Math

Gap: the creative-attribution worker (`apps/api/src/services/cron/creative-attribution.ts`) was never assigned a domain in the 9-domain audit, yet it is load-bearing for every measured leg of the loop: the `pastPerformance` rows Mira's prompt and the dashboard read, the future `revenue_proven` promotion, and Riley's creative visibility. This report audits the worker end to end against live code at the audit worktree HEAD (branch `docs/mira-capability-audit`, parent commit 84083f0c).

Verdict question: when `CREATIVE_ATTRIBUTION_ENABLED` flips on for a pilot org with a published campaign, does this worker produce numbers an operator and Riley can trust?

## Verdict

Mostly yes for the spend leg, structurally no for the booked leg, and Riley sees nothing either way. The worker itself is one of the better-built crons in the repo: kill-switched per invocation (`creative-attribution.ts:225`), idempotent overwrite-in-place with benign `StaleVersionError` skips (`:296-308`, extends D5-F6), a tested NO-DOWNGRADE rule that refuses to erase earned history (`:167-170`), cents normalized to major units exactly once (`source-comparator.ts:61-64`), and null-not-zero trueRoas semantics (`:203`). The 1:1 campaign:creative join it depends on holds by construction: each publish creates its own draft campaign behind a persisted checkpoint that re-publishes reuse (`creative-publish-function.ts:118-129`), and nothing ever clears `metaCampaignId`. But the booked-leg precondition is never satisfied for the north-star path: `ConversionRecord.sourceCampaignId` is only populated by the instant-form adapter and manual operator intents, never by the CTWA path, because the `CtwaAdapter.resolveCampaignId` dependency designed for exactly this is unwired at its only construction site (`apps/chat/src/main.ts:168-181`). A Mira-published CTWA campaign will therefore show correct spend, `booked: {0, 0}`, and `trueRoas: null` forever. That is honest absence, not a lie, and the UI labels it as such ("no booked revenue attributed yet", `creative-detail-page.tsx:171`), but it means the loop's headline number (receipted bookings per creative) cannot appear for the exact wedge the product is built around. Secondary hazards: Meta numerics enter through `parseFloat` with no finite guard and no runtime schema parse (the #939 NaN-blind pattern, with a concrete trigger in the `conversions` field), the write side stores blind while every reader is strict (one bad numeric silently makes a row invisible to all readers and disables its own no-downgrade protection), and trueRoas mixes org-currency cents with ad-account-currency spend with no currency captured or asserted. One M fix (campaign resolution at CTWA intake) plus three S-grade hardenings make this worker trustworthy end to end; the `revenue_proven` promotion that would give Riley visibility remains designed-but-unbuilt.

## What holds up (verified, with counter-evidence hunted)

1:1 join. The comment at `creative-attribution.ts:158-160` declares "1:1 campaign:creative is load-bearing for this join". Verified by construction: `executeCreativePublish` creates one draft campaign per job inside a checkpointed step ("create-draft-campaign", `creative-publish-function.ts:118-129`) that short-circuits when `current?.metaCampaignId` exists, so retries and re-dispatches reuse the same campaign and a second publish of the same job cannot mint a second campaign. Grep across the repo finds exactly one writer of `metaCampaignId` (`updatePublishFields` called from the publish chain); no path clears it, no path assigns an existing campaign id to a second job. Two jobs sharing a campaign is unrepresentable today. The acknowledged break condition (a future slice consolidating creatives into shared campaigns) is documented in the code itself with the required remedy (ad-level insights on `join.metaAdId`).

Window vs pre-publish spend. The window starts at the earliest published job's `createdAt` clamped to 90 days (`creative-attribution.ts:247-253`). Since the campaign object only exists after publish, no pre-publish spend exists for Meta to report and no conversion can carry the campaign id before it exists, so the early window start cannot over-attribute either leg. The 90-day clamp clips both legs consistently (same `from` feeds `getCampaignInsights` and `queryBookedStatsByCampaign`, `:266-278`). Distortions that remain are label-level, see G2-F5.

Idempotency and re-run. The worker has no `step.run` checkpoints, so an Inngest retry re-runs the whole handler; every write is a full-row replace via org-scoped `updateMany` (`prisma-creative-job-store.ts:210-220`) with `count===0` thrown as `StaleVersionError` and absorbed as a `vanishedSkips` counter (`creative-attribution.ts:303-306`). Re-running the same day re-stamps `asOf` and `updatedAt` and changes nothing else (extends the verified D5-F6 behavior). Dispatch fan-out memoizes one `emit-${orgId}` step per org (`:57-61`); duplicate events from at-least-once delivery land on an idempotent worker. The decrypted Meta token is resolved fresh per invocation and never enters Inngest step state (`:236-242`), and the per-instance 60s rate limiter gotcha is respected by construction: one fresh client, one insights call.

Units inside the helper. `trueRoasFromCents` divides `normalizeConversionValue(valueCents)` by major-unit spend, normalizing cents exactly once (`source-comparator.ts:55-64`, the documented #819 trap), and returns null rather than a fabricated 0 when value is unknown or spend non-positive. `computePastPerformance` passes `null` (not 0) when `count === 0` (`creative-attribution.ts:203`), so "no attributed bookings" renders as n/a, never as 0x. The schema documents the convention on both sides (`creative-job.ts:283` "major currency units as Meta reports", `:295` "CENTS, never pre-normalized").

Readers parse, never cast. All four consumers of `pastPerformance` run `CreativePastPerformanceSchema.safeParse` and project nothing on failure: the read-model projection feeding the dashboard and agent-home desk (`performance-projection.ts:11-12`), Mira's PERFORMANCE_CONTEXT via the same read model (`builders/mira.ts:124-149`, wired at `:267`), the brief-enrichment producer (`creative-performance-history.ts:23-27`), and the worker's own no-downgrade check (`creative-attribution.ts:168`). The brief producer additionally self-check parses its own output before persisting (`creative-performance-history.ts:56-62`). Test coverage in `apps/api/src/__tests__/creative-attribution.test.ts` pins the kill switch, the explicit field list, the no-downgrade rule end to end, the 90-day clamp, the StaleVersionError skip, and that produced rows validate against the schema (with well-formed inputs).

## Findings

### G2-F1 (P1, extends, M): the booked leg is structurally dark for CTWA: sourceCampaignId has no producer on the path the product is built around

The worker's booked leg joins `ConversionRecord.sourceCampaignId` to `job.metaCampaignId` (`creative-attribution.ts:273-278, :288`), and the store aggregate requires `sourceCampaignId` non-null with `type: "booked"`, `origin: "live"`, `value > 0` (`prisma-conversion-record-store.ts:270-282`). Tracing every writer of `sourceCampaignId` on booked records: the booked outbox event copies it from the contact's `AttributionChain` (`calendar-book.ts:365-377` via `buildBookedConversionPayload`, `booked-conversion-payload.ts:34`), which is populated at lead intake (`lead-intake-handler.ts:74`). On the CTWA path, the WhatsApp referral carries only `source_id` (an AD id) and `ctwa_clid` (`whatsapp-parsers.ts:27-30`); the campaign id must be resolved from the ad id, and `CtwaAdapter` was designed for exactly that with an optional dep: `resolveCampaignId?: (adId: string) => Promise<string | null>` gated on `adSourceType === "ad"` (`ctwa-adapter.ts:39, :91-104`). That dep is unwired at the adapter's only construction site: `new CtwaAdapter({ ingress: {...}, now: () => new Date() })` (`apps/chat/src/main.ts:168-181`), and grep finds no other implementation or injection of `resolveCampaignId` anywhere in the repo. The resolver primitive already exists and is cached: `MetaAdsClient.getAdCampaignId(adId)` (`meta-ads-client.ts:384-399`). Counter-evidence checked: the instant-form path DOES populate it (`instant-form-adapter.ts:58` `...(lead.campaignId ? { sourceCampaignId: lead.campaignId } : {})`), and operator manual intents accept it as a param (`operator-intents/revenue.ts:49`, `record-verified-payment.ts:91`), so the column is live for other sources; only CTWA, the SG/MY WhatsApp wedge, never writes it. Net effect when the flag flips on: every Alex-booked WhatsApp conversation started by a Mira CTWA ad lands as a booked ConversionRecord with `sourceCampaignId: null` and `sourceAdId` set (the chain preserves the ad id end to end: `outbox-publisher.ts:39`, `prisma-conversion-record-store.ts:62`), the aggregate returns nothing, `booked` stays `{valueCents: 0, count: 0}`, and `trueRoas` stays null forever. The numbers are honest but the loop's prove-leg cannot close. Recommendation: wire an org-scoped `resolveCampaignId` into the CtwaAdapter construction (the chat app already loads `@switchboard/db` and can resolve the org's meta-ads connection; the dep signature needs `orgId` added since the adapter is an org-agnostic singleton), using `getAdCampaignId`. This also covers operator-created ads inside the Mira campaign, which a worker-side `sourceAdId == job.metaAdId` fallback would miss (the operator's finalize-in-Ads-Manager step plausibly creates a new ad set or ad for the WhatsApp conversion location, so the clicking ad is often not the draft ad). A worker-side ad-id fallback leg is still a cheap additive safety net for the draft-ad-runs-as-is case.

### G2-F2 (P2, new, S): Meta numerics enter through parseFloat with no finite guard and no runtime parse; the conversions field is a concrete NaN trigger

`mapCampaignInsight` maps every numeric as `parseFloat(String(raw.x ?? "0"))` / `parseInt(...)` (`meta-ads-client.ts:456-479`). The `?? "0"` default only covers absent/undefined; `null` becomes the string "null", empty string stays empty, and both parse to NaN. No caller runtime-parses the result: grep finds zero `CampaignInsightSchema.parse|safeParse` at the client boundary; the client imports the zod-inferred shape as a type only (`meta-ads-client.ts:3` `import type { CampaignInsightSchema as CampaignInsight }`), so the `z.number()` fields in `schemas/ad-optimizer.ts:93-114` are compile-time fiction at this seam. The concrete trigger: the worker requests `conversions` in its explicit field list (`creative-attribution.ts:33`), and Meta returns action-style fields as arrays of `{action_type, value}` objects, which this very mapper acknowledges for the sibling `actions` field one line below (`meta-ads-client.ts:473-477` "Meta returns `actions` as an array of { action_type, value }"); `parseFloat(String(arrayOfObjects))` is NaN. Downstream, NaN survives every gate in the worker: `insight?.spend ?? 0` passes NaN through (NaN is not nullish, `creative-attribution.ts:174`), and `trueRoasFromCents`'s `spend <= 0` guard is NaN-blind (`NaN <= 0` is false, `source-comparator.ts:62`), so a NaN spend with `count > 0` produces a NaN trueRoas, the exact #939 failure class. A NaN anywhere in the row then poisons it at persistence or read: NaN is not JSON-representable (serializes to null or rejects the write), and zod rejects both NaN and null for `z.number()`, so every reader's safeParse fails silently, see G2-F3. The same unguarded mapper feeds Riley's audit-runner (`audit-runner.ts:209-222` requests `conversions` too), so the blast radius exceeds this worker. Recommendation: `Number.isFinite`-guard the numeric mapping in `mapCampaignInsight` (coerce non-finite to 0 for counts, and surface non-finite spend as a logged skip rather than a silent 0), and handle `conversions` as the actions-style array it is. Tag new, effort S.

### G2-F3 (P2, extends, S): write side stores blind while every reader is strict; one bad numeric silently hides a row from all readers and disables its own downgrade protection

`computePastPerformance` returns an object literal that is never runtime-parsed before persistence, and `setPastPerformance` accepts `Record<string, unknown>` and writes it raw (`creative-attribution.ts:176-205, :297`; `prisma-creative-job-store.ts:210-220`). Every reader, by design, is parse-don't-cast and "project[s] nothing on failure" (`creative-job.ts:262-263`). The schema is stricter than the write path can guarantee: `booked.valueCents: z.number().int()` (`creative-job.ts:295`) over `_sum` of `ConversionRecord.value`, which is `Float` (`schema.prisma:2057`, cents-in-Float per D8-F7) with no `.int()` enforcement at any writer (the operator revenue route validates only `z.number()`, `routes/revenue.ts:27`); and `z.number()` rejects the NaN/null outcomes of G2-F2. So a single fractional-cent conversion value or one malformed Meta numeric produces a row that persists fine, then fails safeParse everywhere: invisible to Mira's prompt, the dashboard, the brief enrichment, AND the worker's own no-downgrade check (`creative-attribution.ts:167-170` requires a successful parse of the prior row to protect it), with no log anywhere. The asymmetry is already solved one file over: `buildPerformanceHistory` self-check parses its output so "an authoring bug here must throw, never persist junk" (`creative-performance-history.ts:56`). Recommendation: `CreativePastPerformanceSchema.parse` the row in `computePastPerformance` (or before `setPastPerformance`) so a bad write throws into the retry/dead-letter contract instead of silently poisoning, and `Math.round` the summed valueCents. Tag extends (the self-check-parse pattern exists in the sibling producer), effort S.

### G2-F4 (P2, new, S): trueRoas divides org-currency cents by ad-account-currency spend with no currency captured or asserted

The numerator is `ConversionRecord.value` cents, written with `currency: input.currency ?? "SGD"` defaults on the revenue path (`lifecycle-service.ts:179`) and `deps.defaultCurrency` on the booked outbox payload (`calendar-book.ts:376`); the denominator is Meta `spend` "as Meta reports" in the ad account's billing currency (`creative-job.ts:283`). Nothing in the row records either currency, and nothing asserts they match; the account currency is available cheaply (`AccountSummarySchema.currency`, `schemas/ad-optimizer.ts:139`) but the worker never reads it. Renderers then hardcode dollar signs: `dollars()` and `$${p.spend.toFixed(2)}` in Mira's PERFORMANCE_CONTEXT (`builders/mira.ts:116-118, :145`). For the SG pilot with an SGD ad account this is cosmetic; for an MY clinic (MYR bookings) on a USD- or SGD-billed ad account, trueRoas is silently wrong by the FX ratio and nothing flags it. Recommendation: stamp `currency: { booked, spend }` on the pastPerformance row, set `trueRoas: null` on mismatch (consistent with the row's "insufficient signal" semantics), and drop the hardcoded `$`. Effort S.

### G2-F5 (P2, new, S): the window starts at brief creation, shared org-wide, not at publish; per-row window labels overstate exposure and the two legs disagree at day boundaries

There is no `publishedAt` column on CreativeJob (verified: the Meta block carries only ids and `metaPublishStatus`, `schema.prisma:1366ff`), so the worker anchors the window on the earliest published job's `createdAt`, the brief-submission time, shared across every job in the org (`creative-attribution.ts:247-253`). A creative whose brief sat in review for two weeks, in an org whose oldest published job is 60 days old, gets a row claiming `window.days ≈ 60` against perhaps 5 days of actual delivery. Totals stay correct (no campaign exists pre-publish, so neither leg can pick up pre-publish data; verified above), but the per-row window label and `days` field are wrong as exposure measures, and the brief-enrichment producer forwards `window.from/to` into the next brief's prompt context (`creative-performance-history.ts:50`). Additionally the two legs use different boundary semantics over the "same" window: Meta gets whole `YYYY-MM-DD` days, until-inclusive, interpreted in the ad account timezone (`creative-attribution.ts:264-267`), while the booked aggregate uses exact UTC instants `gte from, lte to` (`prisma-conversion-record-store.ts:277`), so numerator and denominator can disagree by up to a day of data at each edge. Recommendation: stamp a publish timestamp at the terminal publish checkpoint (`create-ad` step) and record `max(jobPublishedAt, sharedFrom)` as each row's `window.from` (the shared insights query window can stay as-is; this is a labeling fix), and document the timezone skew. Effort S.

### G2-F6 (P2, new, S): page-1-only insights read silently degrades orgs with more published campaigns than one page

The worker's own comment states the design constraint: "the client reads page 1 only", mitigated by the server-side `campaign.id IN` filter that keeps published campaigns inside the first page on accounts that also run non-Mira campaigns (`creative-attribution.ts:260-269`). Verified in the client: `getCampaignInsights` returns `response.data` and never follows `paging.next` (`meta-ads-client.ts:135-137`), and no `limit` param is set, so Meta's default insights page size (25) applies. Once an org accumulates more than a page of published Mira campaigns, the overflow campaigns are absent from insights: fresh jobs get misclassified `no_delivery` (zeroed meta block, `:185, :191-198`) and previously measured jobs freeze behind the no-downgrade rule with an aging `asOf`. The `campaignIds` list also grows unboundedly into a single URL query param. Not a pilot-scale problem (a pilot org will have a handful of campaigns), but it is a silent cliff with no warning log. Recommendation: set `limit: campaignIds.length` on the insights call (or follow paging), and log a warning when `insights.length < campaignIds.length` with the missing ids so misclassification is observable. Effort S.

### G2-F7 (P2, planned, M): revenue_proven has no writer and Riley has no reader; the "Riley sees what creative earns" edge does not exist

The `revenue_proven` DeploymentMemory category is schema-defined as "attributed-performance promotion, Riley-owned writer ONLY... built when its trigger fires: the first measured creative with spend > 0" (`deployment-memory.ts:15-21`), and Mira's builder already parses and renders `revenue_proven:` keyed rows first in its memory context (`builders/mira.ts:43, :76-91`). Grep across `packages/ad-optimizer`, `apps/api`, and `packages/db` finds zero writers of the category, and zero readers of `pastPerformance`/`derivePerformance`/`revenue_proven` anywhere in `packages/ad-optimizer` (Riley's code). So the gap statement's "future revenue_proven promotion" is confirmed unbuilt on both ends: the attribution worker produces the trigger data, but nothing promotes it into DeploymentMemory and Riley's diagnosis loop cannot see which creatives earn. Mira meanwhile carries reader code for rows that cannot exist. This is designed work (slice-2 spec 3.7), not a regression; flagged because flipping `CREATIVE_ATTRIBUTION_ENABLED` on creates the trigger condition with no consumer, and an operator told "Riley reallocates toward what works" would be misled. Recommendation: build the promotion writer as specced (trigger: first measured creative with spend > 0), and give Riley's recommendation engine a read of measured creative rows. Tag planned, effort M.

### G2-F8 (P2, new, S): the daily sweep re-stamps updatedAt on every published job, and Mira's prompt sorts negatives by updatedAt

`setPastPerformance` is an `updateMany` on a model with `updatedAt DateTime @updatedAt` (`prisma-creative-job-store.ts:215-218`, `schema.prisma` CreativeJob block), so every published job's `updatedAt` advances daily once the flag is on (re-stamping verified in D5-F6). `listPublished` filters only on `metaCampaignId: { not: null }` with no status filter (`prisma-creative-job-store.ts:227-232`), so stopped-but-published jobs are swept too. Mira's PERFORMANCE_CONTEXT picks its two "recent negative examples" by sorting on `updatedAt` (`builders/mira.ts:153-156`), so a stopped published creative is re-stamped every morning and permanently pinned as a "recent" negative in Mira's prompt regardless of when it was actually stopped, drowning genuinely recent passes. Any other reader sorting jobs by `updatedAt` sees the same daily churn. Recommendation: sort negatives by `reviewDecidedAt`/stop time rather than `updatedAt`, or move `pastPerformance` writes off the row's `@updatedAt` semantics. Effort S.

## Sub-question scorecard

1. Window clamp vs publish date: no pre-publish spend can be included (campaign exists only post-publish) and the 90-day clamp clips both legs consistently; the window label overstates exposure because it anchors on brief `createdAt`, org-shared, with no `publishedAt` column (G2-F5).
2. metaCampaignId join: genuinely 1:1 today by checkpointed construction; re-publishes reuse the persisted campaign id; multiple jobs per campaign unrepresentable; the documented break condition (shared campaigns) is future work with the remedy named in-code.
3. Booked-leg precondition: fails for CTWA. `sourceCampaignId` writers are instant-form and manual operator intents only; the CTWA resolver dep exists but is unwired at the singleton construction site, so a Mira-published CTWA campaign would not satisfy it today (G2-F1, the domain's one P1).
4. Value units: cents normalized exactly once inside `trueRoasFromCents`; no mixing in the math; currency identity between numerator and denominator is assumed, never captured or asserted (G2-F4).
5. Number.isFinite guards: absent at the Meta boundary; `parseFloat(String(...))` with NaN-blind `<= 0` gates downstream, concrete trigger in the `conversions` field (G2-F2).
6. Write shape vs readers: all readers safeParse and project nothing on failure (good); the writer never self-check parses, and `valueCents: int()` over a Float sum plus NaN-on-read makes silent row poisoning possible (G2-F3).
7. Idempotency/re-run: clean. Overwrite-in-place, org-scoped updateMany with StaleVersionError absorbed, step-memoized dispatch, fresh credentials per invocation, no duplicates possible (extends D5-F6).
