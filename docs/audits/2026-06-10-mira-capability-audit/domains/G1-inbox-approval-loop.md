# G1: The Inbox Approval Loop for Parked Mira Approvals

Gap: three domains (D4, D6, D7) each deferred the dashboard inbox surface for parked governed-workflow approvals to one another, and the end-to-end human gate (parked publish and parked handoff, decided in the inbox, round-tripped to lifecycle dispatch) was never traced. This report traces it: API feed composition, card content, dashboard proxies, inbox components, the approve/reject round trip, and the expiry surface. Every claim cites live worktree code; counter-evidence (tests, callers, sibling patterns) was hunted before each assertion. Baseline: branch docs/mira-capability-audit at 84083f0c.

## Verdict

The spine is real and pinned: a parked `creative.job.publish` or `adoptimizer.recommendation.handoff` lifecycle does reach the inbox as a `workflow_approval` card with correct agent attribution, a bespoke human summary, a binding-hash-carrying detail sheet, and a respond path whose core is exactly right (binding-checked freeze, payload-to-trace before dispatch, dispatch-or-recovery with a max-urgency Retry card, reject ends the unit, server-derived responder, self-approval guard, audit ledger on both legs). Four seams blunt it. First, the approve/reject error contract is dead through the real chain: the api-client throws message-only Errors and the approvals proxy re-labels every failure 500 without the structured `code` the hook branches on, so already-responded and expired settle as alarming error toasts instead of the designed silent clear, and stale-binding auto-refresh never fires; the hook's own tests stub fetch with `code` fields the proxy cannot produce, masking the break. Second, the publish card is decision-blind: a raw job cuid, two static reassurance lines, no creative title, no video link, no destination ad account, and no deep link anywhere (threadHref is hard-null for all workflow approvals and the route-template seam only knows contact URLs), while the detail sheet adds a permanently dead "preview not yet wired" panel. Third, the 24 hour expiry is invisible exactly where it bites: the SLA countdown renders only for handoff-kind cards, the sheet never renders the deadline, and the bespoke cards override the default dataLines that contained the one "expires" line. Fourth, a latent landmine for the planned D4-F2 fix: dashboard actorId and approval respondedBy are the same principal, so the moment a Publish button ships, every operator-initiated publish parks into an approval that same operator cannot approve unless the env-global ALLOW_SELF_APPROVAL is set.

## End-to-end trace (what is wired, with citations)

Park production. Publish: `POST /creative-jobs/:id/publish` pre-flights, then submits intent `creative.job.publish` with `parameters: { jobId: id }`, actor `{ id: request.actorId, type: "user" }` (apps/api/src/routes/creative-pipeline.ts:292-299); the seeded mandatory policy always parks (route comment 268-271; D4 verified). Handoff: the weekly cron submits `adoptimizer.recommendation.handoff` with parameters carrying actionType, campaignId, rationale, evidence {clicks, conversions, days}, learningPhaseActive, brief {productDescription, targetAudience} (apps/api/src/services/workflows/recommendation-handoff-request.ts:51-62; D6 verified the park).

Feed composition. `GET /api/dashboard/decisions` (registered at that prefix, apps/api/src/bootstrap/routes.ts:127-130) calls `listParkedApprovals` (apps/api/src/routes/decisions.ts:20-55): `lifecycleService.listOperatorActionableLifecycles(orgId)` returns expiry-filtered pending plus recovery_required (packages/core/src/approval/lifecycle-service.ts:58-68), sorted by expiresAt BEFORE the 25-row cap so urgent approvals are never hidden (decisions.ts:16-34, warn-logged truncation), and any lifecycle whose trace or revision fails to load renders a degraded card rather than vanishing (decisions.ts:41-49, adaptDegradedParkedApproval). The feed soft-skips parked approvals when lifecycleService is absent (decisions.ts:21), which is test-pinned ("keeps the feed working when lifecycleService is absent") and internally consistent: with no lifecycle service no parks can exist.

Card composition. `adaptParkedApproval` (packages/core/src/decisions/adapters/parked-approval-adapter.ts:84-144) builds the Decision: kind `workflow_approval`, agentKey from `trace.deploymentContext?.skillSlug` (line 98) via the resolver mapping "creative" to mira and "ad-optimizer" to riley (packages/core/src/decisions/agent-key-resolver.ts:3-13); deploymentContext is persisted on every trace from the resolved work unit (packages/core/src/platform/work-trace-recorder.ts:100,158), so publish parks attribute to Mira and handoff parks to Riley. meta carries slaDeadlineAt = expiresAt, riskContract, and the current revision bindingHash (adapter 135-142). recovery_required renders "Approved, but it didn't run" at urgency 100 with primary Retry (adapter 101, 116, 127-131). Urgency floors by risk (high = 70) and ramps to 100 as expiry nears (packages/core/src/decisions/urgency.ts:36-53). The bespoke summarizer registry covers both Mira-loop intents plus pause (apps/api/src/services/workflows/parked-approval-cards.ts:123-127). Unknown intents fall through to a default card that fails closed (high risk, clientFacing true) and includes a parameter preview with sensitive-key redaction and a "Waiting since X, expires Y" line (adapter 47-76, 105-113). API counts fold workflow_approval into the approval count (decisions.ts:105-111). Integration tests pin bindingHash surfacing, riley attribution, sort-before-cap, the degraded card, and the Retry card (apps/api/src/**tests**/api-decisions-parked.test.ts:64-141); the publish copy is pinned (apps/api/src/services/workflows/**tests**/parked-approval-cards.test.ts:74-79).

Dashboard read path. The decisions proxy forwards the body verbatim (apps/dashboard/src/app/api/dashboard/decisions/route.ts:16-25; agent-scoped sibling validates the key upstream). `useDecisionFeed` polls every 60s with `enabled: !!keys` (apps/dashboard/src/hooks/use-decision-feed.ts:20-28). The classic disabled-query trap (isLoading false, data undefined, empty-state render) is mitigated here: the (auth) layout server-fetches the session and hands it to SessionProvider synchronously (apps/dashboard/src/app/(auth)/layout.tsx, apps/dashboard/src/providers/auth-provider.tsx), the session type carries organizationId (apps/dashboard/src/lib/session.ts:10, lib/auth.ts:243), and middleware redirects signed-out users to /login (apps/dashboard/src/middleware.ts:150). InboxScreen checks isError before isLoading before empty (regression-guarded order, inbox-screen.tsx:259-267), discriminates strictly on `kind`, and routes workflow_approval to a dedicated row component and detail sheet (inbox-screen.tsx:289-296, 324-326). The row routes every gesture (swipe-approve, skip, tap) to the detail sheet so a lifecycle id can never hit the recommendation responder (inbox-workflow-approval-item.tsx:29-38). The sheet renders humanSummary, dataLines, risk chips, and a confirm-plus-audit-note flow for requiresConfirmation contracts (approval-detail-sheet.tsx:116, 130-137, 184-208, 243-266, 279-322).

Respond round trip. The detail item refuses approve locally when bindingHash is missing (degraded card, inbox-screen.tsx:106-113, toast "Can't approve from here"); otherwise `useWorkflowApprovalAction` POSTs `/api/dashboard/approvals` with approvalId, action, bindingHash, note, never respondedBy (use-workflow-approval-action.ts:39-54). The proxy calls `client.respondToApproval` (apps/dashboard/src/app/api/dashboard/approvals/route.ts:20-27) which hits `POST /api/approvals/:id/respond`. The API derives respondedBy from the authenticated principal and 403s a mismatched body value (apps/api/src/routes/approvals.ts:196-219), falls back from the empty legacy ApprovalRequest store to the lifecycle-native path (223-235), requires bindingHash for approve (86-92), checks org access (78), and maps every failure to a structured code: not_found 404, already_responded 409, expired 409, conflict 409, admission_failed 409, stale_binding and self_approval 400 (117-154). Core `respondToParkedLifecycle` marks expired-at-respond lifecycles expired and throws (respond-to-parked-lifecycle.ts:118-121); reject ends the unit and writes the ledger (127-145); approve asserts not-self-approval (252-259), binding-checks and freezes via approveLifecycle, writes the frozen payload to the trace BEFORE dispatch, then `runDispatch`, with throw or success:false transitioning to recovery_required so approved work never vanishes (module contract 10-28, 147-205); retry re-validates the binding and keeps dispatch admission strict (207-246). On a 200 the detail item branches the toast on `executionResult.success` ("Approved" / "Approved, but it didn't run", inbox-screen.tsx:123-135), and the dispatch-failed card returns as Retry on the next refetch. Dashboard tests pin bindingHash carry-through, reject, Retry-as-primary, and the degraded refusal (inbox-screen.test.tsx describe 11). The deprecated legacy `GET /api/approvals/pending` (empty for lifecycle parks, approvals.ts:289-335) has no remaining dashboard page consumer (only the POST is consumed), so there is no contradictory second approvals surface.

Answers to the gap questions. (1) Yes: both parked intents appear for any org with lifecycles, org-scoped via API-key metadata, with no entitlement gate hiding them (the inbox, unlike /mira, is ungated, which is what makes the D6-F6 Mira-disabled approve-then-nothing leg reachable). (2) No for an informed decision: confirmed D6-F6 for the handoff card and extended to publish (G1-F2). (3) The core round trip is correct and pinned; the error half of the contract is dead at the proxy seam (G1-F1) and the expiry is invisible (G1-F3). (4) The enabled:false trap is mitigated by server session hydration; error fidelity collapses at the approvals POST proxy (G1-F1); kind classification is sound (explicit three-member union, apps/dashboard/src/lib/decisions/types.ts:5; screen discriminates on kind only).

## Findings

### G1-F1 (P1, extends, S): The approve/reject error contract is dead through the real proxy chain, and the hook's tests mask it

Claim: `useWorkflowApprovalAction` is written against the API's structured codes ("already_responded | expired -> silent success; stale_binding -> refresh", use-workflow-approval-action.ts:18-26, 56-63), but the chain cannot deliver them. The api-client base helper throws a message-only Error on any non-ok status, dropping both the status and the `code` field: `if (!res.ok) { const body = await res.json().catch(() => ({})); throw new Error(body.error || \`API error: ${res.status}\`); }`(apps/dashboard/src/lib/api-client/core.ts:17-19). The proxy catch then returns`proxyError(err instanceof Error ? { error: err.message } : {}, ... 500)`(apps/dashboard/src/app/api/dashboard/approvals/route.ts:28-33), and proxyError emits only`{ error, statusCode }`(apps/dashboard/src/lib/proxy-error.ts), so the browser sees status 500 with no`code`for every API 409/400. The hook's`body.code === "already_responded"`, `"expired"`, and `"stale_binding"`branches are unreachable; everything falls to`throw new Error(body.error ...)`. The unit tests stub global fetch with bodies like `{ error: "already responded", code: "already_responded" }` (use-workflow-approval-action.test.tsx:66-93), green-lighting a contract the proxy never honors; there is no proxy-level test for the approvals POST (no **tests** dir under app/api/dashboard/approvals).

Impact: in the two routine contention cases this gate will actually see (a second operator or window settled the card; the card expired, which D6-F7 shows is structural with a weekly producer and a 24h park) the operator gets "Couldn't approve" with a raw lifecycle-id message instead of a silent clear, and because invalidation runs only on success or the (unreachable) staleBinding branch, the settled card lingers until the 60s poll. Stale-binding's designed "This approval changed. Refreshing." recovery never fires. No false success is possible (approve only succeeds on a real 200), so this is fidelity, not safety.

Recommendation: follow the repo's own raw-passthrough precedent (resumeRaw, replyToEscalationRaw, governance.ts:241-253 and 291-306, both created for exactly this collapse) with a respondToApprovalRaw returning `{ status, body }`, have the proxy forward both verbatim, and pin with a proxy test that feeds the API's real 409 body through. Tag: extends. Effort: S.

### G1-F2 (P1, extends, M): The publish approval card is decision-blind: a raw cuid, no creative preview or link, and a dead placeholder panel in the sheet

Claim: D6-F6 (handoff card hardcodes "refresh creative" regardless of actionType) confirmed as written at parked-approval-cards.ts:29-66; this finding extends the blindness to the publish card. publishCard reads exactly one parameter: `const jobId = str(parameters, "jobId") ?? "a kept creative"` and renders "Mira wants to publish creative ${jobId} to Meta as a paused draft package" plus two static reassurance lines (parked-approval-cards.ts:68-87); the submit carries only `parameters: { jobId: id }` (creative-pipeline.ts:295), so jobId is a database cuid. The approver sees no creative title, no format, no video link or thumbnail, no destination ad account, and no path to the existing creative detail page: adaptParkedApproval takes no routeTemplates (signature, parked-approval-adapter.ts:84-89), sets `threadHref: null` unconditionally (line 133), and dashboardRouteTemplates defines contact URLs only (apps/api/src/lib/route-templates.ts:14-18), so no workflow_approval card can deep-link anywhere. The detail sheet then renders a hardcoded "What this changes" panel labeled "preview not yet wired" with empty Before/After/Confidence/Money-at-risk cells and the caption "wiring it up next week" (approval-detail-sheet.tsx:210-241), shipped to every approval detail.

Impact: this is the human gate on the only external mutating act in Mira's loop, and the human is asked to approve a cuid. The risk contract is honest (high, external, non-financial, requiresConfirmation, parked-approval-cards.ts:77-86) and the paused-no-spend framing is correct, but an operator who wants to verify WHAT they are publishing must leave the inbox, open /mira, and hunt for the job by hand. The handoff card has the same shape (campaign id string, no link to the /riley queue card it mirrors).

Recommendation: three composable steps. (a) Enrich submit parameters at the publish route with display fields the pre-flight already loaded (assertPublishable reads the job): title, format, adAccountRef; the summarizer renders them defensively like the handoff card renders brief fields. (b) Add a creativeDetail template to dashboardRouteTemplates and an optional href on ParkedApprovalSummary that adaptParkedApproval maps to threadHref, giving the card a "View creative" door (the sheet already renders threadHref when present). (c) Either wire or remove the "next week" placeholder panel; a permanently dead promise erodes exactly the trust the sheet exists to build. Must land with D4-F2's doctrine-test amendment if the publish CTA ships in the same arc. Tag: extends (D6-F6, D4-F2). Effort: M.

### G1-F3 (P2, extends, S): The 24h expiry is invisible end to end on workflow_approval cards

Claim: extends D6-F7 (weekly producer vs 24h park drops handoffs) one layer down: the operator is never shown the clock. The inbox card renders the SLA countdown only for handoffs: `const due = isHandoff ? dueIn(decision.meta.slaDeadlineAt, nowMs) : null` (inbox-decision-card.tsx:76), and the non-handoff meta slot shows `relativeTime(decision.createdAt)` instead (lines 143-151), so a workflow_approval card displays how long ago it was proposed, not how soon it dies. The detail sheet renders proposed-ago and undoableUntil but never meta.slaDeadlineAt (approval-detail-sheet.tsx:168-176; no other consumer, confirmed by grep over components/inbox and lib/decisions). The one place expiry existed in copy, the default card's "Waiting since X, expires Y" line (parked-approval-adapter.ts:108-110), is exactly what the bespoke publish/handoff/pause dataLines replace (adapter 119: `summary?.dataLines ?? defaultLines`; neither bespoke card includes an expiry line, parked-approval-cards.ts:38-48, 72-75). The data is already on the wire (decisions.ts meta.slaDeadlineAt = lifecycle.expiresAt, serialized at 115-124) and a dueIn helper with overdue states already exists (lib/decisions/time.ts:27-28). Expiry itself is sweep-handled and audit-logged (lifecycle-expiry.ts:15-23, action.expired) but has no operator surface.

Impact: the urgency score quietly ramps the card up the feed as expiry nears (urgency.ts:47-53) but nothing tells the human why, and an expired card simply vanishes from the feed (listPendingLifecycles is expiry-filtered), so a clinic operator who saw the card yesterday finds it gone with no trace and, per D6-F7, meets it again as a Groundhog-Day re-park next week.

Recommendation: render the existing dueIn chip for kind workflow_approval as well (one-line change to the isHandoff gate plus CSS audit), add the deadline row to the detail sheet header meta, and have bespoke summarizer dataLines append the default expiry line instead of replacing it (compose defaultLines tail in the adapter). Tag: extends (D6-F7). Effort: S.

### G1-F4 (P2, new, M): Self-approval blocks the planned operator publish-then-approve flow, with only an env-global escape

Claim: the respond path asserts `trace.actor.id !== respondedBy` unless selfApprovalAllowed (respond-to-parked-lifecycle.ts:252-259), and both sides of that comparison resolve to the same dashboard principal: mutating routes set `request.actorId = request.principalIdFromAuth` (apps/api/src/decorators/org.ts:59, 86, 133), the publish submit stamps that actorId as the trace actor (creative-pipeline.ts:295), and the approvals route sets respondedBy to the same principalIdFromAuth (approvals.ts:200-210). The escape hatch is process-wide: `const selfApprovalAllowed = !!process.env["ALLOW_SELF_APPROVAL"]` (approvals.ts:163), one posture for every org on the deployment. Today publish is curl-only (D4-F2), so the trap fires only when the curl key shares the dashboard's principal; the moment D4-F2's recommended Publish button ships through the dashboard proxy, every operator-initiated publish parks into an approval that the same operator cannot approve, surfacing (through the G1-F1 seam) as a 500-wrapped "Self-approval is not permitted" toast. Handoff and self-brief parks are unaffected (cron/system actor differs from the human responder). Counter-evidence checked: no per-org or per-intent override exists (grep ALLOW_SELF_APPROVAL and selfApprovalAllowed across apps/api and packages/core; only app.ts wiring and tests).

Impact: the pilot shape is a solo clinic operator, who is precisely the persona four-eyes blocks here. Flipping ALLOW_SELF_APPROVAL deployment-wide to unblock Mira publish would also relax the pause approval and every other parked intent for every tenant at once, a governance posture change disguised as a UI fix.

Recommendation: decide the posture BEFORE the publish button lands: either a per-org selfApprovalAllowed (governance settings read at respond time, defaulting to the env), or have the dashboard publish submit carry a distinct actor identity (e.g. the org's agent deployment principal) so the human gate stays meaningful, or accept and document mandatory-park-with-self-approval for single-operator orgs. Tag: new. Effort: M.

### G1-F5 (OPP, extends, S): Approve success collapses executionResult into "is on it"; the dispatched outcome is one field away

Claim: on a 200 the detail item reduces executionResult to a boolean ("{agent} is on it" vs "Approved, but it didn't run", inbox-screen.tsx:123-135), discarding outputs that already ride the response (approvals respond returns executionResult verbatim, approvals.ts:111-115; the publish executor returns Meta object refs in outputs, the handoff workflow returns child draft state including the skipped/child_no_draft leg D6-F6 flagged). The proxy forwards the 200 body unchanged, so this is purely a render choice.

Recommendation: render a one-line outcome from executionResult.outputs in the success toast (publish: "Paused draft created in Meta"; handoff: "Draft started" or the honest "Mira is not enabled, nothing was created" that closes D6-F6's invisible no-op leg). Tag: extends (D6-F6 recommendation). Effort: S.

## What is sound (counter-evidence honored)

- The feed never hides governed work: expiry-sort before cap, degraded cards for integrity failures, recovery_required surfaces at urgency 100, all integration-tested (api-decisions-parked.test.ts:90-141).
- Kind discrimination is locked: explicit three-member union, screen branches on kind only, workflow rows route every gesture to the detail sheet so the recommendation responder can never receive a lifecycle id (inbox-workflow-approval-item.tsx:14-38), and swipe policy is contract-driven with requiresConfirmation true on every bespoke card.
- The respond core honors the spec contract: binding-hash check, frozen payload written to the canonical trace before dispatch, strict dispatch admission, deterministic per-attempt idempotency, ledger writes on approve and reject, expired-at-respond marks the lifecycle (respond-to-parked-lifecycle.ts throughout).
- respondedBy is server-derived with a 403 on impersonation attempts and a no-principal deny (approvals.ts:196-219); the inbox approve refuses locally when the degraded card has no bindingHash, pinned by test.
- The RQ enabled:false trap was hunted and is mitigated for this surface: SessionProvider is server-hydrated with organizationId in the session shape, and middleware redirects signed-out users, so the disabled-query empty-state window is not reachable in the normal authed flow.
