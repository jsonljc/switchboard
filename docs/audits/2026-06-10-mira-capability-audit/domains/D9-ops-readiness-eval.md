Domain auditor d9, Mira capability audit, 2026-06-10. Branch docs/mira-capability-audit, worktree baseline origin/main at 84083f0c.

## Scope and method

Could a real org run Mira next week, and would we know if it broke? This domain covers: the full Mira flag and env inventory (reader sites, defaults, allowlist and infra registration); reconciliation of the pilot-enablement runbook (docs/runbooks/2026-05-29-mira-pilot-enablement.md) and the in-flight provisioning runbook (primary worktree docs/runbooks/provisioning.md, PR #957, read-only) against live code; cost controls on generation spend; provider rate limits, timeouts, and retry budgets; observability (which failures alert a human, which die silently); evaluation harness coverage; scale posture of the per-org loops; and arch-check status of Mira files. Method: every claim was read from live source in this worktree; greps were run for every flag reader and every dead-letter event consumer; prior-session hypotheses were re-verified and two are explicitly refuted below. Lifecycle representation gaps (the polished crash terminal) are D5 territory; D9 covers their operational consequences only.

## Findings

### D9-F1 (P1) Production alerting is structurally dead by default: no runbook or infra file ever sets OPERATOR_ALERT_WEBHOOK_URL (verified)

Claim: every "alert a human" path in the platform resolves to a silent no-op in a deployment built from render.yaml plus the provisioning runbook, because the alerter is webhook-or-noop and nothing in the production topology sets the webhook.

Evidence: apps/api/src/app.ts:447-454 (`const operatorAlerter: OperatorAlerter = process.env.OPERATOR_ALERT_WEBHOOK_URL ? new WebhookOperatorAlerter({...}) : new NoopOperatorAlerter();`); packages/core/src/observability/operator-alerter.ts:28-32 (`async alert(_payload) { // intentional no-op }`, not even a log line); render.yaml has no OPERATOR*ALERT_WEBHOOK_URL entry (grep returns only CREATIVE*\* lines 123-139); docs/runbooks/provisioning.md step 2 secret lists for api and chat omit it. The one creative function classed alert:true, creative-mode-dispatcher (inngest.ts:1078-1086, riskCategory high), alerts into the Noop. So do the alert:true crons (reconciliation.ts:180, lead-retry.ts:155, scheduled-follow-up-dispatch.ts:145, appointment-reminder-dispatch.ts:132). Counter-evidence checked: SENTRY_DSN_SERVER is declared in render.yaml and the runbook, but Sentry captures only uncaught Fastify route errors; every path that invokes the alerter has already caught its error (platform-ingress safeAlert, Inngest onFailure), so Sentry is blind to exactly the failures the alerter exists for, and the asyncFailure context's own fallback is another NoopOperatorAlerter (inngest.ts:486-490).

Impact on the north star: the loop's failure economics assume a human notices breakage. With a Noop alerter, a creative dispatch outage, a reconciliation failure, or a retry-exhausted lead send produce only AuditLedger rows and Inngest Cloud history that nobody is paged to read. "Would we know if it broke" is currently no, by configuration.

Recommendation: declare OPERATOR_ALERT_WEBHOOK_URL as a sync:false env var on switchboard-api in render.yaml and add it to the provisioning runbook step 2 (value: a Slack incoming-webhook from the vault). Optionally make NoopOperatorAlerter log at error level so a misconfigured prod is at least visible in Render logs. Tag: extends (the alerter and wiring exist; only the environment story is missing). Effort: S.

### D9-F2 (P1) A terminally failed polished render becomes a permanent zombie: no failure persistence, no sweep, no alert, and it eats the self-brief backlog cap forever (verified)

Claim: when creative-job-runner exhausts its 3 retries mid-stage, nothing marks the CreativeJob row; the job reads as in_progress (no stage outputs yet) or awaiting_review (prior stages persisted) indefinitely, counts toward the inFlight backlog cap of 5 forever, and after enough zombies Mira's weekly self-brief is silently throttled for the org with no operator signal.

Evidence: packages/creative-pipeline/src/creative-job-runner.ts:103-165: the stage loop has no try/catch and the injected JobStore interface (lines 23-37) has updateStage/stop/setDurableAsset but no fail method; contrast ugc-job-runner.ts:276-291, which catches, persists `deps.jobStore.failUgc(...)`, and emits creative-pipeline/ugc.failed. The onFailure handler is the generic makeOnFailureHandler (inngest.ts:1095-1101, eventDomain "creative.polished", alert:false) which writes only the audit row and an event; its AsyncFailureContext carries only auditLedger, operatorAlerter, and inngest.send (inngest.ts:486-490), so failure persistence is structurally unavailable to it; grep for creative.polished.failed consumers returns only the registration site (the event name is assembled at async-failure-handler.ts:124 and no function subscribes to it). Status derivation: status-mapper.ts:52-66 marks polished failed only on `productionErrorsWithoutVideo(job.stageOutputs)`; a crash in trends/hooks/scripts/storyboard leaves no error record, so `hasKeys(stageOutputs)` yields awaiting_review, and an operator Approve fires creative-pipeline/stage.approved into a run that no longer waits (creative-job-runner.ts:151-156), a dead button. Backlog coupling: build-read-model.ts:71-73 (`inFlight = awaiting_review + in_progress`) and mira-self-brief.ts:22,153 (`SELF_BRIEF_BACKLOG_CAP = 5; if (model.counts.inFlight >= ...) return { skipped: "backlog_cap" }`). Sharpening from verification: a healthy unattended run self-stops via the 24h APPROVAL_TIMEOUT (waitForEvent timeout leads to jobStore.stop, status "stopped", excluded from inFlight), so a polished row sitting in awaiting_review or in_progress beyond roughly a day is necessarily a crash zombie; the permanence is specific to retry-exhausted runs whose timeout died with them.

The unrepresentable crash terminal itself is D5-F1; this finding is the ops consequence chain: zombie, dead approve button, capped self-brief, alert:false, and no stalled-creative sweep anywhere (lifecycle-stalled-sweep.ts:5-6 covers ConversationLifecycleSnapshot only).

Impact on the north star: each silent generation failure permanently subtracts from Mira's autonomy budget (the cap exists precisely so unacted proposals throttle her); five lifetime failures and the self-initiation loop is dead for that org while every weekly run reports an innocuous-looking backlog_cap skip.

Recommendation: persist a terminal failure on the polished row (either a final catch in the runner mirroring failUgc, or extend the onFailure deps to write a failed marker), and exclude failed from inFlight. A cheap stopgap: an age guard in the read model (in_progress or awaiting_review older than N days flags problem and drops out of inFlight). Tag: new. Effort: M.

### D9-F3 (P1) An approved publish that fails on Meta is invisible to the operator: alert:false, dead-letter event with zero consumers, and metaPublishStatus surfaced nowhere (verified)

Claim: after the human approves a publish, the governed handler returns queued and hands off to the creative-publish Inngest function; if that function dead-letters (bad page id, expired token, persistent Meta 5xx), no surface the operator can see ever changes, so the org believes a creative was published when no paused draft package exists in Ads Manager.

Evidence: creative-publish-workflow.ts:55-59 (`name: "creative-pipeline/publish.requested" ... return { outcome: "queued", ... }`), so the WorkTrace outcome stays queued permanently; creative-publish-function.ts:53-60 (`CREATIVE_PUBLISH_FAILURE_PARAMS = { ... eventDomain: "creative.publish", riskCategory: "medium", alert: false }`); grep for "creative.publish.failed" across apps and packages returns no consumer; grep for metaPublishStatus across apps/dashboard/src and packages/core/src/creative-read-model returns nothing: the read model and the Mira feed never display publish state, and the only reader of PARKED_PAUSED is the workflow's own idempotent-replay check (creative-publish-workflow.ts:39).

Impact on the north star: publish is the moment Mira's work enters the revenue loop. A silent publish failure breaks the Riley-Mira-Alex chain at its highest-leverage seam: spend reallocation toward a creative that does not exist on Meta, with the human believing they completed their approval duty.

Recommendation: minimum, surface metaPublishStatus (parked_paused vs pending vs absent-after-approval) on the Mira job card; better, set alert:true severity warning on CREATIVE_PUBLISH_FAILURE_PARAMS (it is a human-approved action whose failure has a named owner). Tag: new (slice-1 spec consciously shipped the dead-letter; the operator surface was never specced). Effort: M.

### D9-F4 (P1) The Mira pilot-enablement runbook provisions one quarter of the live enablement chain; real-org provisioning is a spec-acknowledged open workstream (verified)

Claim: following docs/runbooks/2026-05-29-mira-pilot-enablement.md end-to-end yields an org whose /mira feed renders but where every governed Mira capability fails closed: no creative deployment (Alex handoff DEPLOYMENT_NOT_FOUND, self-brief skips no_creative_deployment), no governance policies (compose default-denies), and no billing entitlement (org_not_entitled skips).

Evidence: the runbook (lines 1-29) covers only `seedMiraPilotOrgs` (OrgAgentEnablement) plus a /mira render check. The live chain requires: seed-mira-creative-deployment.ts:22-39 ("This is the single live prerequisite for the Alex→Mira draft-only handoff ... without this row the resolver falls back to 'api-direct' and the draft handler fails closed (DEPLOYMENT_NOT_FOUND)"), which also installs the allow/approval policies (imports buildCreativeAllowPolicyInput, buildCreativeBriefComposeAllowPolicyInput, buildCreativePublishApprovalPolicyInput; creative-governance.ts:24: "run explicitly or creative.job.\* default-denies"); billing entitlement at platform-ingress.ts:195-206; and the env flags. The slice-4 spec names the gap: fact 30 ("Real-pilot-org provisioning of the creative deployment + policies runs for org_dev only; that workstream is separate", 2026-06-05-mira-slice4-brain-design.md:60) and section 7's flip checklist (lines 529-541) lists deployment seed, entitlement, and flags as prerequisites. prisma/seed.ts runs seedMiraCreativeDeployment for org_dev only. The provisioning runbook (PR #957) is hosting-scoped and contains no org-level Mira step.

Impact on the north star: this is the literal "could a real org run Mira next week" question. Today the answer requires an engineer who knows the four-part chain from memory; the only written procedure produces a half-enabled org whose failures are named skips in Inngest history that nobody watches (D9-F7).

Recommendation: write the per-org Mira flip runbook the spec already gestures at: seedMiraPilotOrgs + seedMiraCreativeDeployment (one script, idempotent, fails loud without the marketplace listing) + entitlement check + flag flips + a verification checklist (compose allow policy present, self-brief scan outcome, draft lands on /mira). Update the 2026-05-29 runbook in place or supersede it. Tag: planned (spec fact 30 and section 7). Effort: S.

### D9-F5 (P1) Generation provider keys are absent from production infra: a Blueprint-provisioned deploy runs Mira generation-dark (verified)

Claim: KLING_API_KEY, HEYGEN_API_KEY, ELEVENLABS_API_KEY, and OPENAI_API_KEY appear in neither render.yaml nor the provisioning runbook's secret lists, so a production deployment built per the runbook has no video generation (kling/heygen clients undefined), no voiceover, no image generation, and no whisper captions, all discovered only when a job runs.

""`). Downstream, a UGC spec with no live provider fails per-spec ("no_allowed_provider" / burned attempts, production.ts:282-298), an outcome visible only inside job output JSON. Verification strengthened the claim: the polished production stage independently constructs `new KlingClient({ apiKey: process.env.KLING_API_KEY ?? "" })` (run-stage.ts:144), so an unset key also makes polished video calls fail per-clip at runtime; both modes are generation-dark, not just UGC.

Impact on the north star: Mira's product is rendered video. The provisioning story ships her able to brief and storyboard but not produce, and the failure mode is per-job and quiet rather than at deploy time.

Recommendation: declare all four as sync:false env vars on switchboard-api in render.yaml and add them to provisioning.md step 2, marking which are pilot-required (Kling at minimum) versus optional-degrade (HeyGen, ElevenLabs, OpenAI). Tag: extends. Effort: S.

### D9-F6 (P2) Cost control is per-job only: no per-org or per-period generation ceiling, and Claude tokens are unmetered (verified)

Claim: the platform caps a single job (governance parks above a $15 estimate; production hard-stops at totalJobBudget, default $50, accrued per attempt) but nothing bounds jobs-per-org-per-day or aggregate provider spend, and LLM token spend has no accounting anywhere.

Evidence: creative-governance.ts:43 (`CREATIVE_SPEND_APPROVAL_THRESHOLD = 15`); ugc-job-runner.ts:190 (`totalJobBudget: (budgetConfig.totalJobBudget as number) ?? 50`); phases/production.ts:271-280 (per-attempt accrual, "the budget guard caps worst-case retry spend"); call-claude.ts:74-84 (fresh Anthropic client per call, default SDK retries, no token counting, no usage persistence); creative-job-submit-workflow.ts has no per-org rate or count cap (grep cap|limit|concurrent returns nothing). The only org-level throttle in the system is the self-brief backlog cap, which governs Mira-initiated drafts only.

Impact on the north star: pilot-scale exposure is bounded by human approval cadence (every job above $15 parks), so this is not a launch blocker; but the moment trust levels auto-approve under-threshold jobs, a misbehaving submitter (or Alex handoff loop) can run unbounded sub-$15 generations.

Recommendation: a per-org daily generation budget read at submit (sum of estimates over 24h against an org config default), enforced in the same governance seam as the spend threshold. Record Claude usage (the SDK returns usage on every response) into the job's stage outputs for visibility first. Tag: new. Effort: M.

### D9-F7 (P2) Skip outcomes and zero-output dispatches are observable only by reading Inngest Cloud run history or per-line logs; weekly cadence makes misconfiguration cost weeks (verified, corrected)

Claim: every guarded exit in the Mira loops is a named return value, which is excellent forensics but no telemetry: nothing aggregates or alerts when an org skips org_not_entitled five weeks running, when dispatch fans out to zero orgs, or when attribution short-circuits no_meta_credentials daily; the only traces are per-occurrence log lines and Inngest run output.

Evidence: mira-self-brief.ts:140-153 (skips: disabled, mira_not_enabled, no_creative_deployment, backlog_cap returned silently; only parse/submit failures hit deps.warn); executeMiraSelfBriefDispatch returns `{ dispatched: orgs.length }` with no zero-case handling (mira-self-brief.ts:39-50); creative-attribution.ts:225-241 logs disabled and no_published_jobs at info and no_meta_credentials at warn (correction: the original draft said all skips log at info; the warn line still alerts nobody and aggregates nothing); there is no metrics emission anywhere in the creative path (no prometheus/statsd/posthog grep hits), and the Inngest client has no Sentry middleware (grep sentry in bootstrap/inngest.ts: none); apps/api Sentry init exists (bootstrap/sentry.ts:14) but Inngest catches function errors for retry management, so they do not reach it. The provisioning runbook's monitoring step (provisioning.md:127) lists UptimeRobot on /api/health/deep, which proves liveness, not work.

Impact on the north star: Mira's loops are weekly (self-brief) and daily (attribution, taste). A flag typo, a missing policy, or an expired Meta token shows up as weeks of well-named skips that no one reads. This compounds D9-F4: the half-enabled org and the silent skip stream are the same incident.

Recommendation: a weekly ops summary step at the end of each dispatch (counts by skip reason across orgs, logged at warn and optionally posted via operatorAlerter severity warning when a degradation reason persists 2+ runs). Cheap and uses only existing plumbing. Tag: new. Effort: M.

### D9-F8 (P2) No evaluation harness exists for any Mira output: brief quality, abstain calibration, and creative descriptors are all unbenchmarked (verified)

Claim: evals/ contains four harnesses (alex-conversation, claim-classifier, governance-decision, riley-recommendation) and nothing for Mira; the compose path is parse-validated but never quality-evaluated, so a prompt regression in the brief composer, hook generator, or taste-context threading would ship undetected.

Evidence: ls evals/ = alex-conversation, claim-classifier, governance-decision, riley-recommendation; evals/governance-decision/README.md:7-11 describes the model-free, DB-free template ("every case is resolved through the real gate and compared to an expected decision ... runs in CI as a plain vitest test") that the Riley audit reused for riley-recommendation; the only Mira-side validation is parseMiraComposeOutput schema parsing (mira-self-brief.ts:223-235) and unit tests. CI runs all eval suites' unit tests via evals/vitest.config.ts (ci.yml:398-406), so a new harness inherits the gate for free.

Impact on the north star: Mira's differentiation is turning Riley diagnoses and Alex funnel reality into winning creative. Without even a deterministic floor eval, "winning" is unfalsifiable and every prompt edit is a blind deploy; the taste loop (Keep/Pass into prompts) can drift quality with no regression tripwire.

Recommendation: start model-free per the governance-decision template: a fixture matrix over the self-brief floor (flag/enablement/deployment/cap/signal permutations pinned to named outcomes), compose-output parse fixtures (good, fenced, abstain, malformed), and descriptor extraction. An LLM-judged brief-quality eval (the alex-conversation pattern) is the follow-on once fixtures exist from pilot traffic. Tag: new. Effort: M.

### D9-F10 (P2) Scale flip-gates are documented but live only in a spec: handoff enrichment serializes LLM composes inside the weekly-audit step, and dispatch fan-out spends one Inngest step per org (verified)

Claim: two known scale ceilings are recorded in the slice-4 spec but in no operational artifact: (1) MIRA_HANDOFF_BRIEF_ENRICHMENT_ENABLED puts one synchronous LLM compose per candidate inside the weekly audit's per-deployment Inngest step, which already serializes rate-limited Graph calls; (2) both dispatch crons emit per-org events as individual step.run calls, which is one checkpoint round-trip per org and approaches Inngest per-run step limits at high tenancy.

Evidence: 2026-06-05-mira-slice4-brain-design.md:541-547 ("FLIP GATE: ... At pilot tenancy this is minutes of headroom; before enabling at larger tenancy, move the compose onto its own step or event"); mira-self-brief.ts:44-49 and creative-attribution.ts:57-61 (`for (const orgId of orgs) { await step.run(\`emit-${orgId}\`, ...) }`). Fine at pilot scale (single-digit orgs); the workers themselves fan out correctly (one event per org, parallel worker invocations).
