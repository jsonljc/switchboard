## The learning loop end to end, dead legs marked

1. Keep/Pass gesture writes `reviewDecision`/`reviewDecidedAt` on the job (firewalled route, untouched per spec). LIVE.
2. Daily taste sweep (06:00 UTC) projects decided jobs into `taste:{decision}_{mode}_{segment}` buckets on the JOB's deploymentId, watermarked, capped at 500 with eviction parity. LIVE. (`creative-taste-sweep.ts:201-258`)
3. Taste buckets feed polished trend/hook prompts at the surfacing threshold (0.66 confidence, 3 sources, `packages/schemas/src/deployment-memory.ts:85`). LIVE.
4. UGC taste buckets (structure-keyed) have NO generation-time reader. DEAD LEG, planned follow-on (slice4 spec section 8; provider comment `creative-taste-context.ts:61-66`).
5. Taste plus revenue_proven feed compose `TASTE_CONTEXT`; the same predicate is the self-brief signal floor (`hasSurfacedCreativeMemorySignal`, `mira.ts:72-80`, shared with `mira-self-brief.ts:157-163`). Code LIVE; both automated initiators FLAG-DARK, so nothing composes in production today.
6. Publish-side measurement: daily attribution worker (06:30 UTC) writes `measured_performance` rows to `pastPerformance` with a no-downgrade rule. FLAG-DARK (`CREATIVE_ATTRIBUTION_ENABLED=false`) and additionally gated on an operator activating a paused ad and pointing it at a conversion-attributed destination (slice2 spec section 7).
7. `pastPerformance` projects into the read model (`performance-projection.ts`) and renders as compose `PERFORMANCE_CONTEXT` top performers with trueROAS and booked dollars (`mira.ts:131-149`). Code LIVE, data-empty until leg 6 flows.
8. `pastPerformance` aggregates into a `performance_history` row on each NEW submitted brief and renders a measured block in polished prompts (`creative-job-submit-workflow.ts:42-55`). Code LIVE, data-empty until leg 6 flows.
9. Riley promotion of attributed winners into `revenue_proven` memory: NOT BUILT. DEAD LEG with a defined build trigger (slice2 spec 3.7: first measured row with spend > 0). Until it ships, leg 5's "Measured winner" lines and the floor's revenue_proven arm can never fire (D3-F1).
10. Riley insights into compose: the handoff enrichment passes the recommendation (actionType, campaignId, rationale, clicks/conversions/days) as `TRIGGER_CONTEXT` (`mira.ts:170-182`). FLAG-DARK.
11. Alex funnel reality into compose: only the booked-conversion leg (ConversionRecord cents riding `pastPerformance.booked` into `PERFORMANCE_CONTEXT`). No conversation-level signal (questions, objections) reaches Mira. GAP, opportunity (D3-F10).
12. Compose verdict to action: a `propose` becomes a draft-only `creative.concept.draft` child (system_auto_approved, internal-only, no pipeline fire). The row renders as an In-production card ("Writing concept") with no fund affordance and no reason attached; the only way to act on Mira's proposal is to re-enter it through the brief box. DEAD-END LEG, planned desk affordance (D3-F3, D3-F8).

Net: the only loop segment that both writes and reads live data on main today is gesture -> taste -> polished prompts (legs 1-3). Every measured leg is dark behind `CREATIVE_ATTRIBUTION_ENABLED`, both brain initiators are dark behind their flags, and the revenue_proven leg has no producer.

## Findings

### D3-F1 (verified). P1. The revenue_proven half of creative memory has zero writers; the brain's measured-winner channel is structurally dead

Claim: no code anywhere writes a DeploymentMemory row with category `revenue_proven`, while the brain reads it in three places, so the "Measured winner" lines and the revenue_proven arm of the self-brief signal floor can never fire.

Evidence: repo-wide grep for `revenue_proven` hits only the schema enum (`packages/schemas/src/deployment-memory.ts:21`, comment at `:15`: "attributed-performance promotion, Riley-owned writer"), the brain reader (`packages/core/src/skill-runtime/builders/mira.ts:43` REVENUE_PROVEN_KEY, `:76-78` signal predicate, `:89-99` renders revenue_proven lines FIRST), tests, and specs. Slice4 spec fact 19 acknowledges: "revenue_proven exists in the DeploymentMemory category enum with ZERO writers". Slice2 spec 3.7 defines the promotion contract (Riley-owned writer in the outcome-attribution cron wrapper, floors spend >= 50, booked >= 2, trueRoas >= 1.5, source-scan test pinning writer location) with an explicit build trigger: "the first CreativeJob whose pastPerformance reaches delivery: 'measured' with meta.spend > 0".

Impact on the north star: the loop's promise is reallocating creative effort toward what PAYS. Today the only durable creative memory is subjective operator taste; attributed winners never persist as memory, so even with attribution flipped on, Mira's compose would weight a measured winner only while it sits inside the 200-job read-model window, and the system never accumulates "this pattern earned money" knowledge across cohorts. The deliberate reader-before-writer inversion (slice4 3.3 built the read "so the brain picks the signal up the week it appears") is sound, but the trigger condition is itself blocked behind the attribution flag, so this leg needs explicit sequencing in the pilot flip plan, not just a trigger.

Recommendation: when the build trigger fires (first measured row with spend), implement slice2 3.7 exactly as designed: injected seam in `apps/api/src/services/cron/riley-outcome-attribution.ts`, provenance-carrying content, the source-scanning writer-location test. Add the trigger check to the pilot flip runbook so it is not forgotten.

Tag: planned (slice2 spec 3.7; slice4 spec section 8 "Riley revenue_proven promotion writer"). Effort: M.

### D3-F2 (verified). P1. The entire brain is dark on main: two flags off plus three per-org prerequisites; zero orgs can experience Mira deciding what to make

Claim: both automated compose initiators are disabled by default, and even with flags on, a given org composes only if it has the seeded creative deployment plus compose allow policy (today org_dev only), Mira enablement, and billing entitlement; consequently no production org exercises the slice-4 brain.

Evidence: `.env.example:349,354` default both flags false; readers `apps/api/src/bootstrap/inngest.ts:1036` ("default off so the deploy is dark until deliberately enabled", `:1009`) and `:339`. The worker floor returns named skips for each missing prerequisite (`mira-self-brief.ts:140-143` flag/enablement/deployment, `:205` `org_not_entitled`). The compose allow policy is installed only by `seedMiraCreativeDeployment` (`seed-mira-creative-deployment.ts:123-132`), which "the dev seed runs ... for org_dev" only (`creative-governance.ts:20-24`); org_dev is unentitled by default (slice4 spec fact 31). The gate test proves an unseeded org default-denies compose.

Impact on the north star: the brain is the synergy piece (Riley diagnosis + taste + measured outcomes deciding the next creative). All of it is unproven against live traffic; the loop currently runs operator-brief-driven only. Every week dark is a week of zero compounding signal for the agent the moat depends on.

Recommendation: this is an operations task, not code: run the provisioning + entitlement + flag flip for one pilot org per slice4 spec section 7 (the flip checklist is already written and the skips are observable by name in Inngest run history). Pair with the attribution flag decision since the floors and PERFORMANCE_CONTEXT stay empty without it.

Tag: planned (slice4 spec section 7 "What flips it live"; memory note: NEXT = provisioning, flip runbook). Effort: S.

### D3-F3 (corrected during verification: desk-label citation only; substance and severity stand). P1. A Mira proposal cannot become a generated creative: concept drafts are inert rows with no funding affordance

Claim: the self-brief and handoff paths terminate in a draft-only CreativeJob that never fires the pipeline, the desk renders it as an inert In-production card with no action, and no intent exists to promote a concept row into generation, so the operator must re-type the proposal into the brief box.

Evidence: `creative-concept-draft-workflow.ts:140-141` ("NO inngestClient.send", draft-only by design); concept rows map to `in_progress` (`packages/core/src/creative-read-model/status-mapper.ts:57-66`, currentStage stays at the "trends" default) and render in the In-production tray with the trends-stage label "Writing concept" (`apps/dashboard/src/lib/cockpit/mira/desk-copy.ts:6`; the original report cited "Drafting", which is the production-stage label at `:10`); the tray itself carries zero action affordances (no onClick/Button in `mira-in-production-tray.tsx`), and review affordances gate on reviewable rows with a video. Repo grep confirms `creative.concept.draft` is the only concept intent: no promote/fund intent exists anywhere. Slice4 spec section 8 defers "Concept-to-brief desk affordance + concept-row presentation" with trigger "pilot operators receiving weekly self-briefs". `creative.job.continue` cannot start a never-dispatched job (see D3-F9).

Impact on the north star: "the brain proposes; the human disposes" currently means the human can only dispose by ignoring or manually re-entering. The proposal-to-production conversion cost is the whole value of self-initiation; without the affordance, weekly self-briefs would measure as noise in a pilot and poison the case for the loop.

Recommendation: ship the deferred affordance: a desk action on concept rows that submits `creative.job.submit` pre-filled from the concept's productDescription/targetAudience (governed, spend-thresholded, same as the brief box), and carry the compose reason onto the card (see D3-F8). Should land before or with the MIRA_SELF_BRIEF_ENABLED flip.

Tag: planned (slice4 spec section 8). Effort: M.

### D3-F4 (verified). P2. creative.job.submit trusts a caller-supplied deploymentId, leaving the taste topology invariant unenforced at the write seam

Claim: the legacy submit route accepts an arbitrary `deploymentId` in the request body and the workflow persists it unvalidated onto the CreativeJob, so the taste sweep would write taste buckets onto whatever deployment the caller named, including a conversational agent's deployment in the same org, where ContextBuilder demonstrably surfaces them as learnedFacts.

Evidence: `apps/api/src/routes/creative-pipeline.ts:16-21` (`SubmitBriefInput` takes `deploymentId: z.string().min(1)` from the client); `creative-job-submit-workflow.ts:6-11,57-69` uses `input.deploymentId` for the AgentTask and CreativeJob with no org-scope or slug check (contrast `creative-concept-draft-workflow.ts:96-103`, which re-validates `deployment.organizationId !== orgId`, and the mira-brief route, which resolves the deployment server-side at `mira-brief.ts:112-121`). The sweep writes to `job.deploymentId` (`creative-taste-sweep.ts:126-131` via `TasteCandidate.deploymentId`, `prisma-creative-job-store.ts:42-53`). The isolation test documents the consequence: taste rows DO flow into the owning deployment's learnedFacts (`taste-memory-isolation.test.ts:70-87`), and the slice2 spec calls the topology a "stated TOPOLOGY INVARIANT" (3.4).

Impact on the north star: a buggy dashboard build or any API client passing the wrong deployment id silently pollutes Alex's conversation memory with creative-taste statements and lets taste rows compete in Alex's 500-entry eviction pool, exactly what the firewall exists to prevent. Same-org only (taste writes are org-scoped), so no cross-tenant leak, hence P2 not P0.

Recommendation: in `buildCreativeJobSubmitWorkflow`, resolve or validate the deployment server-side: assert the row exists, belongs to `workUnit.organizationId`, and has `skillSlug === "creative"` before creating the job (one findById, mirroring the concept-draft workflow's defense-in-depth), or drop the client-supplied field and resolve by org+slug as the mira-brief route does.

Tag: new. Effort: S.

### D3-F5 (verified). P2. The handoff compose closure degrades to the api-direct sentinel instead of skipping when an org has no creative deployment

Claim: when `MIRA_HANDOFF_BRIEF_ENRICHMENT_ENABLED` is on and a Riley org lacks a creative deployment, the compose submit closure resolves the targetHint via `resolveDeploymentForIntent`, which returns `deploymentId: "api-direct"` on a failed or null resolve, so the compose would be submitted against a sentinel deployment; today this is contained only because the compose allow policy is seeded together with the deployment, so governance default-denies and the path falls back to synthesis.

Evidence: `contained-workflows.ts:555-567` (`deployment ?? await resolveDeploymentForIntent(...)` then unconditional submit); `apps/api/src/utils/resolve-deployment.ts:11-30` (both the no-resolver and catch branches return `deploymentId: "api-direct"`; a null resolver result throws inside `toDeploymentContext` and lands in the same catch); `handoff-brief-enrichment.ts:20-27` documents the intent ("the closure resolves the org's CREATIVE deployment by intent prefix itself") but not the failure mode. Containment: compose default-denies without the seeded policy (`mira-brief-compose-gate.test.ts:150-154`), and the policy ships only with the deployment (`seed-mira-creative-deployment.ts:123-132`).

Impact on the north star: latent, not live. The day anyone seeds the compose allow policy independently of the deployment (or a broader allow policy matches), enrichment composes would run with an api-direct deployment id: empty taste memory, a mislabeled WorkTrace deployment, and an LLM call per emitted recommendation for an org Mira is not provisioned in. The self-brief worker explicitly avoids this class ("no api-direct fallback can leak into a submit: the floor fails closed first", `inngest.ts:1012-1014`); the handoff path lacks the same guarantee.

Recommendation: make `submitMiraBriefCompose` fail closed when no deployment was passed and `resolveByOrgAndSlug(orgId, "creative")` returns null: return an `ok:false` shape (or throw) so `resolveHandoffBrief` takes its existing synthesize fallback. One null check plus a test.

Tag: new. Effort: S.

### D3-F6 (verified). P2. The off-scope brief guard is client-side only; the API accepts off-scope question briefs into a paid pipeline

Claim: `classifyBriefIntent` (the guard that keeps scheduling/results QUESTIONS out of the brief box) is consumed only by the dashboard component; the `/agents/mira/brief` route and the legacy `/creative-jobs` route never call it, so any non-dashboard client submits an off-scope brief straight into `creative.job.submit`.

Evidence: `packages/schemas/src/mira-brief.ts:84-93` defines the guard with the comment "The Intent Preview uses this to redirect instead of submitting"; repo grep shows the only non-dist consumer is `apps/dashboard/src/components/cockpit/mira/mira-brief-box.tsx:45`; the route file `apps/api/src/routes/agent-home/mira-brief.ts` imports `MiraBriefRequestSchema` and `mapMiraBriefToCreativeBrief` only (`:23-27`) and submits at `:126-140`.

Impact on the north star: bounded waste, not safety: an off-scope brief ("how much revenue did the ad make?") becomes a garbage draft and render spend is still governed by the spend-approval threshold. But the server is the contract surface; a future chat-channel or API integration repeats the dashboard's work or silently skips it.

Recommendation: run `classifyBriefIntent(parsed.data.promoting)` in the route and return 422 with a redirect hint for `off_scope`, keeping the dashboard preview as UX. One branch plus a test.

Tag: new. Effort: S.

Recommendation: implement the named follow-on at its named wiring point, injecting surfaced ugc taste lines into `runUgcScriptWriter` constraints, triggered when ugc buckets exist with real source counts (slice3 spec lines 438-440 and 606; slice4 spec section 8).

Tag: planned. Effort: M.

### D3-F8 (verified). P2. Compose provenance never reaches the operator: abstain reasons and proposal rationale live only in Inngest history and WorkTrace

Claim: when Mira abstains, the reason exists only as the worker's return value in Inngest run history; when she proposes, the concept row carries no reason and no injected taste keys, so the desk card shows a bare Drafting row with no "why".

Evidence: outcome shapes `{ abstained: reason } | { skipped } | { jobId }` are returned, not persisted to any operator surface (`mira-self-brief.ts:122-125,235`); the draft submit passes only `brief.productDescription/targetAudience` (`mira-self-brief-request.ts:79-85`); the concept workflow outputs only `{ jobId }` (`creative-concept-draft-workflow.ts:141-145`). Injected keys flow to WorkTrace via the rich builder result (`skill-mode.ts:105-118`), which is auditable but not a product surface.

Impact on the north star: the human-in-the-loop is supposed to train Mira through gestures; an unexplained proposal earns a reflexive Pass, which then writes a MISLEADING taste observation against the pattern Mira chose for reasons the operator never saw. Provenance is also the trust currency for the eventual autonomy ratchet.

Recommendation: when the concept-row desk treatment ships (D3-F3), persist `reason` (and optionally the rendered taste lines) into the AgentTask input or a concept-row field at draft creation, and render it on the card. Until then, at minimum log the abstain reason through a queryable surface rather than run history only.

### D3-F9 (verified). P2. creative.job.continue on a never-dispatched concept row reports queued success while doing nothing

Claim: the decision workflow's not-awaiting guard only rejects finished or stopped jobs, so a `continue` against a draft-only concept row passes the guard, emits a `stage.approved` event that no pipeline run is waiting for, and returns `outcome: "queued"` with `action: "approved"`, a phantom success.

Evidence: guard checks only `ugcDone || polishedDone || job.stoppedAt` (`creative-job-decision-workflow.ts:34-46`); a concept row has `currentStage: "trends"`, never dispatched (`creative-concept-draft-workflow.ts:140` "NO inngestClient.send"); the event emit and unconditional queued return are at `:68-85`. Inngest `waitForEvent` only matches events sent while waiting, so the emitted approval is dropped.

Impact on the north star: not reachable from the desk today (review affordances gate on reviewable rows), but reachable through the API (`POST /creative-jobs/:id/approve`), and it will sit directly under any future "fund this" affordance built naively on `continue` (D3-F3): the natural implementation of the missing funding path would silently no-op. Classic queued-as-success trap adjacent to the #860 lesson.

Recommendation: in the decision workflow, reject jobs that have never been dispatched (empty `stageOutputs` and stage at the initial value, or better: a dispatch marker) with a named error such as `CREATIVE_JOB_NOT_RUNNING`, and add a test. Document that funding a concept requires a fresh `creative.job.submit`, not `continue`.
