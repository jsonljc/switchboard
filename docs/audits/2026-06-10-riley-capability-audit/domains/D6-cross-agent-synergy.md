# D6 — Cross-agent synergy (Alex–Riley–Mira flywheel)

Audit date: 2026-06-10. Repo read at worktree `riley-capability-audit` (branch `docs/provisioning-runbook`, top `9f213538`). Baseline: `docs/audits/2026-06-02-riley-improvement-audit/FINDINGS.md` + `docs/audits/2026-06-03-agent-synergy-architecture/findings.md`. All file:line references verified against current code.

## Thesis

Since the 2026-06-03 synergy audit declared "Riley→{Mira,Alex} handoffs do not exist in code," the repo shipped the entire Contract-3 orchestration plane: a governed, mandatory-approval, system-principal Riley→Mira advisory handoff fired end-to-end on org*dev, with real-gate integration tests, plus all three publish blockers (durable asset producer, pageId setter, dead-lettered Meta chain). The skeleton of the flywheel is therefore **real and correctly governed**. What is missing is the \_payload*: the handoff arrives at Mira stripped of everything Riley knew (campaign, evidence, history, provenance); the initiator cannot see its own failures (response discarded — the exact defect class the newer pause initiator alarms on); the loop is provisioned for exactly one dev org; Riley still cannot see Mira's creatives as creatives; and Riley still says nothing to Alex. The system is one provisioning script, one submitter retrofit, and one brief-context PR away from a pilot-grade flywheel — and one larger join (creative provenance in Riley's engine) away from the compounding-learning moat.

## Findings detail

### D6-1 (P1, net-new) — Handoff initiator is response-blind

`apps/api/src/bootstrap/inngest.ts:357-366`: `await options.submitRecommendationHandoff(input, {...})` — the `SubmitWorkResponse` is discarded; only thrown errors are caught. Consequences, none logged: governance deny (`ok:true, outcome:"failed"`), entitlement miss (`ok:false, entitlement_required`), and — worst — an unexpected immediate execute if the seeded mandatory policy is missing/relaxed, which would create a Mira draft **without the human approval the contract promises** (draft-only/no-spend, hence P1 not P0). The Phase-C pause initiator written weeks later pins exactly these branches loudly (`apps/api/src/bootstrap/riley-pause-submitter.ts:47-81`: named entitlement skip, loud deny, "UNEXPECTEDLY executed without approval" alarm, park-truth return). Amplifier: report ownership annotates `mira_handoff` from gate-eligibility alone, never submit truth (`recommendation-ownership.ts:15-19`), so the cockpit says "Mira-owned" while submits silently deny. **Fix (S):** retrofit the pause-submitter branch ladder; optionally feed park truth into ownership like `riley_self`.

### D6-2 (P1, known-open) — Loop is org_dev-seed-only

The five-piece set (Riley deployment, handoff policies, system principal, creative deployment, Mira enablement — per `docs/superpowers/specs/2026-06-04-riley-handoff-org-dev-live-loop-design.md`) is installed exclusively by the dev seed (`packages/db/prisma/seed.ts:95,611,620`). The seed functions are idempotent and reusable (`seed-riley-ad-optimizer-deployment.ts`, `seed-mira-creative-deployment.ts:107-121`) but have **no production caller**: no install flow, no script, and the runbooks cover only Mira enablement and infra. **Fix (M):** an audited per-org provisioning script (riley-pause-flag.ts pattern) + a five-piece readiness assertion.

### D6-3 (P1, net-new) — Riley's context discarded at the Mira brief seam

The handoff WorkUnit carries `recommendationId/campaignId/rationale/evidence`; the approval card shows them (`parked-approval-cards.ts:29-66`); the human approves on them. The draft child then receives only `{productDescription, targetAudience}` (`recommendation-handoff-workflow.ts:79-87`; `RecommendationHandoffSubmitInput.brief`, `recommendation-handoff-request.ts:13`), with `pastPerformance: null` (`creative-concept-draft-workflow.ts:136` — measured-history enrichment exists only on the operator `creative.job.submit` path, `creative-job-submit-workflow.ts:42-55`). The synthesis fallback is a generic medspa template (`creative-brief-synthesis.ts:22-23`); enrichment is dark; even enriched output is product/audience only (`handoff-brief-enrichment.ts:99-102`); `valueContext` is supported downstream but never populated on the Riley path; the Mira desk has no Riley-provenance state (`desk-model.ts:5-16`). This is the baseline's meta-finding ("computed then discarded") live at the loop's most synergy-dense seam. **Fix (M):** carry campaignId/recommendationId/evidence onto the draft brief, run measured-history enrichment on the draft path, render a "from Riley" provenance line on /mira.

### D6-4 (P2, known-open) — Edge C (Mira→Riley learn-back) still absent

Zero `metaCampaignId/metaAdId/metaVideoId` in `packages/ad-optimizer/src`. The creative-attribution worker exists but is dark (`CREATIVE_ATTRIBUTION_ENABLED=false`) and Mira-facing only. Implicit campaign-grain closure exists (published Mira campaign enters Riley's audit) but is provenance-blind: a `refresh_creative` on a Mira-made campaign hands back to Mira with no link to the original CreativeJob. **Fix (L, post-publish-live):** CreativeProvenanceReader port into audit deps; chain handoffs to predecessor jobs.

### D6-5 (P2, known-open) — Riley→Alex lead-quality leg missing (baseline 5.6)

`lead_quality_issue`/`lead_quality_degradation` (`metric-diagnostician.ts:78,115`) map to no recommendation action, no ownership class, no Alex-readable surface; their only effect is blocking the scale rule via `diagnoses.length === 0` (`recommendation-engine.ts:223`). **Fix (M):** read-surface signal (per the synergy doc: NOT an ingress mutation) + report line.

### D6-6 (refinement, net-new) — Receipted truth not in Riley's ladder

`purchased` ConversionRecords carry `sourceCampaignId` but both Riley feeders filter `type:"booked"` (`prisma-conversion-record-store.ts:236,274`). Booked-truth-only is consistent with the stated north star today; paid-vs-booked divergence per campaign is the natural Spec-1B input. **Fix (S).**

### D6-7 (verify-shipped) — Contract 3 + publish blockers genuinely shipped and tested

Confirmed against current code: governed handoff E2E through the REAL ingress→gate→workflow seam (`recommendation-handoff-harness.ts:1-9` and the cron-full-loop / approval-loop / chat-bridge suites); anchored+escaped policy regexes; system principal verbatim; no `system_auto_approved` on the spend-adjacent path (`contained-workflows.ts:408-421`); fail-closed learning default and child-jobId phantom-success guard in the workflow; durableAssetUrl producers, pageId setter, and the dead-lettered publish fn all exist. The synergy doc's priority-1 and priority-3 items are done; its risk list (system-principal bite, spend-path auto-approve, producer-in-same-PR) was followed.

### D6-8 (refinement) — Five dark flags, no flip plan

With all off, the live flywheel is deterministic recommendations + a template-brief handoff. Needs an ordered flip runbook (fold into D6-2's provisioning workstream).

## What is sound (with evidence)

1. **The governance shape of the handoff is exactly right.** System principal verbatim (`recommendation-handoff-request.ts:49`), deterministic idempotency keys at both hops (`handoff:riley:<rec>:<action>` ingress; `handoff-draft:<rec>:<action>` child), allow+mandatory seeded as a pair with anchored regexes (`recommendation-handoff-governance.ts:26-89`), `system_auto_approved` confined to the no-spend draft child with the spend-path warning written into the registration comment (`contained-workflows.ts:390-421`). The #788 landmine is explicitly designed around.
2. **Abstention is pure, shared, and re-checked.** One `shouldAbstainFromHandoff` (`recommendation-handoff-abstention.ts:41-52`) used by initiator and handler; the handler's learning-flag default fails closed (`recommendation-handoff-workflow.ts:40-42`); dropped recommendations can never become approval tasks (`recommendation-handoff-dispatch.ts:97`).
3. **Phantom success is hunted at every arm.** The workflow rejects parked/failed children, and uniquely requires the child's `jobId` so an approved handoff never reports a draft that doesn't exist (`recommendation-handoff-workflow.ts:90-137`). The approve leg ends in dispatch-or-recovery by construction (`respond-to-approval.ts:15-18`).
4. **The test posture matches the repo's own systemic-risk diagnosis.** The harness drives the real `executeWeeklyAudit → PlatformIngress → GovernanceGate (seeded policies) → workflow` seam, the precise live-path-test discipline the 2026-06-03 doc demanded after the spy-test publish-gate miss.
5. **The choreography plane (Edge D) is the contract the doc prescribed.** Booked truth flows as schema-typed events/records, never through ingress; Riley reads it via a port; receipts ride the same bus.

## Bottom line

Live: Alex→Riley truth, Riley→Mira handoff mechanics (on org_dev), the approval loop, the publish path. Dark: every learning layer (5 flags). Missing: provisioning beyond org_dev, initiator failure visibility, brief context, creative provenance in Riley, Riley→Alex. Fix D6-1 (S) + D6-2 (M) + D6-3 (M) and the pilot flywheel is real; D6-4 is the moat work after publish goes live.
