# D2 — Perception, Meta integration & operational scale (riley-capability-audit, 2026-06-10)

> Re-audit of baseline D2-perception-signals + D8-ops-scale (2026-06-02), against current main-tracking worktree. Every claim re-verified against current code; file:line relative to repo root.

## Thesis

Riley's perception **semantics** improved markedly since the baseline — the breach denominator is action-type-aware and NaN-guarded, the outcome-attribution Meta adapter's `breakdowns:["day"]` bug is fixed, per-source spend attribution degrades honestly, the reconciliation runner is real, token refresh runs daily. But the perception **substrate** — the thing that must survive contact with the real Graph API at pilot scale — is where the baseline's headline findings (D8 a/b/d/e/f/h, D2 G8) live, and essentially none of them shipped. The weekly audit still cannot plausibly complete one real deployment inside its single Inngest step; failures are silent by configuration; and two net-new discoveries (likely-invalid insights fields; tokens in Inngest step state) mean the pipeline has never been proven against, and is not safe to point at, a production Meta account as-is.

**Domain verdict: cannot run a 5–10 org / 5+ campaign pilot today.** It fails over (step wall-time vs serve-transport timeout) and goes silently blind (alert:false everywhere, no zero-output signal) simultaneously. Crucially, every failure is toward _inaction_, not wrong money actions — the abstention instincts are good — but a pilot where Riley silently does nothing for weeks is a failed pilot. All fixes are S/M effort, and the batching fix alone (D2-7: the account-level data is already over-fetched N times and discarded) removes most of the scale problem without adding risk.

## Wall-time math (re-derived, current code)

Per deployment inside the single `audit-${id}` step, behind the 60s limiter on one shared client: Step-1 (2 campaign-insights + 3 getAccountSummary, Promise.all races collapse to ~2 slots ≈ 2 min) + account ad-set attribution (2 calls) + per campaign: learning (1 acct insights + /adsets + adset insights = 3) + breach window (1) = 4N. **N=1 → ~10 min; N=5 → ~25 min; N=10 → ~45 min — inside ONE HTTP invocation** of the Inngest serve transport behind Render (starter web service, render.yaml). retries:2 re-run the identical step. Plus 6 unthrottled signal-health calls. 10 orgs × 5 campaigns serial ≈ 4+ hours even if no single invocation were killed. The in-code TODO (inngest-functions.ts:148-158) frames this as a future multi-tenant scale issue; it is actually a **single-deployment** correctness issue now.

## Findings detail

See structured findings D2-1 … D2-17. Headlines:

- **D2-1 (P1, known-open)** — audit step wall-time vs serve-invocation budget: the pilot-fatal item. Fix = batch (D2-7) + per-deployment event fan-out (pattern already in-repo at inngest-functions.ts:460-504).
- **D2-2 (P1, net-new)** — `status`/`effective_status`/`revenue` are almost certainly invalid AdsInsights fields (#100). All tests mock the client; no integration fixture; mapCampaignInsight would silently default them even if tolerated. One live sandbox call settles it; until then the audit pipeline must be presumed never-run-against-real-Meta.
- **D2-3 (P1, net-new)** — serial loop + step-failure-fails-function + cred resolver that ignores `needs_reauth` (bootstrap/inngest.ts:394-402, vs the status-filtering siblings at :700 and meta-insights-adapter.ts:32) = one customer's dead token silently disables Riley for every org behind it, weekly.
- **D2-4 (P1, net-new)** — decrypted Meta tokens returned from `step.run("creds-…")` are memoized into Inngest Cloud step state; the repo's own creative-publish wiring explicitly avoids this ("never serialized into Inngest step state", bootstrap/inngest.ts:493-496).
- **D2-5 (P1, known-open)** — no throttle/backoff/error-classification anywhere on the Meta surface; dead tokens burn full retries and then trip D2-3.
- **D2-6 (P1, verify-shipped-with-gap)** — the Gate-1 action-type denominator is fully shipped _and producer-starved_: no seed/wizard writes `conversionActionType` (seed writes only budget/targets — as strings — seed-riley-ad-optimizer-deployment.ts:52-57), so production breaches still run on aggregate `conversions`. The platform's documented safety-gate-needs-producer-population pattern, recurring.
- **D2-13 (P2, verify-shipped-with-bug)** — reconciliation is real now but window-mismatched (all-time vs 7-day) → permanently "failing" drift once an org has history; it feeds /roi/summary and nothing alerts on it either way.
- Supporting: D2-7 batching (the highest-leverage fix), D2-8 coverage gate unwired, D2-9 observability, D2-10 token-lifecycle half-measures, D2-11 signal-health duplicate fetches, D2-14 fresh-client limiter bypass on the outcome worker, D2-15 computed-then-discarded daily check, D2-16 racy limiter, D2-17 trends/saturation still dead.

## What is sound (with evidence)

- **Fail-safe degradation discipline at every new seam.** The account ad-set attribution fetch catches, logs at error level, and degrades to lead-share → honest abstain instead of crashing the run (inngest-functions.ts:234-248); getCampaignStatus degrades to null with an explicit rationale comment (meta-ads-client.ts:365-382); the breach detector skips zero-spend days and floors zero-conversion-day counting on a 20-click volume gate (meta-campaign-insights-provider.ts:131-158); page-1 truncation on /adsets is documented as coverage-drop → honest abstain (meta-ads-client.ts:189-192). Failures consistently point toward inaction, not wrong action.
- **The Gate-1 denominator implementation itself.** Action-type selection rides a pinned `action_attribution_windows`, the raw `actions` array is surfaced verbatim, and the per-day read is `Number.isFinite`-guarded (meta-ads-client.ts:124-129,475-477; meta-campaign-insights-provider.ts:144-150). The code is right; only the producer is missing (D2-6).
- **The outcome-attribution Meta adapter (verify-shipped, confirmed).** timeIncrement:1 replaces the invalid breakdown, the inclusive/exclusive bound conversion is correct, credentials are re-resolved per call by design, and a non-finite account-spend sum is reported as absence so corroboration cannot be fabricated (meta-insights-adapter.ts:24-25,46-60,70,84) — the #939 lesson applied.
- **The fan-out pattern exists and works where it was used.** Outcome attribution and creative attribution use dispatcher→per-org-event workers with kill-switch flags and per-org isolation (inngest-functions.ts:460-504; bootstrap/inngest.ts:921-1000) — the exact architecture the weekly audit needs is already proven in the same file it's missing from.
- **Signal-health honesty semantics.** dedupRate is null (not 0) when uncomputable, preventing false dedup_low breaches, and warning-tier breaches are suppressed when the ratio signal is itself unreliable (signal-health-checker.ts:42-55,232-240,356-380).
- **Token refresh cadence fixed.** The refresh cron is daily (03:00) with per-connection steps and needs_reauth marking (meta-token-refresh.ts:105-126) — the baseline's "weekly refresh can miss mid-week expiry" criticism is addressed; what remains is consumption of that state (D2-10/D2-3).
