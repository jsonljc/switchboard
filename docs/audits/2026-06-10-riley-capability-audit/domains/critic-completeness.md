# D10 — Completeness Critic: What the Riley Capability Audit Missed

## Thesis

Nine domains audited Riley's judgment, perception, economics, control plane, governance, synergy, learning, cockpit, and backlog — and every one of them implicitly assumed Riley _has_ a live Meta credential. Nobody audited where that credential comes from or how it stays alive. That axis turns out to contain the audit's single largest un-flagged pilot blocker: **a production org cannot credential Riley through any product surface, and even a hand-provisioned token silently dies at day 60 because the "shipped" refresh cron reads a field no producer writes.** This is the baseline's own meta-finding ("consumer shipped, producer never populates") recurring inside a claim D2 marked as a shipped improvement — exactly the failure mode a verify-shipped pass is supposed to catch, one field-name deep.

A second un-owned seam: the facebook-oauth routes are the only Riley-adjacent API surface with **zero org-scoping** (the marketplace routes right next to them do it correctly), plus an unsigned OAuth `state`, giving cross-tenant use of a victim org's decrypted Meta token today and an OAuth-CSRF tenant-binding hole the moment the flow is unbroken.

Everything else on the completeness checklist (schema/migrations, indexes, env allowlists, credential crypto, multi-tenancy of the recommendation stores, PII, LLM cost) was checked and is substantially sound — detailed below so future audits don't re-litigate.

## Findings

### D10-1 (P1, net-new) — No working production path to credential Riley

Two parallel credential worlds exist (the repo's own `Connection` vs `DeploymentConnection` gotcha), and Riley lives entirely in the one the product cannot populate. The dashboard's manual "System User token" paste lands in org-level `Connection`; every Riley cron and the pause executor resolve from `DeploymentConnection`; the only `DeploymentConnection` producer for `meta-ads` is the OAuth callback, and both OAuth browser legs 401 against the Bearer-only auth middleware in any auth-enabled environment. Net: for a marketplace/production-provisioned org, Riley's whole stack silently no-ops (`getDeploymentCredentials` → null → skip). This is the credential-axis twin of the D4/D5 policy-seeding gap, and it sat between D2 (audited the resolver) and D8 (audited the cockpit) with no owner. Fix: one canonical store (cheapest: resolver falls back to org `Connection` serviceId `meta-ads`), plus auth-exempt the two OAuth redirect legs **only together with D10-3's signed state**.

### D10-2 (P1, verify-shipped FAILED) — meta-token-refresh cron is inert: `tokenExpiresAt` has no producer

The cron's first read is `creds.tokenExpiresAt`; missing → `console.warn` and return, status stays `active`, nobody notified. The OAuth callback writes `expiresAt`; the manual UI writes nothing; the only `tokenExpiresAt` writers are the cron's own success leg and a dead module. Every connection that exists or can be created today is permanently skipped: a pilot org goes dark at ~day 60 with zero alert. D2 counted this cron as a shipped improvement and even flagged its softer edges (failures notify no one, `needs_reauth` unconsumed) — but missed that the cron never reaches those edges. One-field-deep producer→consumer tracing on verify-shipped items is the lesson.

### D10-3 (P1, net-new) — facebook-oauth routes: zero org-scoping + unsigned OAuth state

`/facebook/:deploymentId/accounts` decrypts the stored token for ANY deploymentId and calls Meta with it — any tenant's API key (or any caller where auth is disabled) can enumerate another org's ad accounts on the victim's token, today. The callback trusts `state` as a raw deploymentId and binds whatever Meta account completes the dialog to that deployment: once D10-1 unbreaks the flow, this is a standard OAuth-CSRF that lets an attacker bind attacker-controlled ad-account credentials to a victim's Riley — i.e., redirect the money agent's perception. The fix is small and has an in-repo template (the marketplace routes' org check) plus an HMAC-signed state.

### D10-4 (P2) — FACEBOOK*\* vs META*\* env split for one Meta app

Onboarding and refresh read different env families with empty-string fallbacks; configuring only the one onboarding needs silently re-breaks token lifecycle later. Collapse to one family with a startup readiness check.

### D10-5 (refinement) — Dead duplicate refresh path (`refreshMetaOAuthToken`, Connection table, zero callers, `refreshToken: access_token` oddity). Delete or unify when fixing D10-2.

### D10-6 (refinement) — No `appsecret_proof` anywhere; app secret + tokens in GET query strings on the OAuth/refresh legs. Blocks enabling Meta's "Require App Secret" hardening; relevant given D2's tokens-in-Inngest-state finding and the Meta App Review critical path. The MetaAdsClient itself correctly uses Bearer headers.

### D10-7 (refinement) — `PendingActionRecord` lacks composite indexes for `listResolvedForAgent` (org+sourceAgent+status+resolvedAt) and `latestByAgent` (org+sourceAgent+createdAt) on a shared high-write table. Fine at pilot scale; add when the wins surface goes live.

### D10-8 (verified sound) — Completeness sweep closing the brief's suspicion list

- **Credential crypto**: AES-256-GCM, per-row scrypt salt, mandatory `CREDENTIALS_ENCRYPTION_KEY`, no dev-default key.
- **Multi-tenancy of Riley's stores**: every recommendation list/act/machine-transition is org-scoped at the store or handler layer; cross-tenant act conflates to NOT_FOUND without leaking existence; `getById` is unscoped but both its callers re-check `row.orgId` before any use (`act.ts:54`, `operator-intents/recommendation.ts:28`).
- **API auth**: production refuses unscoped static keys at startup and runtime (`auth.ts:82-98, 177-183`) — the tenant-impersonation primitive is closed.
- **Env flags & allowlists**: both Riley flags allowlisted and documented default-off.
- **Migrations**: all three Riley tables have migrations, including the raw-SQL CHECK-constraint ones the schema comments promise.
- **LLM cost**: Riley's decision path is fully deterministic — no per-decision token spend exists to audit.
- **PII**: lead PII (email/phone/name from `meta-leads-ingester`) flows to contact records, not into recommendation rows, outcome rows, or `ConversionRecord` (contactId reference only); the leadgen webhook surface has signature-verification tests (`apps/api/src/__tests__/ad-optimizer-signature.test.ts`).

## What this says about the audit

The per-domain decomposition reproduced the known cross-slice-seam failure mode (`feedback_per_slice_review_misses_cross_slice_seams`): every domain traced producer→consumer seams _inside_ its slice, while the credential lifecycle — provisioning → storage → resolution → refresh — crosses dashboard, API routes, db stores, and Inngest bootstrap and was assumed owned elsewhere. The two P1s here (D10-1, D10-2) belong on the same pilot-blocking list as D4/D5's policy-seeding gap and D9's config-producer gap; together they form a single theme: **Riley's shipped machinery is real, but the org-onboarding producers that feed it (policies, targets, denominators, and now credentials) were never built.**
