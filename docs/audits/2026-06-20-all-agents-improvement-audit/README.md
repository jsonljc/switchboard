# All-Agents Improvement Audit (2026-06-20)

Full multi-agent improvement audit of Switchboard `main` as of 2026-06-20. Covers all seven governed capabilities (Alex, Riley, Casey, Ledger-lite, Quinn-lite, Robin, Mira) plus four cross-cutting seam tracks (governance/security, prod-wiring/activation, test integrity, proof-chain/data-integrity, operational go-live). Each finding was produced by a per-agent or seam auditor, then adversarially re-verified against current `main` by a second pass that read the cited code and traced producer to consumer.

## (a) Executive summary

The platform is structurally sound: the core invariants (mutating actions through `PlatformIngress.submit()`, `WorkTrace` as canonical persistence, approval as lifecycle state, no mutating bypass paths) hold across every path traced. No P0 code bugs survived verification. The materially important work clusters into three themes, all of which gate the weekly-receipted-bookings north star or the go-live that activates it:

1. **Multi-tenant send identity and attribution are not finished.** Proactive WhatsApp sends (reminders, follow-ups, greeting, recovery) all resolve a single global phone-number-id and token, so tenant #2 would send from tenant #1's WhatsApp number to its own patients (rank 1). CTWA inbound leads are attributed to a channel-connection id rather than the Alex `AgentDeployment` id, so per-agent activity rollups silently miss every CTWA lead (rank 4). These are the highest-leverage go-live blockers because CTWA is a primary paid-lead funnel and WhatsApp is the primary outbound channel.

2. **The proof chain has links that are structurally inert.** A parked-then-approved booking never gets its `WorkTrace.approvalId` stamped, so the receipted-booking `humanApprovalId` is always null (rank 5). Automatic duplicate-contact detection does not exist, so `duplicate_contact_risk` only ever appears via a manual operator flag, and duplicate Contacts can be created freely across intake sources (rank 3). An `attended -> no_show` correction leaves the calendar receipt stuck at `held` (rank 12).

3. **Autonomous-money and consent guardrails advertise protections that are not wired.** Riley's blast-radius guardrails and automated prior-budget rollback are documented as the breach safety net but have no consumer (rank 6); Riley scales budget on lead COUNT, not paid VALUE (rank 9). Casey's first-touch greeting skips the template-approval and 24h-window gates every sibling proactive send enforces (rank 2). Robin burns an entire cohort's dedup rows if dispatched while the template is still draft (rank 7).

Counts after dedupe: 0 P0, 7 P1, 17 P2, 11 P3 (35 total). Two near-identical pairs were merged (greeting-gate-bypass surfaced by both Casey and the governance seam; duplicate-contact-detection surfaced by both Casey and the proof-chain seam). The known-dark items (recovery.mode off, LEDGER_WEEKLY_REPORT_ENABLED off, consentState not enforcing, unset WhatsApp/Resend creds, draft templates, RILEY_REALLOCATE_SELF_EXECUTION_ENABLED false) are treated as go-live sequencing context, not as code bugs, except where the wiring itself is broken or asymmetric.

## (b) Per-agent findings

### Alex (capture/booking)

1. **Attendance correction `attended -> no_show` leaves the receipt stuck at `held`** (P2, proof-chain). `attendance.ts:58-64` only promotes booked to held when `outcome === "attended"`; `prisma-receipt-store.ts:63-74` has no inverse held to booked path. `is-paid-visit.ts:23` and `ReceiptedBookingView` keep reporting `held=true` for a booking the operator marked `no_show`. The report-level held-rate self-heals (it reads `Booking.attendance`), so only the receipt-derived proof view desyncs. Add `demoteCalendarHeldToBooked` on `no_show` plus an attended-then-no_show test.

2. **`booking.create` failure can leave a `pending_confirmation` row blocking the slot** (P2, correctness). `calendar-book.ts:351-361` persists the booking as `pending_confirmation` before the provider call; if the compensating `failureHandler.handle` tx itself throws (`booking-failure-handler.ts:88-137`), the row stays pending and the overlap guard (`prisma-booking-store.ts:56-65`, status notIn [failed, cancelled]) durably blocks the org-wide slot. No reaper exists. Add a stalled-pending sweep or exclude `pending_confirmation` after a TTL; at minimum metric/log when `handle()` throws.

3. **reschedule/cancel resolve the SOONEST upcoming booking, silently acting on the wrong appointment** (P2, correctness). `calendar-reschedule.ts:42-47` `resolveTarget` returns `(narrowed.length>0 ? narrowed : bookings)[0]`: when a `service` filter matches nothing it falls back to all bookings and picks the soonest. For cancel this is an irreversible wrong-target mutation. When `service` is supplied but matches no upcoming booking, escalate `NO_UPCOMING_BOOKING` instead of falling back. Add a two-service mismatch test.

4. **Booked outbox `occurredAt` is the future slotStart, not the booking time** (P2, code-quality/attribution). `calendar-book.ts:460` stamps the booked conversion event at the future appointment time. `OutboxPublisher` -> `ConversionEvent.occurredAt` -> Meta CAPI `event_time` and Google `conversionDateTime`, so the conversion appears to occur days in the future (CAPI rejects future `event_time`; the 7-day stale guard cannot catch it). Use `now` for `occurredAt`; `slotStart` already lives in metadata (line 466).

### Riley (ads optimizer)

5. **Blast-radius guardrails + automated prior-budget rollback are decorative** (P1, prod-wiring). `blast-radius-contract.ts:147-155` ships guardrails (`account_booked_conversions_drop_share` 0.2, `freed_budget_absorbed_share` 0.5) and a `reset_prior_budget` rollback, documented as the breach response the "forward outcome-attribution monitor" evaluates. The named cron `riley-outcome-attribution.ts` and `outcome-attribution.ts` contain ZERO references to blastRadius/guardrail/rollback/prior_budget; a repo-wide grep finds no consumer. The executor captures `observedPriorCents` precisely so a rollback could restore it, but nothing consumes it. The only real protection is the pre-write cap. Either wire a real guardrail-evaluation + `reset_prior_budget` step into the cron, or down-scope the contract so operators do not over-trust it. Go-live blocker for the self-execution flag.

6. **Reallocation moves budget on booked COUNT (CPA), not paid VALUE** (P2, correctness). The executable money-move fires only on a `scale` rec (`riley-budget-dispatch.ts:72`), produced purely from count-based CPA (`recommendation-engine.ts:340-360`, `getCPA` denominator = Meta `conversions` count). The paid-value query (`queryBookedValueCentsByCampaign`) feeds only `trueROAS` display, never the scale gate. A cheap-per-lead campaign whose leads never pay still gets auto-scaled. Gate the scale-to-reallocate transition on a paid-value/trueROAS floor before flipping `RILEY_REALLOCATE_SELF_EXECUTION_ENABLED`.

7. **v1 reallocation only scales UP; never moves budget away from losers** (P3, arch, needs-human). `budget-reallocation-plan.ts:48` `REALLOCATE_SCALE_FACTOR = 1.2`, documented v1 scope; the cross-source shift is advisory-only. Not a bug, but the "reallocate" naming overstates a zero-sum rebalance that is really monotonic spend growth on cheap-CPA campaigns. Ensure approval cards and the go-live brief make clear executing reallocation = budget increase only.

### Casey (consent/intake/identity)

8. **`meta.lead.greeting.send` skips the 24h-window/opt-in AND template-approval gates its siblings enforce** (P1, correctness). Merged with the governance-seam finding of the same shape. `meta-lead-greeting-workflow.ts:89-101` gates only on `evaluateConsentGate` then POSTs `input.templateName` directly (148-169); reminder/follow-up/Robin all use `evaluateProactiveSendEligibility` which enforces `canSendWhatsAppTemplate` (24h window OR opt-in) and `template.approvalStatus === "approved"`. The greeting `getSendContext` resolves no approvalOverlay. Once WhatsApp creds flip, a draft/unapproved first-touch template could fire, risking Meta number quality. Note the fix is not a mechanical reroute: the WhatsApp template registry has no greeting/first-touch intentClass, and whether the CTWA ad-click counts as an always-open window is a product call. Route the greeting through `evaluateProactiveSendEligibility` (or add an approvalOverlay check in `getSendContext`); decide the window question explicitly.

9. **No contact identity matcher: `duplicate_contact_risk` is structurally dead and duplicate Contacts are created freely** (P1, proof-chain). Merged with the proof-chain-seam finding of the same shape. Contact has `@@index([organizationId, phoneE164])` but no `@@unique`. The WhatsApp gateway path dedups via `findByPhone`, but both CTWA and Instant Form leads enter via `LeadIntakeHandler`, which dedups ONLY on `idempotencyKey` (different per source), so one person produces two Contact rows with split consent/attribution. `duplicate_contact_risk` is hardcoded `false` at issuance and read; the only producer is a manual `flag_duplicate` reconcile. Consent on one contact id does not protect the duplicate. Add an automatic matcher (normalized `phoneE164` per org) feeding `evaluateExceptions`, or a nullable-safe `@@unique([organizationId, phoneE164])`; product decision (auto-merge vs flag-only) required.

10. **Consent revocation via operator route does not propagate to in-flight bookings** (P2, correctness). `consent-service.ts:245-307` `recordRevocation` only flips conversation status + saves a handoff when `openConversationSessionId` is provided; the admin revoke handler (`operator-intents/consent.ts:88-123`) never threads a session, and the schema carries no session id. `consentRevokedAt` IS stamped (so the next governed turn blocks), but no `human_override` flip or handoff fires, unlike the gateway scanner. Resolve the contact's open session in the operator intent and pass `openConversationSessionId`. Add a test asserting an operator revoke flips an open conversation to `human_override`.

11. **Greeting disclosure detection and consent snapshot rely on live mutable Contact state** (P3, proof-chain). `pdpa-consent-gate.ts:182-183` uses a brittle substring `includes()` for disclosure (its own comment warns whitespace drift breaks it; observe-only). The receipt read path recomputes `missing_consent` from current `consentGrantedAt/RevokedAt` rather than a snapshot. Replace the substring heuristic with a structured sentinel (`sentinelDetected` already scaffolded at line 213); snapshot `pdpaJurisdiction` + consent timestamps onto the ReceiptedBooking at issuance (mirror `expectedValueAtIssue`) for audit-grade proof.

### Ledger-lite (weekly owner report)

12. **Resend send result error is never inspected: an API-rejected email is recorded as a successful delivery** (P2, correctness). `send-email.ts:33-43` does `await resend.emails.send(...); return { ok: true }`. The Resend 6.x SDK resolves (not throws) with `{data:null, error}` on unverified-domain/invalid-recipient/rate-limit, so the try/catch never fires and `{ok:true}` flows to a `completed` WorkTrace and a "delivered to N recipient(s)" report. Most likely to bite at go-live: the default `EMAIL_FROM` is `noreply@switchboard.app`, which must be a verified sending domain. Destructure `{ error }` and return `{ok:false, reason:"send_error"}`; apply the same to `email-escalation-notifier.ts:55`.

13. **`no_recipients` records a `completed` success weekly with zero operator visibility** (P2, prod-wiring). `weekly-report-delivery.ts:122-127` returns `no_recipients` when recipients are empty; the cron worker (`ledger-weekly-report.ts:126-134`) treats any `completed` outcome as success with no warn, collapsing it into the same path as a real delivery. This is the inverse of the care taken for `entitlement_required` (line 113). Surface `no_recipients` distinctly (a `{skipped:"no_recipients"}` branch keyed off `delivered === false && recipientCount === 0`, or a warn).

### Quinn-lite (policy/approvals)

14. **Parked `WorkTrace.approvalId` is never stamped: receipted-booking `humanApprovalId` is structurally always null** (P1, proof-chain). The root cause is the approval-RESPONSE path, not the parking leg: both live response paths route through `writeApprovedPayloadToTrace` (`lifecycle-dispatch.ts:47-76`), whose `traceStore.update` writes `approvalOutcome`/`approvalRespondedBy`/`approvalRespondedAt` but OMITS `approvalId`. The only path that stamps it (`platform-lifecycle.ts:137-142`) is legacy and not wired into the API/channel routes. So `prisma-receipted-booking-store.ts:231` `humanApprovalId` can never demonstrate a human approved. The store supports the write (`prisma-work-trace-store.ts:545`). Add `approvalId: lifecycle.id` to the `writeApprovedPayloadToTrace` update (one change covers both live paths); add a parked-then-approved test asserting non-null `humanApprovalId`.

15. **Governance-verdict escalation hook is built and tested but never fired** (P2, prod-wiring). `claim-classifier.ts:391` emits `action: "escalate"` in enforce mode, but the verdict store is instantiated with `onWrite: recordGovernanceVerdictMetric` only (`skill-mode.ts:194-195`), and the lifecycle `registerVerdictWriteHook` is a no-op at both call sites (`app.ts:900`, `inngest.ts:1142`). So an escalate verdict only increments a Prometheus counter and never escalates the conversation. A second inertness layer: `app.ts:899` hardcodes `readMode: async () => "off"`. Compose the escalation registrar into `PrismaGovernanceVerdictStore.onWrite` AND address the hardcoded `readMode: "off"`. Add an integration test (escalate verdict with lifecycle mode `on` produces an `escalated` transition).

16. **`GovernanceVerdictStore.listByConversation`/`listByDeployment` have no org filter (latent cross-org read)** (P3, security). `prisma-governance-verdict-store.ts:98-119` filters only on conversationId/deploymentId; the model has no `organizationId` column (org is only reachable via the deployment relation). No production caller invokes these read methods today (only `.save(...)` is used), so it is latent. Same shape as F12/#1134. Add an `organizationId` parameter (join on `deployment.organizationId` or denormalize) before the first verdict-history consumer is wired.

### Robin (recovery/show-rate)

17. **Transient send/DB failure permanently burns a no-show's single recovery attempt** (P2, correctness, needs-human). `robin-recovery-executor.ts:240-249` marks the claimed row `failed` on a Graph `!ok` or any thrown error; the dedup key `recovery:{kind}:{org}:{bookingId}` has no week/attempt component, and the store offers no reclaim path. A later cron run P2002-skips the failed booking forever. The single-attempt design is deliberate and test-locked, so the bounded-retry-vs-single-attempt choice is a product decision. Distinguish terminal skips (consent revoked, no phone) from transient failures and allow a bounded retry for `failed` rows.

18. **Dispatch while the template is still draft consumes the entire cohort's dedup rows** (P1, go-live). `proactive-eligibility.ts:89-96` returns `template_not_approved` AFTER the claim-first `store.create` in `robin-recovery-executor.ts:173-225`, so every recipient is claimed and `markSkipped`. The approvalOverlay is org-wide (identical for every recipient), exactly like the creds gap that DOES short-circuit without claiming (lines 136-156). When the template is later approved, the next run P2002-skips the whole first cohort permanently. Since templates ship draft and go-live order is mode-on then template-approval, this is a likely trap. Hoist the org-wide template-availability check ahead of the claim.

19. **Cohort self-rebooked exclusion is computed at freeze time, not re-checked at dispatch** (P2, correctness). `robin-recovery-dispatch.ts:92-100` drops self-rebookers and freezes the cohort; the campaign parks for mandatory approval (freeze-to-dispatch can be days); `getSendContext` re-resolves only consent/phone/jurisdiction, never future bookings. A patient who self-rebooks during the park window still gets a "sorry you missed us" message the consent gate cannot catch. Re-check future-booking status at dispatch (reuse `findFutureBookingContactIds`) and `markSkipped("already_rebooked")`, mirroring the existing dispatch-time consent re-validation.

20. **A fully-failed recovery campaign reports outcome `completed` with no failure-rate signal** (P3, prod-wiring). `robin-recovery-executor.ts:253-257` always returns `completed` even when `failed === total`; unlike the creds-missing path there is no per-recipient failure metric and no high-failure-ratio alert (`metrics.ts` has only `whatsappProactiveSendSkipped`, no `...Failed`). Emit a per-recipient send-failure metric (label intent + reason) and/or surface a degraded outcome when `failed/total` exceeds a threshold.

### Mira (creative; deferred)

21. **Legacy creative-pipeline polling hooks are dual-lifecycle-blind** (P3, correctness). `use-creative-pipeline.ts:23,44` gate refetch-stop solely on `currentStage !== "complete"`; a UGC job advances `ugcPhase` not `currentStage`, so a finished UGC job would poll every 30s forever. Currently dead code (no page mounts `useCreativeJobs`/`useCreativeJob`). Delete the unused hooks, or make the predicate mode-aware (mirror `ugc-job-runner.ts:264`).

22. **Mira Keep/Pass writes a publish-gating field via direct Prisma `updateMany` outside PlatformIngress** (P3, prod-wiring, needs-human). `mira-decision.ts:59-62` writes `reviewDecision`/`reviewDecidedAt` directly; `reviewDecision === "kept"` is a hard publish gate (`creative-publish-preconditions.ts:73-79`) and a learning trigger. The route is allowlisted and the paid spend stays governed at `creative.job.publish`, so this is not an invariant violation, but there is no WorkTrace recording who kept/passed what. Emit a lightweight audit/WorkTrace entry (or route through an `operator_mutation` ingress intent); tighten the route comment.

23. **UGC runner injects hardcoded `deploymentType="standard"`, making the approval config's deploymentType axis inert** (P3, code-quality). `inngest.ts:1404` returns a literal `type: "standard"` discarding the real `deployment.type`; `shouldRequireApproval` (`approval-config.ts:22-33`) never reads `deploymentType`. Dead on both ends; harmless today (trustLevel 0 forces approval). Pass the real `type` through the adapter, or drop `deploymentType` from the signature/context.

## (c) Cross-cutting seams

### Governance and security

24. **Meta lead-greeting send path skips the template-approval gate** (governance seam). Merged into finding 8 (Casey) above as the canonical P1.

25. **`system_auto_approved` financial bypass relies on a 3-prefix denylist + known spend keys** (P3, security, needs-human). `governance-gate.ts:103-134`: `FINANCIAL_AUTO_APPROVE_DENYLIST` hardcodes only the three `adoptimizer.campaign.*` prefixes, and `isFinancialIntent` otherwise depends on `OUTBOUND_SPEND_KEYS`. A future payout/refund/deposit-return intent mis-registered `system_auto_approved` with a bespoke amount field off the denylist could short-circuit before the spend gate. No currently-registered intent fits the gap. Promote to a registration-time invariant: any outbound-money executor binding must be `spendBearing:true`; add a taxonomy test. Keep the denylist as defense-in-depth.

26. **Spend-autonomy reversibility brake degrades to mutationClass-only in production** (P3, prod-wiring). `spend-approval-threshold.ts:60-66`: no cartridge populates reversibility in prod (operator/skill modes never load one), so the brake reduces to `mutationClass !== "destructive"`. Latent foot-gun for the deferred payment-charge/refund executors: a reversible-looking `write` intent becomes auto-executable under threshold on an autonomous deployment. Register charge/refund as `destructive` or `reversibility:"none"`, and pin a test that an irreversible-financial intent is not relaxed under threshold, before any payment-mutation executor ships.

### Prod-wiring and activation

27. **CTWA inbound leads attributed to a DeploymentConnection id, not the Alex AgentDeployment id** (P1, correctness). `managed-webhook.ts:161` passes `deploymentId: gatewayEntry.deploymentConnectionId` (a channel binding) into the CTWA lead.intake, flowing to `ActivityLog.deploymentId` AND `Contact.deploymentId`. The Meta producer instead uses the resolved Alex AgentDeployment id (`meta-lead-intake-workflow.ts:128`). `listByDeployment` filters by id, so a per-Alex activity feed (agent CRM `activity.list`) surfaces Meta IF leads but silently misses every CTWA lead. `deploymentId` is a plain String (no FK), so it orphans without error. CTWA is a primary paid-lead funnel. Resolve the org's Alex AgentDeployment (`resolveByOrgAndSlug(orgId, "alex")`) and pass that id; add a test.

28. **Alex deployment is NOT seeded by `provisionOrgAgentDeployments`** (P3, prod-wiring, needs-human). `provision-org-agents.ts:55-57` deliberately omits `ensureAlexForOrg`; `meta.lead.intake` submits `targetHint:{skillSlug:"alex"}` and the resolver throws `No active deployment found` if Alex is absent. Today every caller also ensures Alex (org-creation route + `provisionPilotOrg`), and self-serve is closed, so the failure is latent. Fold the idempotent `ensureAlexForOrg` into the always-run transaction (decision-gated because the omission was a deliberate trade-off to avoid re-upserting Alex on the hot GET /config route).

29. **Stale `EXECUTOR_NOT_WIRED placeholder` comment on the Riley reallocate intent** (P3, code-quality). `contained-workflows.ts:559-560` says the executor is a fail-closed placeholder, but `riley-budget-executor.ts:88-152` builds the real read-modify-re-read executor, registered and invoked by the cron. Misleading provenance on a flag-gated money-move path. Update the comment to point at `riley-budget-executor.ts` and the `RILEY_REALLOCATE_SELF_EXECUTION_ENABLED` gate.

### Test integrity

30. **api test harness wires `resolveAuthoritativeDeployment(null)` with no predicate** (P2, test). `test-server.ts:470` (no options) + `:472` (`entitlementResolver: undefined`) means the resolver short-circuits ALL intents to platform-direct before the predicate is consulted, and the entitlement gate is dead. Production wires both. The strict-lookup/`deployment_not_found`/trustLevelOverride-forwarding/entitlement_required branches are exercised only by dedicated `*-live-path` unit tests, never by the shared harness, so any NEW intent relying on the harness for e2e coverage gets a false-green. Thread `{ isPlatformDirectIntent, entitlementResolver }` into `BuildTestServerOptions`, or document the limitation at the wiring site.

31. **Owner-report route to period-rollup to `computeReceiptedBookingRevenue` seam validated only against an empty fixture** (P3, test). `test-server.ts:411` `listForCohort: async () => []`, so `api-reports.test.ts` validates only route plumbing over a zero-row cohort. The compute fn and the prisma store are independently well-tested, so this is seam-completeness only. Optionally seed one populated `ReceiptedBookingView` so a regression dropping `paidRevenueCents`/`cohortSize` between route and rollup is caught at the integration boundary.

### Proof-chain and data integrity

32. **Report assembles the full receipted-booking cohort TWICE per cache-miss (N+1 doubled, divergent `now`)** (P2, correctness). `period-rollup.ts:62-63` runs `computeReceiptedBookingQuality` and `computeReceiptedBookingRevenue` in one `Promise.all`; both independently call `listForCohort`, which fans out `getView` (~8 org-scoped reads/booking). A single cache-miss runs `2 x (1 + ~8N)` queries, and the two passes default `now` to independent wall-clock reads. The count is unaffected (pure performance/consistency defect). Assemble the cohort once and pass the resolved `ReceiptedBookingView[]` into both compute fns; thread a single `ctx` clock.

33. **Receipt partial-unique exists only in raw-SQL migration, not in `schema.prisma`** (P2, correctness). `migration.sql` defines `Receipt_org_kind_externalRef_key` (partial unique WHERE `externalRef IS NOT NULL`), but the Receipt model declares no `@@unique` and no comment pointing at the raw-SQL index (unlike `LifecycleRevenueEvent`, which does carry that convention). The double-mint guard rests on this index + a `findFirst`-before-create with no P2002 catch; `computeBookingPaidValue` sums `amount` over all paid receipts, so a dropped index lets concurrent webhook replays double-count proven-paid revenue. `check-prisma-drift.sh` cannot see a raw-SQL partial-unique, so a `db push`/shadow-reset would not be caught. Add the mirror comment to the Receipt model AND a positive integration/db assertion that the index exists.

34. **Automatic duplicate-contact detection is permanently disabled** (proof-chain seam). Merged into finding 9 (Casey) above as the canonical P1.

### Operational go-live readiness

35. **Proactive WhatsApp sends use a single global phone-number-id/token; per-org WhatsApp Connection is never resolved** (P1, prod-wiring). All four proactive SEND workflows (`conversation-reminder-send-workflow.ts:103-104`, `conversation-followup-send-workflow.ts:105`, `meta-lead-greeting-workflow.ts:126`, `robin-recovery-executor.ts:111-135`) read the global env phone-number-id and token, while inbound/reply (`runtime-registry.ts:172-176`) and template-create (`whatsapp-template-create.ts:80`) resolve the per-org Connection. The per-tenant `{token, phoneNumberId}` already persists (`whatsapp-connection-data.ts:52-59`). Once tenant #2 onboards with its own WABA, its reminders/follow-ups/greetings/recovery POST to the GLOBAL phone-number-id with the global Bearer: tenant #1's number sends to tenant #2's patients. The executor self-documents it as the known multi-tenant follow-up. Thread a per-org send-cred resolver mirroring #1197's `calendar-provider-factory.ts`, falling back to global env only for the single-tenant pilot. Hard cross-tenant blocker; gate tenant #2 onboarding on this.

36. **DEPLOYMENT-CHECKLIST documents only the global WhatsApp token; `.env.example` carries a stale #1156 warning** (P2, go-live). `DEPLOYMENT-CHECKLIST.md:38` lists the global WhatsApp vars with no multi-tenant caveat; `.env.example:129-131` still says first-touch greeting is dark until #1156 lands, but `meta-lead-greeting-workflow.ts:17-18` already resolves the token alias, so #1156 has landed. Update the checklist to mark the global vars as the single-tenant fallback and require embedded-signup per-org onboarding for multi-tenant; replace the stale #1156 paragraph.

37. **Per-org calendar provider cache has no TTL/invalidation** (P2, prod-wiring). `calendar-provider-factory.ts:31` memoizes the resolved provider by orgId for process lifetime; the google-calendar OAuth callback (`google-calendar-oauth.ts:191-208`) writes new creds but never invalidates the cache. A clinic that books against the fallback/noop provider, then self-serve connects its own Google Calendar, keeps writing bookings to the wrong calendar until API restart. The in-code comment flags it as a deliberate beta deferral. Invalidate the orgId cache entry on the OAuth callback (or a short TTL); at minimum document the restart-after-connect go-live step.
