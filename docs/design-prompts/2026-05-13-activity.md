# Claude Design Prompt — /activity

Generated 2026-05-13. Backend-verified against `packages/schemas/src/audit.ts`, `packages/core/src/audit/list-entries.ts`, `apps/api/src/routes/dashboard-activity.ts`. Existing UI at `apps/dashboard/src/app/(auth)/(mercury)/activity/` — this is a deliberate rebuild that surfaces more of the backend.

---

Redesign the /activity page for "Switchboard" — the org-wide audit log feeding operator forensics and compliance. An existing version lives at apps/dashboard/src/app/(auth)/(mercury)/activity/ and works correctly; the goal here is a calmer, denser, more scannable rebuild that exposes the full backend capabilities the current UI underuses.

## Domain (matches packages/schemas/src/audit.ts)

Each AuditEntryBrowseRow has:

- id, eventType (one of 45 types — see below), timestamp (ISO)
- actorType: "user" | "agent" | "service_account" | "system"
- actorId, entityType, entityId
- riskCategory: "none" | "low" | "medium" | "high" | "critical"
- visibilityLevel: "public" | "org" (server-filtered; client always trusted)
- summary: short prose
- snapshotKeys[]: ONLY allowlisted keys (id, kind, source, actionType, decisionId, recommendationId, approvalId, envelopeId, agentKey, targetEntityType, targetEntityId, correlationId, traceId) — full snapshot values are server-redacted
- redactedKeyCount: number of fields redacted (display as "+3 redacted")
- evidencePointers[]: each has type, hash (full), hashPrefix (16 chars for display) — storageRef is NEVER exposed
- entryHash, previousEntryHash — hash chain integrity (display chain anchor)
- envelopeId (nullable), traceId (nullable)

Event types fall into bands:

- Action lifecycle (19): action.proposed → action.resolved → action.enriched → action.evaluated → action.approved / partially_approved / rejected / patched / queued / executing / snapshot / executed / failed / denied / expired / cancelled / undo_requested / undo_executed / approval_expired
- Identity & governance (16): identity.created/updated, overlay.activated/deactivated, policy.created/updated/deleted, connection.established/revoked/degraded, competence.promoted/demoted/updated, delegation.chain_resolved, entity.linked/unlinked/resolved
- Events & reactions (3): event.published, event.reaction.triggered, event.reaction.created
- Agent & WorkTrace (6): agent.activated/emergency-halted/resumed, work_trace.persisted/updated/integrity_override

There's an OPERATIONAL allowlist of 16 of these for the default "operational" scope view — surface this as a meaningful default, not just a filter.

## API capabilities (GET /api/dashboard/activity)

Query params to expose as UI affordances:

- scope: "operational" | "all" | (auto-becomes "custom" if narrowing params)
- eventType, actorType, entityType, entityId
- after, before (ISO range)
- cursor (keyset, base64url {timestamp,id}); limit 1–100, default 50

Response: { rows[], nextCursor, scope, appliedFilters }. No polling — pagination breaks on autorefresh. Manual "refresh" button only.

## Layout

Single-column dense table, page-width 74rem. Filter rail on top, sticky.

Top filter strip:

- Scope segmented control: [Operational] [All] [Custom*] (\* lights up when narrowing params present)
- Event-type combobox (autocomplete over the 45 types)
- Actor-type pills (User / Agent / Service / System)
- Date range picker (after/before)
- Entity selector (entityType then entityId)
- "Clear filters" pill (right-aligned, only when active)

Row (compact, 44px tall):

- Time column (left): "14:23:11" in mono, with relative "4m ago" below in muted; expands to full ISO in detail
- Event-type column: monospaced badge, e.g. "action.executed" — color is the existing palette only, never raw red/green
- Actor: type icon (TEXT glyph "USR" / "AGT" / "SYS" / "SVC") + actorId, mono, truncated
- Entity: entityType · entityId, mono
- Summary: free text, primary font, truncated to ~80ch
- Risk: subtle right-edge tint (none = no tint; critical = darkest amber)
- Chevron: expand to inline detail (not modal — inline accordion preserves context for scrolling forensic work)

Inline expanded detail block:

- Snapshot keys: list of present keys; "+N redacted" badge
- Evidence: hash + hashPrefix, copy-to-clipboard buttons; storageRef intentionally absent — label the absence ("evidence reference held server-side")
- Hash chain: entryHash (full, mono) and previousEntryHash (mono, with "view previous" button that scrolls to that row if visible)
- envelopeId / traceId: if present, both copyable, both link to the related surface (/approvals/[envelopeId], /traces/[traceId])

Footer pagination:

- "Showing 50 of …" (no total — keyset can't know)
- [Newer] / [Older] buttons stacking cursor history (already implemented)

## Design system (verified globals.css)

- Background --sw-base; surface --sw-surface; raised --sw-surface-raised
- Borders --sw-border / --sw-border-strong (hairlines, not 2px)
- Text --sw-text-primary / -secondary / -muted
- Accent --sw-accent (#A07850) ONLY for active filter chip + selected row
- Font: --font-sans Inter for prose; mono (JetBrains/system mono) for IDs, hashes, event types, timestamps, snapshot keys
- Motion: 280ms ease-standard for row expand
- Section labels: 11px uppercase tracked-wider, --sw-text-muted

## State coverage

- Loading: skeleton rows (preserve row height to prevent layout jitter)
- Empty (zero): "No activity recorded yet" + last-recorded timestamp from ledger health
- Empty (filtered): "No matches" + "Clear filters" CTA
- Error: inline banner above table; don't unmount table
- Stale (no auto-refresh): subtle "fetched 4m ago · refresh" pill bottom-right

## Anti-patterns

- No infinite scroll — cursor pagination is canonical.
- No traffic-light colors; risk is hairline + amber depth.
- No real-time tail by default (mention an opt-in toggle but don't build it).
- Don't surface full snapshot values — the server already redacted them.
- Don't expose storageRef. Surface the _absence_ explicitly.

## Deliverable

React + Tailwind page + row component + inline detail. Fixtures: 30 rows spanning all 4 actor types, ~12 distinct event types across bands, mix of risk categories, 4 with quorum/approval lineage, 6 with hash-chain neighbors visible in the same page, 2 with redactedKeyCount > 0.
